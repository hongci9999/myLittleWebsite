# 논리 ER 다이어그램 (Entity-Relationship Diagram)

## 📌 핵심 요약
개체와 개체 간의 관계를 약속된 도형을 사용하여 시각적으로 표현한 도구입니다. 특정 [[DBMS]] 환경에 종속되지 않는 독립적인 설계 모델이며, 업무 영역의 변경이 발생할 때 모델링 구조가 변경됩니다.

---

## 📝 상세 정리

### 1. ER 모델의 특징
- 목표 시스템의 기술적 환경(H/W, OS 등)을 고려하지 않는 논리적 설계 단계입니다.
- 업무 수행 방식이 변해도 데이터 구조 자체는 크게 영향을 받지 않으나, **업무 영역 자체가 변경**될 경우에는 설계 변경이 필수적입니다.

### 2. 피터 챈(Peter Chen) 표기법
기사 시험에서 가장 빈번하게 출제되는 고전적인 표기법입니다. 각 요소를 고유한 도형으로 정의합니다.



| 구분 | 도형 | 설명 |
| :--- | :--- | :--- |
| **[[개체 타입(Entity Type)]]** | 직사각형 | 독립적으로 존재하는 실체 |
| **[[약한 개체 타입(Weak Entity Type)]]** | 이중 직사각형 | 상위 개체의 식별자에 의존하는 개체 |
| **[[속성(Attribute)]]** | 타원 | 개체의 일반적인 성질 |
| **[[기본키 속성(PK Attribute)]]** | 타원(밑줄) | 개체를 유일하게 식별하는 속성 |
| **[[다중값 속성(Multi-valued Attribute)]]** | 이중 타원 | 하나의 속성에 여러 값을 가질 수 있는 경우 (예: 전화번호) |
| **[[관계 타입(Relationship Type)]]** | 마름모 | 개체 간의 논리적인 연결 상태 |
| **연결** | 실선 | 요소들 간의 관계를 잇는 선 |

### 3. 정보공학(Information Engineering, IE) 표기법
개체와 속성을 하나의 그룹(박스)으로 묶어 표현하며, 관계형 데이터베이스 설계에 최적화된 방식입니다. 일명 **까마귀 발(Crow's Foot)** 표기법으로도 불립니다.



- **대응수(Cardinality):** $1:1$, $1:N$, $N:M$ 등을 선의 끝 모양으로 표현합니다.
- **참여도:** - **필수(Mandatory):** 실선 또는 바(`|`)로 표시하며, 반드시 데이터가 존재해야 함을 의미합니다.
    - **선택(Optional):** 원(`O`)으로 표시하며, 데이터가 존재하지 않을 수도 있음을 의미합니다.

## 🏗️ 시나리오를 통한 다이어그램 추출 (예시)

> **[시나리오]**
> "학생은 여러 과목을 수강할 수 있고, 하나의 과목은 여러 학생에 의해 수강될 수 있다. 학생 개체는 학번(PK)과 이름을 가지며, 수강 관계에서는 성적 속성이 발생한다."

1. **개체 추출:** `학생`, `과목`
2. **속성 추출:** 학번(PK), 이름, 과목코드(PK), 과목명
3. **관계 추출:** `수강` (N:M 관계)
4. **결과:** 피터 챈 표기법 사용 시, 학생(사각형) - 수강(마름모) - 과목(사각형) 구조로 연결하며, 수강 마름모에 '성적' 타원을 연결합니다.

## 4. 확장 ER 다이어그램 (EERD)
기존 ERD에 객체 지향적인 개념을 추가한 모델입니다.

### 4.1 서브타입 (Subtype)
- 하나의 상위 개체 타입이 다수의 하위 개체 타입과 관계를 맺는 구조입니다.
- **배타적 서브타입 (Exclusive):** 상위 개체가 하위 서브타입 중 단 하나하고만 연결됩니다.
- **포괄적 서브타입 (Inclusive):** 상위 개체가 하나 이상의 서브타입과 동시에 연결될 수 있습니다.

### 4.2 설계 방식의 분류
| 개념 | 설명 | 예시 |
| :--- | :--- | :--- |
| **[[일반화(Generalization)]]** | 하위 개체들의 공통점을 찾아 상위 개체를 정의 (상향식) | 승용차, 트럭 → **차량** |
| **[[구체화(Specialization)]]** | 상위 개체를 세부적인 하위 개체로 분리 (하향식) | **사람** → 학생, 교수 |
| **[[집단화(Aggregation)]]** | 관련 개체들을 묶어 새로운 상위 개체 생성 (Part-of 관계) | 엔진, 바퀴, 차체 → **자동차** |
| **[[분해화(Decomposition)]]** | 결합된 개체를 다시 세부 구성 요소로 나눔 | **컴퓨터** → CPU, RAM, HDD |
| **[[분류화(Classification)]]** | 공통 속성을 가진 개체들을 그룹화하여 정의 | 철수, 영희 → **학생** |
| **[[인스턴스화(Instantiation)]]** | 개체 타입으로부터 실제 개체(인스턴스)를 생성 | **도서** → '자바의 정석' 책 1권 |

> [!INFO] 💡 AI 보충 설명
> **IS-A 관계 vs PART-OF 관계:** > - 일반화/구체화는 '학생 IS-A 사람'과 같이 상속 개념이 강한 **IS-A** 관계입니다. 
> - 집단화/분해화는 '엔진은 자동차의 부품이다'와 같은 부분-전체 개념의 **PART-OF** 관계입니다.

## 🎯 빈출 키워드
- **[[피터 챈 표기법]]**: 마름모(관계), 타원(속성), 직사각형(개체)을 사용하는 전통적 ERD 표기법입니다.
- **[[까마귀 발 표기법]]**: 정보공학(IE) 표기법에서 다($N$)의 관계를 나타내는 선의 모양입니다.
