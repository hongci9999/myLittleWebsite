# OSI 7계층 및 TCP/IP 프로토콜

## 📌 핵심 요약
네트워크 통신의 표준 모델인 **OSI 7계층**과 실제 인터넷에서 사용되는 **TCP/IP 4계층** 모델을 비교 분석합니다. 각 계층의 핵심 기능(오류 제어, 흐름 제어, 주소 지정 등)과 주요 프로토콜(HTTP, TCP, UDP, IP 등)을 상세히 다룹니다.

---

## 📝 상세 정리

## 1. 통신 프로토콜 (Communication Protocol)
컴퓨터 간 원활한 데이터 통신을 위해 정의된 규약과 약속입니다.
- **프로토콜의 3요소**:
    - **구문(Syntax)**: 데이터의 형식, 부호화, 신호 레벨 등을 정의.
    - **의미(Semantics)**: 제어 정보, 오류 제어 및 응답 방식 정의.
    - **시간(Timing)**: 통신 속도 및 순서 관리 기법 정의.
- **주요 기능**: 주소 설정([[Addressing]]), 순서 제어, 분할 및 재조립, 캡슐화, 흐름 제어, 오류 제어, 동기화, 다중화.



## 2. OSI 7계층 (Open Systems Interconnection)

### 2.1 하위 계층 (Physical, Data Link, Network)
- **1계층: 물리 계층 ([[Physical Layer]])**: 물리적 매체를 통해 비트(Bit) 신호를 전송.
    - 장비: 허브(Hub), 리피터(Repeater), 케이블.
- **2계층: 데이터 링크 계층 ([[Data Link Layer]])**: 인접 노드 간 신뢰성 있는 데이터 전송 및 오류/흐름 관리.
    - 단위: 프레임(Frame).
    - 특징: [[MAC]] 주소 기반 통신.
    - 장비: 브리지(Bridge), 스위치(Switch).
- **3계층: 네트워크 계층 ([[Network Layer]])**: 데이터의 목적지까지 최적의 경로를 설정([[Routing]]).
    - 단위: 패킷(Packet).
    - 특징: 논리적 주소인 [[IP]] 부여.
    - 장비: 라우터(Router), L3 스위치.

### 2.2 상위 계층 (Transport, Session, Presentation, Application)
- **4계층: 전송 계층 ([[Transport Layer]])**: 종단 간(End-to-End) 신뢰성 있고 투명한 데이터 전송 보장.
    - 기능: 오류 검출/복구, 흐름 제어, 다중화.
    - 프로토콜: [[TCP]], [[UDP]].
- **5계층: 세션 계층 ([[Session Layer]])**: 응용 프로세스 간의 대화 유지 및 관리.
    - 기능: 대화 제어, 동기점([[Checkpoint]]) 삽입을 통한 복구.
- **6계층: 표현 계층 ([[Presentation Layer]])**: 데이터의 형식 설정 및 독립성 제공.
    - 기능: 코드 변환, 데이터 압축, **암호화/복호화**.
- **7계층: 응용 계층 ([[Application Layer]])**: 사용자가 네트워크 서비스에 접근할 수 있도록 인터페이스 제공.
    - 프로토콜: HTTP, FTP, SMTP 등.

---

## 3. TCP/IP 응용 계층 프로토콜

### 3.1 전자 우편 (E-Mail)
- **[[SMTP]] (Simple Mail Transfer Protocol)**: 메일을 송신 측 서버로 전송.
- **[[POP3]]**: 서버에서 메일을 로컬 컴퓨터로 다운로드(수신 후 서버 삭제가 기본).
- **[[IMAP]]**: 서버에서 메일 제목/헤더를 분석하여 관리(서버와 동기화).
- **[[MIME]]**: 멀티미디어 메일 전송을 위한 확장 규격.
- **[[S/MIME]]**: 보안이 강화된 MIME.

### 3.2 원격 제어 및 파일 전송
- **[[Telnet]]**: 원격 접속 서비스. 평문 전송으로 보안 취약(23번 포트).
- **[[SSH]] (Secure Shell)**: 암호화된 원격 접속(22번 포트).
- **[[FTP]]**: 파일 업로드/다운로드. (Active/Passive 모드)
    - **Anonymous FTP**: 익명 파일 전송.
    - **TFTP**: 단순화된 파일 전송(UDP 기반).
- **[[RTP]] (Real-time Transport Protocol)**: 실시간 멀티미디어 데이터 전송.

### 3.3 웹 서비스 (WWW)
- **[[HTTP]]**: 웹 문서 전송 규약(80번 포트).
- **[[HTTPS]]**: HTTP에 [[SSL/TLS]] 암호화 계층 추가(443번 포트).

---

## 4. 전송 계층 (Transport Layer)

### 4.1 [[TCP]] (Transmission Control Protocol)
신뢰성 있는 연결 지향성 통신 프로토콜입니다.
- **특징**: [[3-way Handshaking]]을 통한 연결 설정, 흐름 제어, 혼잡 제어.
- **TCP 헤더 구조 (AI 보충)**:
    - **Source/Destination Port**: 송수신 프로세스 식별.
    - **Sequence Number**: 패킷의 순서 번호(조립 시 사용).
    - **Acknowledgment Number**: 수신 확인 번호.
    - **Checksum**: 데이터 무결성 검사.
    - **Window Size**: 흐름 제어를 위한 수신 윈도우 크기.

### 4.2 [[UDP]] (User Datagram Protocol)
비연결성, 비신뢰성 프로토콜로 오버헤드가 적고 속도가 빠릅니다.
- **특징**: 확인 응답 없음, 스트리밍이나 실시간 서비스에 적합.
- **사용 사례**: TFTP, SNMP, RIP, NTP, RTP 등.

---

> [!INFO] 💡 AI 보충 설명: TCP 3-way Handshaking 과정
> 1. **Client -> Server (SYN)**: 클라이언트가 연결을 요청하며 시퀀스 번호를 보냄.
> 2. **Server -> Client (SYN + ACK)**: 서버가 요청을 수락하고 응답 번호($SEQ+1$)와 서버측 시퀀스 번호를 보냄.
> 3. **Client -> Server (ACK)**: 클라이언트가 최종 확인 응답을 보냄으로써 연결 확립.

## 🎯 빈출 키워드
- **[[OSI 7계층]]**: 국제 표준 통신 모델 (물-데-네-전-세-표-응).
- **[[TCP]] vs [[UDP]]**: 연결 지향(신뢰성) vs 비연결(속도)의 차이.
- **[[SMTP]] / [[POP3]] / [[IMAP]]**: 전자 우편 시스템의 3대 프로토콜.
- **[[SSH]]**: Telnet의 보안적 대안으로 암호화 통신 지원.



# IP 계층 프로토콜 및 서브네팅

## 📌 핵심 요약
인터넷 계층의 핵심인 **[[IPv4]]**와 **[[IPv6]]**의 구조를 분석하고, 주소 부족 문제를 해결하기 위한 **[[CIDR]]**, **[[NAT]]** 기술을 정리합니다. 특히 시험에 반드시 출제되는 **서브네팅([[Subnetting]])** 계산법(FLSM, VLSM)을 상세히 다룹니다.

---

## 📝 상세 정리

## 1. 인터넷 계층 (Internet Layer) 주요 프로토콜

### 1.1 [[IP]] (Internet Protocol)
패킷 단위의 데이터를 목적지까지 전달하는 비연결성/비신뢰성 프로토콜입니다.
- **[[ARP]] (Address Resolution Protocol)**: 논리적인 IP 주소를 물리적인 MAC 주소로 변환.
- **[[RARP]] (Reverse ARP)**: MAC 주소를 IP 주소로 변환.
- **[[ICMP]] (Internet Control Message Protocol)**: 네트워크의 오류 보고 및 상태 정보 제공.

### 1.2 [[IPsec]] (IP Security)
IP 계층에서의 보안 강화를 위한 프로토콜 모음입니다.
- **주요 구성**: 
    - **AH**: 인증 및 무결성 보장.
    - **ESP**: 인증, 무결성 + **기밀성(암호화)** 보장.
    - **IKE**: 키 교환 및 보안 설정.
- **동작 모드**:
    - **전송 모드**: 데이터(Payload)만 보호.
    - **터널 모드**: 패킷 전체(Header + Payload)를 보호.

---

## 2. IP 헤더 구조 (AI 보충 설명)

### 2.1 IPv4 헤더 구조

- **Version**: 4비트, IPv4는 '4'.
- **IHL (Header Length)**: 헤더의 길이.
- **Type of Service (ToS)**: 서비스의 우선순위.
- **Total Length**: 패킷 전체 길이.
- **Identification / Flags / Fragment Offset**: 패킷 분할(Fragmentation) 관련 필드.
- **TTL (Time To Live)**: 라우터를 거칠 때마다 1씩 감소하여 0이 되면 패킷 폐기 (무한 루프 방지).
- **Protocol**: 상위 계층 프로토콜 (TCP: 6, UDP: 17).
- **Header Checksum**: 헤더 오류 검사.
- **Source/Destination IP Address**: 송수신자 IP.

### 2.2 IPv6 헤더 구조


[Image of IPv6 Header structure]

- **Version**: '6'.
- **Traffic Class**: 서비스 우선순위 관리.
- **Flow Label**: 특정 실시간 흐름(Flow) 식별.
- **Payload Length**: 헤더 뒤 데이터 길이.
- **Next Header**: 다음 상위 프로토콜 헤더 타입 (IPv4의 Protocol 필드 역할).
- **Hop Limit**: IPv4의 TTL과 동일 역할.

---

## 3. IP 주소 체계 및 서브네팅

### 3.1 [[IPv4]] 주소 클래스
- **A 클래스**: `0.0.0.0 ~ 127.255.255.255` (국가/대형 기관)
- **B 클래스**: `128.0.0.0 ~ 191.255.255.255` (중대형 기관)
- **C 클래스**: `192.0.0.0 ~ 223.255.255.255` (소규모 기관)
- **D 클래스**: 멀티캐스트 전송용.
- **E 클래스**: 미래 예약용.

### 3.2 주소 부족 해결 방안
- **[[CIDR]] (Classless Inter-Domain Routing)**: 클래스 경계를 무시하고 비트 단위로 주소를 할당하여 낭비 방지.
- **[[DHCP]]**: 동적인 IP 주소 자동 할당.
- **[[NAT]] (Network Address Translation)**: 사설 IP를 공인 IP로 변환하여 주소 절약 및 보안 강화.

---

## 📝 상세 정리

## 1. 서브네팅의 기초 개념
- **목적**: IP 주소 절약, 트래픽 제어(충돌 방지), 관리 편의성.
- **계산 원칙**: 서브넷을 나눌 때는 반드시 $2^n$의 배수 단위로 나누어야 합니다.
- **예약 주소**: 각 서브넷에서 다음 두 가지 주소는 호스트에게 할당할 수 없습니다.
    1. **네트워크 주소**: 해당 서브넷의 첫 번째 주소.
    2. **브로드캐스트 주소**: 해당 서브넷의 마지막 주소.
    - 사용 가능한 호스트 수 공식: $2^{호스트 비트} - 2$

## 2. FLSM (Fixed Length Subnet Mask)
동일한 크기로 서브넷을 나누는 방식입니다. 각 그룹의 호스트 수가 유사할 때 사용합니다.

### 2.1 FLSM 실습 예제
- **조건**: `192.168.1.0/24` 네트워크를 **6개의 서브넷**으로 분할.
- **단계별 풀이**:
    1. **서브넷 비트 결정**: 6개를 수용하기 위해서는 $2^2=4$는 부족하므로, $2^3=8$개를 만들어야 합니다. 따라서 **3비트**를 서브넷 비트로 빌려옵니다.
    2. **서브넷 마스크 변화**: 
        - 기존: `/24` ($255.255.255.0$)
        - 변경: `/27` ($255.255.255.224$) → 24비트 + 3비트 = 27비트.
    3. **비트 구조 분석**:
        - `11000000.10101000.00000001.`**[SSS]**[HHHHH]
        - **S(Subnet)**: 빌려온 3비트 ($000$ ~ $111$까지 8개 조합)
        - **H(Host)**: 남은 5비트 ($00000$ ~ $11111$까지 32개 주소)
    4. **결과**:
        - 서브넷당 총 IP 개수: $2^5 = 32$개.
        - 실제 사용 가능 IP: $32 - 2 = 30$개.
        - **IP-Subnet-Zero**: 과거에는 0번 서브넷을 사용하지 않았으나, 현재는 이 옵션을 적용하여 첫 번째 서브넷($000$번)부터 모두 사용합니다.

---

## 3. VLSM (Variable Length Subnet Mask)
서브넷마다 서로 다른 크기로 가변적으로 나누는 방식입니다. 호스트 수가 크게 차이 날 때 효율적입니다.

### 3.1 VLSM 실습 예제 (요청 내용)
- **조건**: `192.168.1.0/24`를 다음 세 그룹으로 분할.
    - **A그룹**: 60대
    - **B그룹**: 30대
    - **C그룹**: 16대

- **단계별 풀이 (큰 그룹부터 할당하는 것이 원칙)**:



#### ① A그룹 할당 (60대 필요)
- 60대를 수용하기 위한 최소 $2^n$은 $2^6 = 64$입니다.
- **호스트 비트**: 6비트 필요.
- **서브넷 마스크**: $32 - 6 = /26$
- **범위**: `192.168.1.0 ~ 192.168.1.63`
- **사용 가능 IP**: `1.1 ~ 1.62` (62개)

#### ② B그룹 할당 (30대 필요)
- 남아있는 `192.168.1.64`부터 시작합니다.
- 30대를 수용하기 위한 최소 $2^n$은 $2^5 = 32$입니다.
- **호스트 비트**: 5비트 필요.
- **서브넷 마스크**: $32 - 5 = /27$
- **범위**: `192.168.1.64 ~ 192.168.1.95`
- **사용 가능 IP**: `1.65 ~ 1.94` (30개) - *딱 맞음*

#### ③ C그룹 할당 (16대 필요)
- 남아있는 `192.168.1.96`부터 시작합니다.
- **주의**: 16대를 수용하려면 $2^4 = 16$으로는 부족합니다 ($16 - 2 = 14$개만 사용 가능하므로). 따라서 한 단계 위인 $2^5 = 32$개를 할당해야 합니다.
- **호스트 비트**: 5비트 필요.
- **서브넷 마스크**: $32 - 5 = /27$
- **범위**: `192.168.1.96 ~ 192.168.1.127`
- **사용 가능 IP**: `1.97 ~ 1.126` (30개)

---

> [!INFO] 💡 AI 보충 설명: 서브네팅 계산 꿀팁
> 1. **블록 사이즈**: 서브넷 마스크 숫자를 $256$에서 빼면 해당 서브넷의 크기가 나옵니다. 
>    - 예: `/27`은 $255.255.255.224$이므로, $256 - 224 = 32$가 한 블록의 크기입니다.
> 2. **CIDR 표기법**: `/24`는 앞에 1이 24개 있다는 뜻이며, $255.255.255.0$과 같습니다. 실기 시험에서는 10진수와 비트수를 빠르게 변환하는 것이 중요합니다.

## 4. [[IPv6]] (Internet Protocol version 6)
- **특징**: 128비트 확장, 보안/QoS 개선, 주소 자동 설정([[Auto-configuration]]).
- **주소 형식**: 16비트씩 8부분, 콜론(`:`)으로 구분. 16진수 표기.
- **전송 방식**:
    - **[[Unicast]]**: 1:1 전송.
    - **[[Multicast]]**: 1:N 전송.
    - **[[Anycast]]**: 가장 가까운 노드 1개와 전송. (IPv4의 브로드캐스트 대체 성격)

## 🎯 빈출 키워드
- **[[ARP/RARP]]**: IP와 MAC 주소 간의 상호 변환 프로토콜.
- **[[ICMP]]**: Ping 테스트 등에 사용되는 제어 메시지 프로토콜.
- **[[TTL]]**: 패킷의 수명을 제한하여 무한 루프를 방지하는 값.
- **[[서브넷 마스크]]**: 네트워크 영역과 호스트 영역을 구분하는 기준.
- **[[IPv6]]**: 128비트 주소 체계로 애니캐스트(Anycast)를 지원함.
