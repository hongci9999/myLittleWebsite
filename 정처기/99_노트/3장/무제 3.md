# 데이터베이스 정규화 (Normalization) 상세 분석

## 📌 핵심 요약
정규화는 데이터 구조의 **유연성**을 높이고 **중복을 제거**하여, 데이터 조작 시 발생하는 **[[이상 현상(Anomaly)]]**을 방지하는 필수 과정입니다. 함수적 종속성($X \to Y$)에 근거하여 테이블을 무손실 분해합니다.

---

## 📝 상세 정리

### 1. 함수적 종속 및 암스트롱 규칙 (Armstrong's Axioms)
단순 메모를 넘어 시험에 나오는 추론 규칙을 구체화합니다.

#### 1.1 기본 규칙 (반드시 암기)
* **반사(Reflexive):** $Y \subseteq X$ 이면 $X \to Y$ 이다. (당연한 포함 관계)
* **증가(Augmentation):** $X \to Y$ 이면 $XZ \to YZ$ 이다. (양변에 같은 속성 추가 가능)
* **이행(Transitive):** $X \to Y, Y \to Z$ 이면 $X \to Z$ 이다. (가장 빈출되는 정규화 대상)

#### 1.2 보조 규칙 (유도 가능)
* **결합(Union):** $X \to Y, X \to Z$ 이면 $X \to YZ$ 이다.
* **분해(Decomposition):** $X \to YZ$ 이면 $X \to Y, X \to Z$ 이다.
* **의사이행(Pseudo-transitive):** $X \to Y, WY \to Z$ 이면 $WX \to Z$ 이다.

---

### 2. 정규화 단계별 사례 (Example-driven)

#### 1NF: 제1정규형 (원자값 확보)
* **대상:** 한 칸에 여러 값이 들어있는 경우 (예: '취미' 컬럼에 '축구, 독서'가 같이 있음)
* **해결:** 중복된 행을 만들더라도 모든 속성을 **원자값(Atomic Value)**으로 분리.

#### 2NF: 제2정규형 (부분 함수 종속 제거)
* **대상:** 기본키가 **복합키(속성A + 속성B)**일 때, 일반 속성이 키의 **일부(속성A)**에만 종속된 경우.
* **예시:** `[학번, 과목코드]`가 기본키인데, `이름` 속성이 `학번`에만 종속되어 있다면 분리.
* **공식:** $2NF = 1NF - \text{Partial Functional Dependency}$



#### 3NF: 제3정규형 (이행 함수 종속 제거)
* **대상:** 기본키가 아닌 속성 간에 종속성이 존재하는 경우.
* **예시:** `학번(A) → 학과코드(B) → 학과명(C)` 관계에서 `학과명`은 `학번`에 이행적으로 종속됨.
* **해결:** `[학번, 학과코드]` 테이블과 `[학과코드, 학과명]` 테이블로 분리.
* **공식:** $3NF = 2NF - \text{Transitive Functional Dependency}$

#### BCNF: 보이스-코드 정규형
* **대상:** 모든 결정자가 **후보키**가 아닌 경우.
* **예시:** 수강 테이블에서 `[학생, 과목] → 교수`인데, 특정 `교수`가 한 과목만 담당하여 `교수 → 과목` 종속성이 생길 때, `교수`는 결정자이지만 후보키가 아니므로 분리.



---

### 3. 이상 현상 (Anomaly) 상세
정규화를 안 했을 때 발생하는 3대 비극입니다.

1.  **삽입 이상:** 학과에 학생이 없으면 학과 정보를 입력하지 못함 (불필요한 데이터 강제 삽입).
2.  **삭제 이상:** 학생 한 명을 자퇴 처리했더니 학과 정보 자체가 증발함 (연쇄 삭제).
3.  **갱신 이상:** 학과 이름이 바뀌었는데 일부 학생 데이터만 수정되어 데이터 불일치 발생.

---

### 4. 논리 데이터 모델 품질 검증 항목 (메모 보강)
데이터 모델링의 마지막 단계에서 체크할 '객관적 기준'입니다.

* **완전성(Completeness):** 누락된 업무 요구사항이 없는가?
* **중복 배제(Non-redundancy):** 데이터의 유일성을 확보했는가?
* **안정성(Stability):** 비즈니스가 바뀌어도 구조가 크게 흔들리지 않는가?
* **간결성(Conciseness):** 가장 단순하고 효율적인 형태로 설계되었는가?
* **준거성(Compliance):** 데이터 표준화 지침과 법적 제약(개인정보 등)을 준수했는가?

> [!INFO] 💡 AI 보충 설명: 정규화의 역설
> 정규화는 무조건 많이 할수록 좋을까요? 아닙니다. 정규화가 심화될수록 테이블이 잘게 쪼개져 조인($Join$) 연산이 늘어납니다. 이는 **조회 성능 저하**를 초래할 수 있으므로, 실무에서는 성능을 위해 고의로 정규화를 무시하는 **[[역정규화(Denormalization)]]**를 수행하기도 합니다.
