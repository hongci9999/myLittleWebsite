# 객체지향 기술 및 디자인 패턴

## 📌 핵심 요약
현실 세계의 개체를 데이터(속성)와 동작(메서드)이 결합된 **[[객체]]**로 모델링하는 방법론을 학습합니다. 객체지향의 5대 원칙(**[[SOLID]]**)과 설계의 효율성을 높이는 **[[GoF 디자인 패턴]]** 23가지를 생성, 구조, 행위별로 구분하여 정립하는 것이 중요합니다.

---

## 📝 상세 정리

### 1. 객체지향(Object-Oriented) 개요 및 구성요소
- **개요**: 현실 세계의 개체(Entity)를 속성(Attribute)과 메서드(Method)가 결합된 형태의 디지털 객체로 대응하여 표현하는 방법론입니다.
- **구성요소**:
    - **[[클래스]] (Class)**: 객체를 생성하기 위한 틀(Blueprint)이자 공통된 특성을 추상화한 단위입니다.
    - **[[객체]] (Object)**: 클래스에 의해 구현된 실체로, 고유한 속성을 가지며 정의된 메서드를 수행합니다.
    - **[[인스턴스]] (Instance)**: 특정 클래스로부터 메모리에 할당된 실제 객체를 의미합니다.
    - **[[메시지]] (Message)**: 객체 간의 통신 및 상호작용을 위한 인터페이스입니다.
    - **[[메서드]] (Method)**: 객체가 수행하는 연산이자 기능입니다.

### 2. 객체지향 기법 (Key Concepts)
- **[[캡슐화]] (Encapsulation)**: 데이터(속성)와 기능을 하나로 묶어 인터페이스를 단순화하고 재사용성을 높입니다.
- **[[정보 은닉]] (Information Hiding)**: 내부 구현을 외부에 노출하지 않아 외부의 영향으로부터 객체를 보호합니다.
- **[[추상화]] (Abstraction)**: 불필요한 부분을 생략하고 객체의 속성 중 가장 중요한 공통 특징을 추출합니다.
- **[[상속]] (Inheritance)**: 상위 클래스의 속성과 메서드를 하위 클래스가 물려받아 코드의 재사용성을 높입니다.
- **[[다형성]] (Polymorphism)**: 동일한 메시지에 대해 여러 객체가 서로 다른 방식으로 반응하는 성질입니다.
    - **[[오버로딩]] (Overloading)**: 동일 이름의 메서드이나 매개변수의 타입/개수를 다르게 하여 정의함.
    - **[[오버라이딩]] (Overriding)**: 상위 클래스의 메서드를 하위 클래스에서 재정의함.

### 3. 객체지향 분석 및 설계 방법론
- **[[Rumbaugh]] (럼바우) 방법론**: 가장 일반적으로 사용되며, 모든 소프트웨어 구성 요소를 그래픽 표기법으로 모델링합니다.
    1. **객체 모델링 (Object Modeling)**: 객체 다이어그램(Information) 활용. 가장 중요하고 선행됨.
    2. **동적 모델링 (Dynamic Modeling)**: 상태 다이어그램(Control)을 활용하여 시간의 흐름에 따른 상태 변화 기술.
    3. **기능 모델링 (Functional Modeling)**: 자료 흐름도(DFD)를 활용하여 데이터 변환 과정 기술.
- **기타 방법론**:
    - **[[Booch]] (부치)**: 미시적(Micro) 및 거시적(Macro) 개발 프로세스 사용.
    - **[[Jacobson]]**: 사용 사례(Use Case) 기반의 시나리오 강조.
    - **[[Coad]]-Yourdon**: E-R 다이어그램을 활용하여 객체 모델링.
    - **[[Wirfs-Brock]]**: 분석과 설계의 구분이 모호하며 고객과의 의사소통 강조.

### 4. 객체지향 설계 원칙 (SOLID)
- **[[SRP]] (단일 책임 원칙)**: 클래스는 하나의 문제(책임)만 해결해야 함. (응집도↑, 결합도↓)
- **[[OCP]] (개방 폐쇄 원칙)**: 확장은 열려 있어야 하고, 수정에는 닫혀 있어야 함.
- **[[LSP]] (리스코프 치환 원칙)**: 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 함.
- **[[ISP]] (인터페이스 분리 원칙)**: 사용하지 않는 메서드에 의존하지 않도록 인터페이스를 구체적으로 분리.
- **[[DIP]] (의존성 뒤집기 원칙)**: 상위 모듈이 하위 모듈에 의존하지 않고, 둘 다 추상화에 의존해야 함.

## 📝 GoF 디자인 패턴 (Design Patterns)

### 1. 생성 패턴 (Creational Patterns)
- **[[Factory Method]]**: 객체 생성을 서브 클래스에서 결정하도록 위임합니다.
    - //요청응답: 쉽게 말해, 부모는 "붕어빵을 만든다"는 틀만 정하고, 실제 "팥"을 넣을지 "슈크림"을 넣을지는 자식 클래스가 결정하는 방식입니다.
- **[[Abstract Factory]]**: 구체적인 클래스에 의존하지 않고 서로 연관된 객체들의 조합을 만드는 인터페이스를 제공합니다.
    - //요청응답: "가구 공장" 인터페이스가 있다면, "모던 가구 세트"나 "클래식 가구 세트"처럼 연관된 제품군을 통째로 생성해주는 방식입니다.
- **[[Builder]]**: 복잡한 인스턴스를 생성하는 단계와 표현 방법을 분리하여 동일한 절차에서 다른 결과를 만듭니다.
- **[[Prototype]]**: 원형이 되는 인스턴스를 복사(Copy)하여 새로운 객체를 생성합니다.
- **[[Singleton]]**: 클래스의 인스턴스가 오직 하나만 존재하도록 보장합니다.

### 2. 구조 패턴 (Structural Patterns)
- **[[Adapter]]**: 호환되지 않는 인터페이스를 가진 클래스들이 함께 작동하도록 변환합니다. (220V 플러그를 110V로 바꿔주는 돼지코 역할)
- **[[Bridge]]**: 구현부에서 추상층을 분리하여 각각 독립적으로 확장할 수 있게 합니다.
- **[[Composite]]**: 객체들의 관계를 트리 구조로 구성하여 단일 객체와 복합 객체를 동일하게 다룹니다.
- **[[Decorator]]**: 객체에 동적으로 새로운 서비스를 추가합니다.
- **[[Facade]]**: 복잡한 서브 시스템에 대해 단순화된 통합 인터페이스를 제공합니다.
- **[[Flyweight]]**: 많은 수의 소형 객체들을 공유하여 메모리를 절약합니다.
- **[[Proxy]]**: 실제 객체에 대한 접근 권한 제어나 대리 역할을 수행합니다.

### 3. 행위 패턴 (Behavioral Patterns)
- **[[Interpreter]]**: 언어의 문법 규칙을 정의하고 해석합니다.
- **[[Template Method]]**: 알고리즘의 구조(뼈대)는 상위에서 정의하고 구체적 단계는 하위에서 구현합니다.
- **[[Chain of Responsibility]]**: 요청을 처리할 수 있는 객체들이 연결되어 있어 순서대로 요청을 처리합니다.
- **[[Command]]**: 요청을 객체로 캡슐화하여 요청을 큐에 저장하거나 로그를 남기고 취소(Undo)할 수 있게 합니다.
- **[[Iterator]]**: 내부 구조를 노출하지 않고 집합 객체의 요소들에 순차적으로 접근합니다.
- **[[Mediator]]**: 객체들 간의 복잡한 상호작용을 중재자 객체로 캡슐화하여 결합도를 낮춥니다.
- **[[Memento]]**: 객체의 상태를 저장하여 나중에 해당 상태로 복구(Rollback)할 수 있게 합니다.
- **[[Observer]]**: 한 객체의 상태가 변하면 의존하는 다른 객체들에게 알림을 보냅니다.
- **[[State]]**: 객체의 내부 상태에 따라 스스로 동작을 변경하도록 합니다.
- **[[Strategy]]**: 동일 계열의 알고리즘들을 캡슐화하여 상호 교체가 가능하게 만듭니다.
- **[[Visitor]]**: 데이터 구조와 처리 알고리즘을 분리하여 새로운 연산을 추가하기 쉽게 합니다.

> 2. **SOLID 원칙의 목적**: 소프트웨어의 가독성을 높이고, 유지보수 비용을 줄이며, 변화에 유연하게 대처하기 위함입니다.
> 3. **생성/구조/행위 구분**: 시험에서는 특정 패턴이 어느 그룹에 속하는지 묻는 문제가 매우 자주 출제됩니다. (예: 다음 중 구조 패턴이 아닌 것은?)

## 🎯 빈출 키워드
- **[[Rumbaugh]] 모델링**: 객체(객체 다이어그램), 동적(상태 다이어그램), 기능(자료 흐름도) 순서.
- **[[캡슐화]]와 [[정보은닉]]**: 데이터와 기능을 하나로 묶고 내부를 보호함.
- **[[SOLID]] 원칙**: SRP, OCP, LSP, ISP, DIP 5가지 설계 원칙.
- **[[Singleton]]**: 인스턴스를 단 하나만 생성하여 공유하는 생성 패턴.
- **[[Observer]]**: 상태 변화를 감시하고 통지하는 행위 패턴.
- **[[다형성]]**: 오버로딩(매개변수 차이)과 오버라이딩(상속 재정의).
