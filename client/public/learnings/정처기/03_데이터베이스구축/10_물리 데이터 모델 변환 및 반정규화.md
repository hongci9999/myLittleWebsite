# 물리 데이터 모델 변환 및 반정규화

## 📌 핵심 요약
논리 모델을 물리 모델로 변환하는 과정에서의 **개체/관계 변환 전략**과 시스템 성능 향상을 위한 **[[반정규화]](De-normalization)** 기법을 다룹니다. 특히 슈퍼타입/서브타입 변환 기준과 테이블 분할/통합을 통한 최적화 방안을 상세히 정리합니다.

---

## 📝 상세 정리

## 1. 물리 데이터 모델 변환

### 1.1 개체(Entity)의 테이블 변환
기본키([[PK]])와 외래키([[FK]])를 통해 다른 테이블과의 관계를 설정합니다.
- **식별자 매핑:**
    - **식별자(Identifier):** [[후보키]](Candidate Key)로 변환
    - **주식별자:** [[기본키]](Primary Key)로 변환
    - **보조 식별자:** [[대체키]](Alternate Key)로 변환
    - **외래 식별자:** [[외래키]](Foreign Key)로 변환

### 1.2 슈퍼타입/서브타입 모델 변환

- **슈퍼타입(Super-type) 기준 변환 (Single Table):** 서브타입을 슈퍼타입에 통합하여 하나의 테이블로 만듭니다.
- **서브타입(Sub-type) 기준 변환 (Many Table):** 슈퍼타입의 속성들을 각각의 서브타입에 추가하여 서브타입별로 테이블을 만듭니다. (서브타입에 많은 양의 속성이나 관계가 있는 경우 유리)
- **개별 타입 기준 변환 (One-to-One Table):** 슈퍼타입과 서브타입을 각각 별도의 테이블로 변환합니다.
- **변환 선택 기준 (통합이 필요한 경우):**
    - 전체 데이터에 대한 처리가 자주 발생하는 경우
    - 통합하는 테이블의 컬럼 수가 지나치게 많은 경우
    - 트랜잭션이 주로 슈퍼타입에서 발생하는 경우

### 1.3 관계 변환 (Relationship Transformation)
- **1:M 관계 변환:** 가장 보편적인 형태로, 1측의 기본키를 M측의 외래키로 추가합니다.
- **1:1 관계 변환:** 두 테이블 중 하나를 선택하여 상대방의 기본키를 외래키로 추가합니다.
- **1:M 순환 관계 변환:** 동일 테이블 내에서 자신의 기본키를 참조하는 외래키 컬럼을 추가하여 데이터의 계층 구조(Tree 구조)를 표현합니다.
- **배타적 관계 변환:** 한 개체가 둘 이상의 다른 개체 중 하나와만 관계를 맺는 경우입니다.

> [!INFO] 💡 AI 보충 설명: 배타적 관계 변환 예시
> 사원 테이블이 '정규직' 혹은 '계약직' 테이블 중 하나와만 관계를 맺을 때:
> 1. **외래키 분리:** `정규직_ID`, `계약직_ID` 컬럼을 각각 생성 (관계별 칼럼 생성)
> 2. **외래키 결합:** `참조_ID`와 `구분_코드` 컬럼을 생성하여 하나의 컬럼으로 통합 관리

### 1.4 컬럼 변환 및 데이터 표준 적용
- **관리 목적 컬럼 추가:** 시스템 관리나 업무 수행 속도 향상을 위해 논리 모델에 없던 컬럼을 추가합니다.
- **데이터 명명 규칙:** 논리명을 물리적 영문명으로 전환하며, 영문 약어를 사용하여 가독성과 표준을 준수합니다.

## 2. 데이터베이스 반정규화 ([[De-normalization]])

### 2.1 개념 및 목적
시스템 운영의 단순화를 위해 정규화된 모델을 의도적으로 중복, 통합, 분할하는 기법입니다.
- **목적:** 조회 성능 향상.
- **대상:** - 다량의 범위나 특정 범위 데이터를 자주 처리하는 경우.
    - 집계 정보(Sum, Avg 등)가 자주 요구되는 경우.

### 2.2 주요 유형
- **중복 데이터 추가:** 자주 사용되는 데이터를 중복 저장하여 조인 횟수를 줄입니다.
- **테이블 조합:** 조인 성능 저하를 막기 위해 둘 이상의 테이블을 하나로 합칩니다.
- **테이블 분할:** 컬럼의 사용 빈도가 극명하게 갈리는 경우 수직/수평 분할을 수행합니다.
- **테이블 제거:** 더 이상 사용하지 않는 테이블을 삭제합니다. (관리 소홀이 주된 원인)
- **컬럼의 중복화:** 서로 다른 테이블에 분산된 자주 쓰는 컬럼들을 한 곳에 모읍니다. 단, 데이터 무결성 유지를 위해 다른 방법(인덱스 등)을 우선 검토해야 합니다.

---

## 3. 시스템 카탈로그 및 타입 변환

### 3.1 [[시스템 카탈로그]] (System Catalog)
- **정의:** 데이터베이스 객체(테이블, 뷰, 인덱스 등)에 대한 정의와 명세를 메타데이터 형태로 유지 관리하는 **데이터 사전**.
- **특징:** DBMS가 스스로 생성하고 유지하며, 사용자가 직접 `UPDATE` 등으로 변경할 수 없습니다. `DDL`을 통해 객체가 변경되면 DBMS에 의해 자동 갱신됩니다.

### 3.2 데이터 타입 변환
- **묵시적 형변환:** 서로 다른 타입 비교 시 우선순위에 따라 자동 변환됩니다.
- **주의사항:** 문자열과 숫자 비교 시 숫자의 우선순위가 높아 문자열이 숫자로 변환됩니다. 이때 문자열 컬럼에 인덱스가 있어도 형변환으로 인해 인덱스를 타지 못할 수 있으므로 주의해야 합니다.
