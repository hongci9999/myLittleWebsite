# 관계와 조인 — 계층형·상호배타적 관계

## 핵심 요약

**조인**은 관계가 있는 테이블을 SQL로 **연결해 조회**하는 방법이다. 정규화로 테이블이 쪼개지면 **조인 비용**이 커져 **반정규화**를 검토하기도 한다. **계층형 데이터 모델**은 같은 테이블이 **자기 자신을 참조**(Self-Join, **상사ID → 직원ID**). **상호배타적 관계**는 자식이 둘 이상 부모 중 **하나에만** 연결되며(개인고객 **또는** 법인고객), ERD에서는 **호(Arc)·XOR** 로 표기한다.

## 왜 중요한가

- 정규화(분해)와 **조인 성능·반정규화**의 연결을 묻는다.
- 조직도·상하 관계는 **자기 참조 FK** 설계·계층 쿼리로 이어진다.
- 개인/법인 주문처럼 **XOR** 는 **고객구분코드 + FK** 설계 문제로 출제된다.

## 1. 조인 (Join)

| 항목 | 내용 |
|------|------|
| **정의** | **관계가 있는 테이블**을 SQL로 **연결**하여 데이터를 조회하는 방법 |
| **정규화와의 관계** | 정규화로 **분해된 테이블**은 조회 시 **조인**이 늘어남 |
| **성능** | 조인이 많으면 성능이 **저하**될 수 있음 → 필요 시 **반정규화**로 테이블을 다시 합침 |

```mermaid
flowchart LR
  N[정규화 분해] --> J[조인 증가]
  J --> P[성능 이슈]
  P --> D[반정규화 검토]
```

> SQL 문법(INNER, LEFT 등)은 **2과목 SQL**에서 다룸. 여기서는 **모델·설계 관점**의 조인이다.

## 2. 계층형 데이터 모델 (자기 참조)

### 개념

| 항목 | 내용 |
|------|------|
| **정의** | 테이블이 **자기 자신의 컬럼**을 참조하는 관계 |
| **구현** | **Self-Join**, **계층 쿼리**(DBMS별 계층 질의) |
| **용도** | 조직도, BOM, 카테고리 트리 등 |

### 직원 테이블 예

| 직원ID (PK) | 이름 | 직위 | 상사ID (FK→직원ID) |
|-------------|------|------|---------------------|
| 1 | 박이사 | 대표이사 | **NULL** |
| 2 | 이부장 | 부장 | 1 |
| 3 | 김과장 | 과장 | 2 |
| 4 | 서대리 | 대리 | 3 |
| 5 | 차대리 | 대리 | 3 |

**자기 자신 참조:** `상사ID` → `직원ID`

### 조직 계층 (개념도)

```
박이사 (CEO)
  └ 이부장
      └ 김과장
          ├ 서대리
          └ 차대리
```

### SQL 관점 (참고)

- 같은 테이블을 **별칭**으로 두 번 조인: `직원 e` · `직원 m` ON `e.상사ID = m.직원ID`
- 최상위: `상사ID IS NULL`

## 3. 상호배타적 관계 (Mutually Exclusive / XOR)

### 개념

| 항목 | 내용 |
|------|------|
| **정의** | 한 엔터티(자식)가 **둘 이상의 부모 테이블 중 하나에만** 연관 |
| **ERD 표기** | 두 관계선 위 **호(Arc)** — **배타적 OR(XOR)** |
| **의미** | 개인고객 **또는** 법인고객 — **동시에 둘 다는 불가** |

### E-R 예: 고객 · 주문

| 엔터티 | 속성 |
|--------|------|
| **개인고객** | 고객번호, 개인고객명 |
| **법인고객** | 법인번호, 법인명 |
| **주문** | 주문번호, **고객구분코드**, **개인/법인번호(FK)** |

```
개인고객 ──┐
           ├──( XOR )── 주문
법인고객 ──┘
```

| 필드 | 역할 |
|------|------|
| **고객구분코드** | FK가 **개인**인지 **법인**인지 구분 (**식별자/판별자**) |
| **개인/법인번호** | 구분에 따라 **해당 부모의 PK** 참조 |

### 설계 시 유의

- FK 하나 + **구분 코드**로 다형 참조를 표현하는 패턴이 흔함
- 두 FK를 동시에 채우지 않도록 **업무 규칙·제약** 필요

## 4. 개념 비교

| 구분 | 계층형(자기참조) | 상호배타적(XOR) |
|------|------------------|-----------------|
| 테이블 수 | **1개** 테이블 내부 | **2개 이상** 부모 |
| 참조 | **자기 PK** | **부모 PK 중 하나** |
| 예 | 직원–상사 | 개인고객 vs 법인고객 → 주문 |
| SQL | Self-Join | 구분코드 + 조건부 FK |

## 5. 시험 포인트 / 함정

| 구분 | 내용 |
|------|------|
| 조인 | 관계 테이블 **연결 조회** — 정규화 ↑ → 조인 ↑ |
| 반정규화 | 조인 성능 등 때문에 **의도적 중복** 허용 |
| 계층형 | **상사ID → 직원ID**, 최상위 **NULL** |
| 상호배타 | **둘 중 하나만** — 호(Arc)·XOR |
| 주문 예 | **고객구분코드** + **개인/법인번호** |
| 함정 | XOR를 「둘 다 동시 연결 가능」으로 해석 |
| 함정 | 계층형을 **서로 다른 두 테이블** 조인으로 혼동 |
