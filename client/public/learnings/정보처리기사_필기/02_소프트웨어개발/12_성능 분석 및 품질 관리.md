# 애플리케이션 성능 분석 및 품질 관리

## 핵심 요약
애플리케이션 성능 측정을 위한 **4대 지표(처리량, 응답 시간, 경과 시간, 자원 사용률)**와 분석 도구를 학습합니다. 성능 저하 원인을 파악하여 **[[클린 코드]]**와 **[[리팩토링]]**을 통해 개선하며, **[[정형 기술 검토(FTR)]]**와 **[[소프트웨어 품질 목표]]**를 통해 시스템의 신뢰성과 가용성을 확보하는 과정을 다룹니다.

---

## 상세 정리

### 1. 애플리케이션 성능 측정 및 분석
#### 성능 측정 지표
- **처리량 (Throughput):** 일정 시간 내에 처리하는 업무량
- **응답 시간 (Response Time):** 요청 입력 후 응답이 시작될 때까지의 시간
- **경과 시간 (Turnaround Time):** 요청 입력 후 결과 출력이 완료될 때까지의 시간
- **자원 사용률 (Resource Usage):** 작업 처리 시 사용하는 CPU, 메모리, 네트워크 사용량

#### 성능 분석 도구
- **성능/부하/스트레스 점검 도구:** JMeter, LoadUI, OpenSTA
- **모니터링 도구:** Scouter, Zabbix

### 2. 성능 저하 원인 분석
- **DB 연결 및 쿼리 실행:** DB Lock(대량 데이터 조회 및 업데이트), DB Fetch(과도한 데이터 요청 시 응답 저하)
- **내부 로직:** 인터넷 접속 불량, 대량/대용량 파일 처리, 부적절한 예외 처리
- **외부 호출:** 타임아웃(Timeout) 발생
- **잘못된 환경:** 힙 메모리 크기 등 환경 설정이 너무 작음, 전송 실패 및 지연

### 3. 성능 개선 및 소스 코드 최적화
- **배드 코드 (Bad Code):** **스파게티 코드**(로직 복잡도 증가), 외계인 코드 등
- **[[클린 코드]] (Clean Code):** 가독성이 높고 의존성이 낮으며 중복이 최소화된 코드
- **최적화 기법:** 응집도 향상(클래스당 하나의 역할), 의존성 최소화, 올바른 코드 스타일 준수, 좋은 이름 사용, 주석문 활용

#### 소스 코드 품질 분석 도구
- **정적 분석 도구 (실행 X):** PMD, cppcheck, SonarQube, Checkstyle
- **동적 분석 도구 (실행 O):** Avalanche, Valgrind
- **기타 기법:** [[코드 인스팩션]], 증명(Proof), [[리팩토링]](기능 변경 없이 구조 개선)

### 4. 유지보수 및 품질 보증
#### 유지보수 (Maintenance)
- **회귀 테스트 (Regression Test):** 수정 후 부작용(코드, 데이터, 문서 부작용) 발생 여부 확인
- **유형:**
  1. **하자 보수:** 버그 및 오류 제거
  2. **완전 보수:** 기능 추가 및 성능 개선 (가장 큰 비용)
  3. **적응 보수:** 환경 변화에 대응
  4. **예방 보수:** 미래 요구 예측 및 유지보수성 향상

#### [[정형 기술 검토]] (FTR) 원칙 (원문 10가지 100% 반영)
1. 제품의 검토에만 집중하라.
2. 사전 준비를 강요하라.
3. 검토자 훈련을 실시하라.
4. 검토될 제품에 대한 체크 리스트를 개발하라.
5. 자원과 시간 일정을 할당하라.
6. 문제 영역을 명확히 표현하고 의제를 제한하라.
7. 논쟁과 반박을 제한하라.
8. 참가자 수를 제한하라.
9. 검토의 과정과 결과를 재검토하라.
10. **개선책에 대해 논하지 마라.**

#### 소프트웨어 품질 목표 (McCall)
- 정확성, 신뢰성, 효율성, 무결성, 유지보수 용이성, 사용 용이성, 검사 용이성, 이식성, 상호 운용성, 유연성, 재사용성

### 5. 신뢰도 및 가용성 계산
- **평균 무장애 시간 ($MTBF$):** $MTBF = MTTF + MTTR$
- **평균 장애 시간 ($MTTF$):** 고장 발생 전까지 동작 시간 평균
- **평균 복구 시간 ($MTTR$):** 수리 완료까지 소요 시간 평균
- **가용성 (Availability):** $\frac{MTTF}{MTBF}$
