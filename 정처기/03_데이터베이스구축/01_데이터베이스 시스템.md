# 데이터베이스 시스템 및 관계형 데이터베이스의 이해

## 핵심 요약
본 강의는 **데이터베이스 관리 시스템(DBMS)**의 정의와 특징, 데이터 독립성을 보장하기 위한 **3층 스키마 구조**를 다룹니다. 또한, 현대 데이터베이스의 주류인 **관계형 데이터베이스(RDB)**의 구성 요소(개체, 속성, 도메인, 튜플)와 릴레이션의 특성, 그리고 데이터를 식별하기 위한 다양한 **키(Key)**의 개념을 상세히 설명합니다.

---

## 상세 정리

### 1. 데이터베이스 시스템 (Database System)
데이터를 통합하여 저장하고 운영하는 시스템으로, 다음과 같은 데이터 특성을 가집니다.

* **공유 데이터 (Shared Data):** 여러 응용 프로그램이 공통으로 필요로 하는 데이터입니다.
* **통합 데이터 (Integrated Data):** 중복을 최소화하여 불필요한 데이터 제거한 상태입니다.
* **저장 데이터 (Stored Data):** 컴퓨터가 접근 가능한 저장 매체에 저장된 데이터입니다.
* **운영 데이터 (Operational Data):** 조직의 목적을 위해 필수적이고 지속적으로 필요한 데이터입니다.

#### DB 시스템의 특징
- **실시간 접근성 (Real-Time Accessibility):** 쿼리에 대해 즉각적인 처리와 응답이 가능합니다.
- **계속적인 진화 (Continuous Evolution):** 삽입, 삭제, 갱신을 통해 항상 최신 데이터를 유지합니다.
- **동시 공유 (Concurrent Sharing):** 여러 사용자가 동시에 같은 데이터에 접근할 수 있습니다.
- **내용에 의한 참조 (Content Reference):** 주소가 아닌 데이터의 값(내용)으로 참조합니다.
- **논리적 독립성 (Logical Data Independence):** 응용 프로그램과 데이터베이스의 논리적 구조가 서로 영향을 주지 않습니다.
- **기타:** 중복 최소화, **[[무결성]]** 유지, 보안성, 안정성, 효율성 확보가 중요합니다.

---

### 2. 스키마 (Schema)
데이터베이스의 구조와 제약조건에 관해 공식적으로 정의한 것으로, **[[데이터 사전]] (Data Dictionary)**에 저장됩니다.

* **외부 스키마 (External Schema):**
    - 사용자나 응용 프로그래머의 관점에서 본 데이터베이스 구조입니다.
    - 여러 개가 존재할 수 있으며, 전체 DB의 일부분인 '서브 스키마'라고도 합니다.
* **개념 스키마 (Conceptual Schema):**
    - 모든 사용자의 관점을 통합한 전체 데이터베이스의 논리적 구조입니다.
    - 접근 권한, 보안 정책, 무결성 규칙을 정의합니다.
    - **[[DBA]]**에 의해 구성되며, 데이터베이스당 **하나만 존재**합니다.
* **내부 스키마 (Internal Schema):**
    - 물리적 저장장치 관점에서의 구조입니다.
    - 실제로 데이터가 저장되는 방식, 인덱스, 운영체제의 파일 시스템과의 연결 등을 정의합니다.

---

### 3. 데이터베이스 관리 시스템 (DBMS)
사용자와 데이터베이스 사이에서 인터페이스 역할을 하며 상호작용을 제어하는 소프트웨어입니다.

#### DBMS의 주요 구성 요소
- **질의어 처리기 (Query Processor):** 사용자의 SQL 쿼리를 분석하여 처리합니다.
- **DML 컴파일러:** 응용 프로그램 내의 데이터 조작어(DML)를 분석합니다.
- **데이터베이스 처리기:** DB 연산을 저장 데이터 관리자에 전송합니다.
- **DDL 컴파일러:** DBA가 정의한 스키마(DDL)를 분석하여 메타데이터로 저장합니다.
- **저장 데이터 관리자:** 실제 디스크에 저장된 데이터 파일에 접근합니다.



#### DBMS의 필수 기능
1.  **정의 기능 (Definition):** 데이터의 논리적/물리적 구조를 정의하고 매핑을 제공합니다.
2.  **조작 기능 (Manipulation):** 데이터의 검색(Select), 삽입(Insert), 삭제(Delete), 갱신(Update) 기능을 제공합니다.
3.  **제어 기능 (Control):** **무결성 유지, 보안 및 권한 검사, 병행 제어(동시성 제어), 회복** 등을 수행합니다.

---

### 4. 관계형 데이터베이스 (Relational Database)

#### DB 시스템 유형의 변천
- **파일 시스템:** 파일 단위 관리, 데이터 중복 및 종속성 문제가 큼.
- **HDBMS (계층형):** 트리 구조, 1:N 관계, 접근은 빠르나 유연성 부족.
- **NDBMS (망형):** 그래프 구조, N:M 관계 표현 가능하나 복잡함.
- **RDBMS (관계형):** 테이블(2차원 구조) 기반, 논리적 구조가 명확하고 적응력이 높으나 시스템 부하가 상대적으로 큼.

#### 관계형 DB의 핵심 구성 요소
- **개체 타입 (Entity Type):** 현실 세계의 대상(사람, 사물 등)을 논리적으로 표현한 단위입니다. (RDB의 테이블)
- **속성 (Attribute):** 개체의 특성을 나타내는 가장 작은 논리적 단위입니다. (테이블의 열/Column)
- **도메인 (Domain):** 하나의 속성이 가질 수 있는 **원자 값(Atomic Value)**들의 집합입니다. (예: 성별은 '남', '여'만 가능)
- **튜플 (Tuple):** 릴레이션을 구성하는 각각의 행입니다. (레코드/Record)
- **릴레이션 (Relation):** 튜플들의 집합으로 이루어진 테이블 전체입니다.
    - **차수 (Degree):** 속성(Column)의 전체 개수입니다.
    - **카디널리티 (Cardinality/기수):** 튜플(Row)의 전체 개수입니다.
- **NULL:** "알 수 없음" 혹은 "해당 없음"을 의미하는 특수 값입니다.

> [!INFO] 💡 RDB 구조 예시 (학생 테이블)
> | 학번 (속성1) | 이름 (속성2) | 성별 (속성3/도메인: 남,여) |
> | :--- | :--- | :--- |
> | 2024001 | 홍길동 | 남 |
> | 2024002 | 김철수 | 남 |
> 
> - 위 테이블에서 **차수(Degree)**는 3이며, **기수(Cardinality)**는 2입니다.
> - 각 가로 줄 하나가 **튜플(Tuple)**입니다.

---

### 5. 키 (Key)
튜플을 유일하게 구별하거나 릴레이션 간의 관계를 정의하기 위한 속성입니다.

* **후보키 (Candidate Key):**
    - 튜플을 유일하게 식별할 수 있는 속성들입니다.
    - **유일성**(Unique)과 **최소성**(Minimality)을 모두 만족해야 합니다.
* **기본키 (Primary Key):**
    - 후보키 중에서 DBA가 최종적으로 선택한 대표 키입니다.
    - **NULL 값을 가질 수 없으며(Entity Integrity)** 중복될 수 없습니다.
* **대체키 (Alternate Key):**
    - 후보키 중 기본키를 제외한 나머지 키입니다.
* **슈퍼키 (Super Key):**
    - 유일성은 만족하지만, **최소성은 만족하지 못하는** 키입니다. (예: 학번 + 이름 조합)
* **외래키 (Foreign Key):**
    - 다른 릴레이션의 기본키를 참조하는 속성입니다. 릴레이션 간의 관계를 맺어줍니다.

## 3. 빈출 내용 정리
- **[[DB 시스템]] 특징:** 실시간 접근성, 계속적인 진화, 동시 공유, 내용에 의한 참조.
- **[[스키마]] 3계층:** 외부(사용자), 개념(논리/전체/DBA), 내부(물리/저장장치).
- **[[DBMS]] 필수 기능:** 정의, 조작, 제어 (정조제).
- **[[릴레이션]] 용어:** Degree(열의 수), Cardinality(행의 수).
- **[[키]]의 성질:** - 후보키: 유일성 O, 최소성 O
    - 슈퍼키: 유일성 O, 최소성 X
- **[[무결성]] 제약조건:** 개체 무결성(기본키는 NULL 불가), 참조 무결성(외래키 관련).
