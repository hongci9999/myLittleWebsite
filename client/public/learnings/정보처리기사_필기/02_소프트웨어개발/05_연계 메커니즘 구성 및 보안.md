# 연계 메커니즘 구성 및 보안

## 핵심 요약
- 연계 방식은 구현 방식에 따라 **직접 연계**와 **간접 연계**로 나뉩니다.
- 연계 과정의 모든 단계(송신, 서버, 수신)에서 오류가 발생할 수 있으며, 이를 위해 체계적인 **오류 코드**와 **로그 관리**가 필요합니다.
- 보안을 위해 **채널 암호화(VPN 등)**와 **데이터 암호화(대칭/비대칭키)**를 적용하며, 암호화 시 데이터 길이 변화에 유의해야 합니다.

---

## 1. 연계 방식 비교 (직접 vs 간접)

| 구분 | 직접 연계 | 간접 연계 |
| :--- | :--- | :--- |
| **특징** | 중간 매개체 없이 직접 연결 | 연계 서버(EAI/ESB) 등을 거침 |
| **장점** | 구현 단순, 비용 저렴, 성능 우수 | 다양한 환경 통합, 보안/로직 적용 용이 |
| **단점** | 인터페이스 변경에 민감, 보안 적용 취약 | 구조 복잡, 성능 저하 가능성, 테스트 기간 소요 |

---

## 2. 간접 연계 메커니즘 상세 과정

1. **데이터 생성 및 추출:** 프로그램이나 DB 오브젝트를 이용. 오류 시 로드 테이블/파일에 상세 내역 기록.
2. **코드 매핑 및 변환:** 수신 시스템 규격에 맞게 변환. (송신 전 또는 수신 후 수행)
3. **인터페이스 테이블/파일 생성:** 송신 측은 전송 관리 항목, 수신 측은 수신 관리 항목을 포함하여 생성.
4. **연계 서버:** 전송 주기마다 송수신 처리 수행. (보통 **수신 시스템 구간**에 위치)
5. **로그 기록:** 일련의 모든 활동 결과를 로그 테이블에 기록.
6. **데이터 반영:** 수신 데이터를 운영 DB에 반영하기 위한 별도의 변환 프로그램 필요.



---

## 3. 장애 및 오류 처리 구현

### 장애 발생 구간
- **송신 시스템:** 추출, 코드 매핑, 인터페이스 테이블 등록 단계.
- **연계 서버:** 데이터 변환 및 송수신 전송 단계.
- **수신 시스템:** 데이터 로드, DB 반영 단계.

### 오류 코드 체계 정의 (예시)
오류의 체계적 관리를 위해 식별자 규칙을 정의합니다.
- **접두어:** `E` (Error)
- **발생 위치:** 연계 서버(`S`), 연계 프로그램(`A`)
- **오류 유형:** 형식(`F`), 길이(`L`), 코드(`C`), 데이터(`D`), 권한(`S`), 필수입력(`M`)
- **구성 예시:** `E-S-F-001` (연계 서버에서 발생한 형식 오류 001번)

## 4. 인터페이스 보안 및 암호화

### 채널 구간 암호화
- OSI 7계층의 네트워크 패킷 자체를 암호화합니다.
- **[[VPN]] (가상 사설망)**이나 전용선 솔루션을 적용하여 전체 데이터에 적용합니다.

### 데이터 보안 (애플리케이션 암호화)
- **절차:** 암호화 후 연계 테이블 등록 → 수신 후 복호화하여 운영 DB 반영.
- **주의사항:** 암호화된 데이터는 기존보다 **데이터 길이(Length)가 커지므로**, DB 컬럼 크기를 충분히 확장해야 합니다.

### 암호화 알고리즘 종류
1. **단방향 암호화:** [[Hash]] 기반. 복호화가 불가능함. (예: 비밀번호 저장)
2. **양방향 암호화:** 암호화와 복호화가 모두 가능함.
    - **[[대칭키]] (Symmetric Key):** 암/복호화 키가 동일 (AES, SEED). 속도 빠름.
    - **[[비대칭키]] (Asymmetric Key):** 암/복호화 키가 다름 (RSA). 키 관리 용이, 속도 느림.

## 3. 빈출 내용 정리
- **@오류코드 구성**: 오류 발생 위치와 유형(F, L, C, D 등)을 구분하는 문제.
- **대칭키 vs 비대칭키**: 두 방식의 키 동일 여부와 대표 알고리즘 매칭.
- **데이터 길이 변경**: 암호화 적용 시 DB 컬럼 길이를 늘려야 한다는 실무적 특징.
- **직접/간접 연계 장단점**: 성능이 중요한지, 유연성이 중요한지에 따른 선택 기준.
