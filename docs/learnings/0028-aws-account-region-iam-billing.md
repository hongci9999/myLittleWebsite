# AWS 계정·리전·IAM·과금 (배포 입문)

날짜: 2026-05-04
태그: [배포, aws, 인프라, 도구, 개발방법론]

## 요약

AWS는 **리전(지리적 데이터 센터 묶음)** 단위로 리소스가 존재한다. **루트 계정**은 일상 작업에 쓰지 않고, **IAM 사용자·역할**로 최소 권한을 부여한다. 실험 후 **불필요한 리소스를 삭제**하는 습관이 비용 사고를 막는다.

---

## 핵심 개념

| 개념 | 설명 |
|------|------|
| **리전** | 예: `ap-northeast-2`(서울). 리소스는 대부분 리전에 귀속 |
| **AZ(가용 영역)** | 리전 내 독립 데이터 센터. 고가용 설계 시 여러 AZ 사용 |
| **루트 사용자** | 계정 생성 시 이메일로 로그인하는 최상위 주체. MFA 권장, 일상 작업 비권장 |
| **IAM 사용자** | 사람이 콘솔·CLI로 접속할 때 쓰는 정체성 |
| **IAM 역할** | 서비스(EC2, GitHub Actions OIDC 등)에 **임시 자격**을 붙일 때 사용 |
| **정책(Policy)** | JSON으로 “무엇을 허용/거부”하는 권한 문서 |
| **최소 권한** | 필요한 API·리소스만 열어 두는 원칙 |
| **과금** | 대부분 사용량·시간 기반. 무료 한도가 있어도 **상시 EC2** 등은 초과하기 쉽다 |

## 상세 설명

### 1. 리전을 고르는 기준

- **지연**: 사용자·팀이 주로 있는 곳과 가깝게
- **서비스 가용**: 드물게 특정 서비스가 리전별로 다를 수 있음
- **일관성**: S3 버킷·Beanstalk·RDS를 **같은 리전**에 두면 트래픽·설정이 단순해지는 경우가 많다(예외: CloudFront는 글로벌 엣지)

### 2. IAM을 나누는 이유

- 키 유출 시 피해 범위를 줄인다.
- CI(GitHub Actions)는 **장기 Access Key** 대신 **OIDC로 짧은 역할**을 받는 방식이 권장되는 경우가 많다(0032).

### 3. Access Key를 쓸 때 / 안 쓸 때

- **쓰기 어려운 경우**: 로컬 개발 도구, 레거시 스크립트(가능하면 역할 전환으로 대체)
- **피해야 하는 경우**: 공개 저장소, 클라이언트 번들, 프론트엔드 코드

### 4. 비용·정리 습관

- 실습 리소스에 **이름 태그**(예: `project=mylittlewebsite`, `env=dev`)를 붙인다.
- 주기적으로 **콘솔 비용 탐색기**·예산 알람을 설정한다.
- 실험 종료 후: 인스턴스 종료, Elastic IP 해제, 로드밸런서·불필요 버킷 정리.

### 5. 절차 체크리스트 (계정 준비)

- [ ] 루트에 **MFA** 설정
- [ ] 관리용 **IAM 사용자** 생성, 콘솔 로그인 또는 SSO 정책 정리
- [ ] 사용 **리전**을 하나 정하고 콘솔 우측 상단에 고정
- [ ] 배포에 쓸 **IAM 역할·정책**을 문서화(나중에 CI에 그대로 옮김)
