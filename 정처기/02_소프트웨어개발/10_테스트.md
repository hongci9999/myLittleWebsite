# 애플리케이션 테스트 (Application Testing)

## 핵심 요약
테스트는 소프트웨어에 잠재된 **결함을 찾아내어 문제점을 해결**하고, 제품의 신뢰도를 향상시키는 과정입니다. 결함이 없음을 증명하는 것이 아니라 '결함이 있음을 증명'하는 과정이며, **화이트박스/블랙박스** 테스트 기법의 차이와 **테스트 오라클**의 종류를 구분하는 것이 시험 합격의 핵심입니다.

---

## 상세 정리

### 1. 테스트의 기본 원칙 (7대 원칙)
* **결함 집중 (Defect Clustering)**: 소프트웨어 결함의 80%는 특정 20% 모듈에 집중되어 있다는 원칙 (**파레토 법칙** 적용).
* **살충제 패러독스 (Pesticide Paradox)**: 동일한 테스트 케이스로 반복 실행하면 더 이상 새로운 결함을 발견할 수 없으므로 주기적으로 케이스를 개선해야 함.
* **오류 부재의 궤변 (Absence of Errors Fallacy)**: 요구사항을 만족하지 못한다면, 결함이 없더라도 품질이 높다고 볼 수 없음.
* **테스트는 결함이 있음을 보여주는 것**: 모든 결함을 제거하는 것은 불가능함.
* **완전한 테스트 불가능**: 무한한 경로와 입력값이 존재하므로 산출물 위주로 수행.
* **테스트의 정황(Context) 의존**: 소프트웨어 성격에 따라 테스트 방법이 달라짐.
* **초기 테스팅 (Early Testing)**: 개발 초기 단계부터 테스팅을 시작하여 비용을 절감.

### 2. 테스트 프로세스 및 산출물
1.  **계획**: 테스트 목적, 범위, 일정 수립 (**테스트 계획서**)
2.  **분석 및 디자인**: 테스트 조건 확인 및 설계
3.  **테스트 케이스/시나리오 작성**: 입력값, 실행 조건, 예상 결과 명시 (**테스트 케이스**, **테스트 시나리오**)
4.  **수행**: 실제 테스트 실행 (**테스트 결과서**)
5.  **평가**: 결과 분석 및 리포팅

### 3. 소프트웨어 테스트 유형


[Image of White-box testing vs Black-box testing]


#### A. 정적 테스트 vs 동적 테스트
* **정적 테스트**: 프로그램을 실행하지 않고 명세서나 소스 코드를 분석 (워크스루, 인스펙션 등).
* **동적 테스트**: 프로그램을 **실제로 실행**하여 결함을 발견.

#### B. 화이트박스(White-Box) 테스트
- **개념**: 소스 코드의 내부 로직(제어 구조, 루프 등)을 직접 관찰하며 테스트.
- **주요 기법**:
    * **기초 경로 검사(Basic Path Testing)**: 대표적인 화이트박스 기법으로, 논리적 복잡도를 측정.
    * **제어 구조 검사**: 조건 검사(논리식), 루프 검사(반복 구조), 데이터 흐름 검사(변수 정의 및 사용).
- **순환 복잡도(Cyclomatic Complexity) 계산**:
    * $V(G) = E - N + 2$ (E: 간선 수, N: 노드 수)

#### C. 블랙박스(Black-Box) 테스트
- **개념**: 내부 구조를 보지 않고 사용자의 요구사항(기능) 명세를 기반으로 테스트.
- **주요 기법**:
    * **동등 분할 테스트(Equivalence Partitioning)**: 입력값을 유효/무효 그룹으로 나누어 대표값 추출.
    * **경계값 분석(Boundary Value Analysis)**: 경계선 부근에서 오류 발생 확률이 높음을 이용.
    * **원인-효과 그래프**: 입력과 출력 간의 관계를 분석하여 그래프로 표현.
    * **오류 예측(Error Guessing)**: 테스터의 경험과 감각에 의존.
    * **비교 테스트**: 여러 버전의 프로그램에 동일한 데이터를 입력하여 결과 비교.

### 4. 테스트와 디버깅
* **검증 (Verification)**: **개발자 관점**. "제품을 올바르게 만들고 있는가?" (과정 확인)
* **확인 (Validation)**: **사용자 관점**. "올바른 제품을 만들었는가?" (결과 확인)
* **디버깅 (Debugging)**: 발견된 오류를 찾고 수정하는 과정 (테스트 자체는 아님).

### 5. 테스트 오라클 (Test Oracle)
테스트 결과가 참인지 거짓인지 판단하기 위해 사전에 정의된 참 값을 비교하는 기법입니다.

* **참(True) 오라클**: **모든 입력값**에 대해 기대 결과 생성. (미션 크리티컬한 업무)
* **샘플링(Sampling) 오라클**: **특정 몇 개의 입력값**에 대해서만 예상 결과 제공.
* **휴리스틱(Heuristic) 오라클**: 샘플링 오라클 + 나머지 값은 **추정(가정)**으로 처리.
* **일관성 검사(Consistent) 오라클**: 수정 전후의 결과값이 동일한지 확인.

> [!INFO] 💡 AI 보충 설명: 목적에 따른 테스트 분류
> - **회귀(Regression) 테스트**: 수정된 코드 때문에 새로운 결함이 생기지 않았는지 다시 확인하는 필수 테스트입니다.
> - **강도(Stress) 테스트**: 시스템이 감당할 수 있는 이상의 과부하를 주어 비정상적인 상황에서의 동작을 확인합니다.
> - **회복(Recovery) 테스트**: 시스템 오류 발생 후 다시 정상 상태로 복구되는지 확인합니다.

## 3. 빈출 내용 정리
- **[[테스트 원칙]]**: 결함 집중(파레토), 살충제 패러독스, 오류 부재의 궤변.
- **[[화이트박스 테스트]]**: 기초 경로 검사, 순환 복잡도 공식 ($V(G) = E - N + 2$).
- **[[블랙박스 테스트]]**: 동등 분할, 경계값 분석, 원인-효과 그래프.
- **[[테스트 오라클]]**: 참, 샘플링, 휴리스틱, 일관성 검사 오라클의 특징 구분.
- **[[검증과 확인]]**: Verification(과정) vs Validation(결과).
