# 연계 모듈 구현 및 검증

## 핵심 요약
- **[[EAI]]**와 **[[ESB]]**는 서로 다른 기종의 시스템을 연동하는 핵심 기술입니다.
- 웹 서비스 도입 시 **[[SOAP]]**, **[[RESTful]]**, **[[WSDL]]** 기술을 활용하여 데이터를 송수신합니다.
- 인터페이스의 정상 동작을 확인하기 위해 **[[xUnit]]**, **[[STAF]]**, **[[Selenium]]** 등의 검증 도구를 사용합니다.

---

## 1. EAI (Enterprise Application Integration) 구성 방식
서로 다른 기종의 시스템 간의 연동을 위한 솔루션입니다.

- **Point-to-Point:** 중간 미들웨어 없이 시스템을 1:1로 직접 연결. 변경과 재사용이 어려움.
- **Hub & Spoke:** 단일 접점인 **Hub(허브)** 시스템을 통해 데이터를 전송하는 중앙집중형 방식. 확장/유지보수가 용이하나 허브 장애 시 전체 시스템 마비(Single Point of Failure).
- **Message Bus (ESB):** 미들웨어(Bus)를 두어 확장성과 처리량을 향상시킨 방식. 어댑터가 필요 없고 보안/관리가 용이하나 표준화가 어렵고 성능이 상대적으로 낮음.



---

## 2. 연계 및 웹 서비스 구현

### 환경 구축 및 구현 절차
1. 연계를 위한 DB 설치 및 사용자 계정 생성.
2. 연계 테이블, 로그 테이블, 매핑 테이블 생성.
3. 연계 응용 프로그램 구현:
    - **송신:** 데이터 추출 및 수신 측 코드로 변환.
    - **수신:** 데이터 유효성 검증 및 운영 DB 반영.
    - **보안:** 프로그래밍 언어를 이용해 구현(EAI/ESB의 트리거는 암호화 알고리즘 호출에 제한적임).

### 웹 서비스 방식
- 연계 데이터를 **[[XML]]** 형식으로 생성하는 프로그램이 필요합니다.
- **[[SOAP]]** 또는 **[[RESTful]]** 방식을 사용하여 데이터를 송수신합니다.
- 운영 DB 반영 시 **[[WSDL]]**에 기술된 명세를 따릅니다.

## 3. 인터페이스 구현 검증 도구 (매우 중요)
인터페이스의 정상 작동 여부를 확인하기 위한 자동화 테스트 프레임워크입니다.

| 도구명 | 특징 |
| :--- | :--- |
| **[[xUnit]]** | Java(JUnit), C++(CppUnit) 등 다양한 언어를 지원하는 단위 테스트 프레임워크 |
| **[[STAF]]** | 서비스 호출, 자원 관리 등 다양한 환경을 지원하는 데몬 기반 프레임워크 |
| **[[Fitnesse]]** | 웹 기반의 테스트 케이스 설계/실행/결과 확인 도구 |
| **[[NTAF]]** | **STAF**와 **Fitnesse**를 통합한 네이버의 테스트 프레임워크 |
| **[[Selenium]]** | 다양한 브라우저 및 개발 언어를 지원하는 웹 애플리케이션 테스트 도구 |
| **[[Watir]]** | **Ruby** 기반의 웹 애플리케이션 테스트 프레임워크 |

## 4. 예외 처리 방안 및 HTTP 상태 코드

### 인터페이스 예외 처리
- **송신측:** 데이터 정제 및 정합성 체크 강화.
- **수신측:** 특수문자 치환, 입력 대기 큐(Queue)를 활용한 서버 불안정 해소.

### HTTP 상태 코드 (Status Code)
서버 통신 시 발생하는 예외 상황을 숫자로 구분합니다.
- **400 (Bad Request):** 잘못된 요청.
- **401 (Unauthorized):** 인증 실패.
- **403 (Forbidden):** 접근 거부.
- **404 (Not Found):** 페이지/리소스 없음.
- **500 (Internal Server Error):** 내부 서버 오류.
- **503 (Service Unavailable):** 서버 과부하 등으로 인한 일시적 중단.

### 5. 수신 시스템 예외 처리 및 모니터링

#### 수신측 예외 처리 상세
- **송신 데이터 변환:** 송신 데이터에 포함된 특수문자를 수신 시스템에서 처리 가능한 문자로 치환하여 오류 방지.
- **프로그램 결함:** 논리적 결함 수정 및 충분한 테스트 수행, 필요 시 송신측 프로그램 수정을 요청.
- **서버 리소스 관리:** 서버 불안정 시 **입력 대기 큐(Queue)**를 도입하여 부하를 조절하고 서비스 중단 방지.

#### 연계 시스템 모니터링 방식
- **오류 알람:** 오류 발생 시 사용자/관리자가 즉시 인지하도록 **Email, SMS** 등으로 메시지 발송.
- **관리 도구:**
    - **오류 로그:** 상세 분석이 가능하나 전문성이 필요함.
    - **오류 테이블:** 데이터베이스에 기록하여 내역 관리가 용이함.
    - **모니터링 도구:** 연계 시스템 전반을 시각적으로 관리.
- **[[APM]] (Application Performance Management):** 인터페이스의 기능 동작 상태 및 성능 현황(응답 시간, 자원 사용량 등)을 통합 관리.

## 3. 빈출 내용 정리
- **@[[EAI]] vs [[ESB]]**: 중앙 집중형(Hub)인지 버스(Bus) 방식인지 구분.
- **@인터페이스 검증 도구**: 각 도구의 핵심 키워드(xUnit-언어, STAF-데몬, Selenium-브라우저 등) 매칭.
- **@HTTP 상태 코드**: 4xx(클라이언트 오류)와 5xx(서버 오류)를 구분하는 문제.
- **@APM**: 성능 및 현황 관리를 위한 도구로 언급됨.
