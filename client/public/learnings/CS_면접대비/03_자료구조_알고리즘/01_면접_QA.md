## 핵심 요약

코딩 테스트와 **대규모 차량·로그 데이터** 처리의 기초입니다. 구조별 **시간·공간 복잡도**와 **실무 적용 상황**을 함께 말할 수 있어야 합니다.

### Q1. Array vs Linked List

**질문** — 배열과 연결 리스트의 메모리 구조·탐색/삽입/삭제 시간 복잡도를 비교해 주세요.

**답변** — **배열**은 연속 메모리, **인덱스 접근 O(1)**, 중간 삽입·삭제 **O(n)**(이동 필요). **연결 리스트**는 노드+포인터, 순차 접근 **O(n)**, 알려진 위치 삽입·삭제 **O(1)**(포인터만 변경). 캐시 지역성은 배열이 유리합니다. 차량 센서 **고정 길이 시계열**은 배열/벡터, **삽입이 잦은 이벤트 큐**는 연결 리스트(또는 deque)가 적합합니다.

| 연산 | Array | Linked List |
|------|-------|-------------|
| 접근 | O(1) | O(n) |
| 삽입/삭제(중간) | O(n) | O(1)* |
| 메모리 | 연속, 캐시 친화 | 포인터 오버헤드 |

\* 위치를 알고 있을 때

**핵심 키워드** — 연속 vs 포인터, O(1) 접근 vs O(1) 삽입

---

### Q2. Stack vs Queue

**질문** — Stack(LIFO)과 Queue(FIFO) 개념과 DFS·BFS 활용을 설명해 주세요.

**답변** — **Stack**은 나중에 넣은 것이 먼저 나옵니다(LIFO). **함수 호출 스택**, **괄호 검사**, **DFS**(재귀 또는 명시적 스택)에 씁니다. **Queue**는 먼저 넣은 것이 먼저 나옵니다(FIFO). **BFS**, **작업 스케줄링**, **메시지 버퍼**에 씁니다. 그래프 **최단 경로(가중치 없음)**는 BFS, **경로 존재·위상 정렬**은 DFS가 기본입니다.

**핵심 키워드** — LIFO=Stack/DFS, FIFO=Queue/BFS

**꼬리질문** — Deque? — 양끝 O(1) 삽입·삭제, BFS·슬라이딩 윈도우

---

### Q3. Hash Table, Chaining vs Open Addressing

**질문** — 해시 테이블 동작 원리와 충돌 해결(Chaining, Open Addressing)을 설명해 주세요.

**답변** — **해시 함수**로 키→버킷 인덱스를 계산해 **평균 O(1)** 조회·삽입을 목표로 합니다. **Chaining**은 같은 버킷에 **연결 리스트/트리**로 여러 키를 둡니다. 삭제가 단순하고 load factor가 커도 동작하지만 포인터 오버헤드가 있습니다. **Open Addressing**은 빈 슬롯을 찾아 **선형/제곱 탐사**로 넣습니다. 캐시 효율이 좋지만 **clustering**, load factor 한계(보통 0.7 전후)가 있습니다. Python dict, JS Map은 내부적으로 최적화된 해시 테이블입니다.

**핵심 키워드** — 해시 함수, 충돌, Chaining vs Probing, load factor

**꼬리질문** — worst case? → O(n), 악의적 해시 입력 시

---

### Q4. BST, 최악 경우, 균형 트리

**질문** — 이진 탐색 트리 특징, 최악 시간 복잡도, 해결용 균형 트리를 설명해 주세요.

**답변** — BST는 **왼쪽 < 루트 < 오른쪽**으로 **평균 O(log n)** 탐색·삽입이 가능합니다. **한쪽으로 치우치면** 연결 리스트가 되어 **O(n)**이 됩니다. 이를 막기 위해 **AVL**, **Red-Black Tree**, **B-Tree**(DB 인덱스) 등 **자동 재균형** 트리를 씁니다. Java `TreeMap`, C++ `map`은 보통 Red-Black입니다.

**핵심 키워드** — BST O(log n) 평균, skew O(n), RB-Tree/AVL

---

### Q5. Priority Queue와 Heap

**질문** — 우선순위 큐와 힙의 관계, 삽입/삭제 시간 복잡도는?

**답변** — **우선순위 큐**는 우선순위가 높은 원소를 먼저 꺼냅니다. **이진 힙**으로 구현하면 **삽입 O(log n)**, **삭제(최대/최소) O(log n)**, **peek O(1)**입니다. **다익스트라**, **허프만 코딩**, **작업 스케줄링**(deadline 가까운 작업)에 씁니다. `heapq`(Python), 배열 기반 완전 이진 트리로 구현합니다.

**핵심 키워드** — min/max heap, O(log n) insert/delete, array 구현

---

### Q6. Graph vs Tree

**질문** — 그래프와 트리의 차이를 정의해 주세요.

**답변** — **트리**는 **사이클 없는 연결 그래프**, **루트**에서 **부모-자식** 계층이 있고 **n개 노드에 n-1개 간선**입니다. **그래프**는 **사이클 가능**, 방향/무방향, 가중치 가능합니다. 차량 **도로망**은 그래프, **조직·파일 시스템**은 트리로 모델링합니다.

**핵심 키워드** — 트리=사이클 없음, 연결, n-1 간선

---

### Q7. DFS vs BFS

**질문** — DFS와 BFS 개념, 최단 거리 등 적합한 상황은?

**답변** — **DFS**는 한 경로를 끝까지 파고듭니다(스택/재귀). **모든 경로 탐색**, **사이클 검출**, **위상 정렬**에 적합합니다. **BFS**는 레벨 순(큐). **가중치 없는 최단 경로**, **최소 이동 횟수**에 적합합니다. 가중치 있으면 **다익스트라**·**벨만-포드**로 넘어갑니다.

**핵심 키워드** — DFS=깊이/스택, BFS=너비/큐/최단(무가중)

---

### Q8. Quick Sort vs Merge Sort

**질문** — 퀵소트와 머지소트 원리, 평균/최악 시간 복잡도를 비교해 주세요.

**답변** — **퀵소트** — pivot 기준 분할, **평균 O(n log n)**, **최악 O(n²)**(편향 pivot), **제자리(in-place)**에 가깝고 캐시 효율 좋음. **머지소트** — 분할 후 병합, **항상 O(n log n)**, **O(n) 추가 공간**, **안정 정렬**. Python `sorted()`/`list.sort()`는 **Timsort**(머지+삽입 혼합)입니다.

| | Quick | Merge |
|---|-------|-------|
| 평균 | O(n log n) | O(n log n) |
| 최악 | O(n²) | O(n log n) |
| 공간 | O(log n) | O(n) |
| 안정 | 아니오 | 예 |

**핵심 키워드** — pivot 분할, 병합, Timsort

---

### Q9. DP vs Divide and Conquer

**질문** — 동적 계획법과 분할 정복의 공통점·차이(Overlapping Subproblems)를 설명해 주세요.

**답변** — 둘 다 **문제를 작은 하위 문제로 나눕니다**. **분할 정복**은 하위 문제가 **겹치지 않아** 독립 계산 후 합칩니다(머지소트). **DP**는 **겹치는 하위 문제**를 **메모이제이션·타뷸레이션**으로 한 번만 계산합니다(피보나치, 배낭). DP는 **최적 부분 구조**도 필요합니다.

**핵심 키워드** — overlapping subproblems, memoization, optimal substructure

---

### Q10. Timsort `[경험 연결]`

**질문** — Python list·TypeScript Array 정렬에 Timsort 등을 들어본 적이 있나요?

**답변** — **Timsort**는 Python·Java 기본 정렬입니다. 이미 **정렬된 구간(run)**을 활용해 **실제 데이터에서 빠르고**, worst case **O(n log n)**, **안정 정렬**입니다. V8의 `Array.sort()`는 **Timsort 변형**을 씁니다. [경험 연결] 대부분 직접 구현하지 않지만, **거의 정렬된 로그·타임스탬프** 데이터에서는 O(n)에 가깝게 동작한다는 점을 알면 성능 예측에 도움이 됩니다.

**핵심 키워드** — run 감지, merge, stable, adaptive

### Q11. QuadTree 등 공간 인덱스

**질문** — 대규모 차량 위치 데이터를 효율 탐색하는 자료구조를 아시나요?

**답변** — **QuadTree**는 2D 공간을 **4분할**해 점·영역을 저장합니다. **범위 쿼리**(반경 내 차량)**에 O(log n) 수준**으로 줄일 수 있습니다. 유사하게 **Geohash**, **R-tree**, **H3**(Uber)가 위치 기반 서비스에 쓰입니다. 전부 스캔 O(n) 대신 **공간 인덱스**로 후보를 좁힙니다.

**핵심 키워드** — 2D 분할, range query, Geohash/H3

**꼬리질문** — DB에서는? → PostGIS, MongoDB 2dsphere 인덱스
