# 배포 전 공통 개념 (빌드·환경·정적 vs API)

날짜: 2026-05-04
태그: [배포, 개발방법론, 백엔드, 프론트, 아키텍처]

## 요약

배포는 **로컬에서 돌던 것을 “인터넷에서 접속 가능한 환경”으로 옮기고 유지**하는 일이다. 소스 그대로가 아니라 **빌드 산출물**과 **런타임 설정**이 중심이 된다.

---

## 핵심 개념

| 개념 | 설명 |
|------|------|
| **빌드** | TypeScript·React 소스를 브라우저·Node가 실행 가능한 형태(JS, HTML, CSS, 정적 자산)로 만드는 과정 |
| **산출물(artifact)** | 빌드 결과 폴더. 예: `client/dist`, `server/dist` |
| **런타임** | 산출물을 실제로 실행하는 환경(브라우저, Node 프로세스, OS) |
| **환경(env)** | 로컬 / 스테이징 / 프로덕션 등. 같은 코드라도 **접속 URL·DB·키**가 달라질 수 있음 |
| **환경 변수** | 실행 시 주입되는 설정(`PORT`, `SUPABASE_URL` 등). 코드에 하드코딩하지 않는 것이 보통 |
| **시크릿** | API 키·비밀키. 저장소에 넣지 않고 호스팅 플랫폼·CI의 비밀 저장소에 둔다 |
| **정적 호스팅** | HTML·JS·이미지처럼 **미리 만들어 둔 파일**만 서빙. 요청마다 서버에서 페이지를 조립하지 않음(SPA도 정적 파일 + 클라이언트 라우팅) |
| **API 서버** | HTTP 요청을 받아 **로직·DB 접근**을 수행하는 프로세스. 보통 **항상 켜져 있거나** 서버리스로 요청 단위 실행 |

---

## 상세 설명

### 1. 배포가 하는 일

1. **재현 가능한 빌드**: 같은 커밋에서 같은 명령으로 같은 산출물이 나오게 한다.
2. **실행 환경에 맞게 설정**: 포트, CORS 허용 출처, DB URL 등을 프로덕션 값으로 둔다.
3. **네트워크에 노출**: 도메인·HTTPS·방화벽 규칙을 맞춘다.
4. **관측·복구**: 로그, 헬스 체크, 이전 버전으로 되돌리는 방법을 갖춘다.

### 2. 이 레포 구조와의 대응

| 레이어 | 디렉터리 | 로컬에서 | 배포 후 일반적 모습 |
|--------|----------|----------|---------------------|
| 프론트 | `client/` | `vite` 개발 서버 | `npm run build` → `dist/` 를 정적 호스팅 |
| API | `server/` | `tsx watch` 또는 `node` | `tsc` 후 `node dist/index.js` 를 VM/컨테이너/PaaS에서 실행 |
| DB·인증 | Supabase | 클라우드 URL로 연결 | **동일**(앱만 AWS로 옮겨도 됨) |

클라이언트는 **브라우저에서 실행**되므로, Supabase를 클라이언트에서 직접 쓰지 않는 한(이 프로젝트 규칙상 서버만 DB) **API 서버 URL**이 브라우저 입장에서의 “백엔드 주소”가 된다.

### 3. “정적”과 “API”를 나누는 이유

- **정적**: 트래픽이 많아도 캐시·CDN으로 비용·지연을 줄이기 쉽다.
- **API**: 비즈니스 로직·비밀 키·DB 접근을 **신뢰할 수 있는 서버**에서만 하기 위함.

### 4. 절차 체크리스트 (공통)

- [ ] `README` 또는 루트 스크립트에 **프로덕션 빌드·시작 명령**이 적혀 있는지 확인한다.
- [ ] 프로덕션에 필요한 **환경 변수 목록**을 적어 둔다(이름만이라도).
- [ ] 시크릿은 **Git에 없는지** 다시 확인한다.
- [ ] API와 프론트의 **URL이 바뀌었을 때** CORS·프록시 설정을 점검한다(0031).
