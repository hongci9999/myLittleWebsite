### Q1. Python 런타임 타입 에러 방지 `[경험 연결]`

**질문** — dynamic typing인 Python에서 런타임 타입 에러를 어떻게 방지하나요?

**답변** — **Type Hinting**(함수 시그니처·dataclass), **`mypy`/`pyright` 정적 검사**, **Pydantic**으로 입출력 검증, **단위·통합 테스트**를 조합합니다. [경험 연결] API 경계·설정 파싱에 Pydantic, CI에 `mypy --strict`를 두면 배포 전에 타입 불일치를 잡을 수 있습니다. 런타임에는 여전히 동적이므로 **경계에서 검증**하는 것이 핵심입니다.

**핵심 키워드** — Type hints, mypy, Pydantic, 경계 검증

### Q2. Reference Counting과 Generational GC

**질문** — Python 메모리 관리: 레퍼런스 카운팅과 세대별 GC 원리를 설명해 주세요.

**답변** — CPython은 주로 **레퍼런스 카운팅**으로 객체 참조 수가 0이 되면 즉시 해제합니다. **순환 참조**(A↔B)는 카운트만으로 안 풀려 **Generational GC**가 주기적으로 **0·1·2세대**를 탐색해 회수합니다. young 세대는 자주, old는 드물게 스캔합니다. `__del__`·순환 참조는 예측이 어려워 **context manager·weakref**로 명시적 관리가 좋습니다.

**핵심 키워드** — refcount, 순환 참조, generational GC

---

### Q3. Generator vs Iterator

**질문** — Generator와 Iterator 차이, 대용량 데이터에서 Generator가 메모리 효율적인 이유는?

**답변** — **Iterator**는 `__iter__`/`__next__`로 순회하는 객체입니다. **Generator**는 `yield`로 **이터레이터를 만드는 함수**로, **lazy evaluation**으로 **한 번에 하나씩** 값을 생산합니다. 대용량 파일·DB 커서를 **리스트로 전부 올리지 않고** 한 줄씩 처리해 **메모리 O(1)**에 가깝게 유지합니다.

**핵심 키워드** — yield, lazy, 스트리밍 처리

**꼬리질문** — `yield from`? — 하위 제너레이터 위임

### Q4. TypeScript 컴파일과 런타임 타입

**질문** — TypeScript 컴파일(트랜스파일) 과정과 런타임에 타입이 남는지 설명해 주세요.

**답변** — `tsc` 또는 esbuild/swc가 **타입 검사(선택)** 후 **타입 정보를 제거**하고 ES JavaScript로 변환합니다. **런타임에는 타입이 없습니다**. `interface`는 완전히 사라지고, `enum`·`class` 일부는 JS 코드로 남습니다. 런타임 검증이 필요하면 **Zod**, **io-ts** 등을 씁니다. Express API에서는 **요청 바디를 Zod로 parse**하는 패턴이 흔합니다.

**핵심 키워드** — erase types, tsc→JS, Zod 런타임 검증

---

### Q5. Interface vs Type Alias

**질문** — Interface와 Type Alias 차이는?

**답변** — 둘 다 객체 형태를 표현할 수 있지만, **interface**는 **선언 병합(declaration merging)**이 가능하고 `extends`로 확장하기 쉽습니다. **type**은 **유니온·교집합·tuple·mapped type** 등 **표현력**이 넓습니다. 공개 API·라이브러리 확장은 interface, **복잡한 타입 연산**은 type alias가 많습니다. 성능 차이는 없고 **컴파일 타임**에만 존재합니다.

**핵심 키워드** — interface merge, type=union/intersection

---

### Q6. any vs unknown

**질문** — any와 unknown 차이, any를 지양하는 이유는?

**답변** — **any**는 타입 검사를 **끕니다**. 어떤 연산도 허용해 **버그를 컴파일이 못 잡습니다**. **unknown**은 **안전한 top type**으로, 사용 전에 **narrowing**(typeof, in, type guard)이 필요합니다. 외부 JSON·레거시 연동은 `unknown` + Zod가 권장됩니다.

**핵심 키워드** — any=검사 off, unknown=narrowing 필수

---

### Q7. async/await: Python vs JS/TS

**질문** — Python async/await와 JS/TS Promise 기반 비동기의 공통점·차이는?

**답변** — 공통점은 **비동기 코드를 동기처럼** 쓰고, **await**로 결과를 기다린다는 점입니다. **JS/TS**는 **Promise**가 기본 단위이고, **이벤트 루프**가 microtask를 처리합니다. **Python**은 **asyncio**의 **coroutine(Task)**이며 **단일 스레드 이벤트 루프**입니다. JS는 브라우저·Node 공통, Python은 `asyncio` + `await`able I/O 라이브러리(aiohttp, asyncpg) 필요. 둘 다 **CPU 바운드**는 스레드/프로세스 풀로 보완합니다.

**핵심 키워드** — coroutine, event loop, Promise vs Task

### Q8. Structural Typing / Duck Typing `[경험 연결]`

**질문** — 구조적 타이핑·덕 타이핑을 두 언어와 연결해 설명해 주세요.

**답변** — **덕 타이핑**(Python) — "오리처럼 걷고 울면 오리"로 **실제 메서드 존재**가 중요, 명목적 타입 이름은 런타임에 무관합니다. **구조적 타이핑**(TypeScript) — **형태(shape)**가 같으면 호환, `class` 이름이 달라도 `{ name: string }`이면 통과합니다. [경험 연결] TS API 응답 타입을 interface로 두면 **필드만 맞으면** 다른 소스 데이터도 수용하기 쉽습니다.

**핵심 키워드** — nominal vs structural, shape compatibility

---

### Q9. Structural Subtyping

**질문** — TypeScript 구조적 서브타이핑이 무엇인가요?

**답변** — **서브타입** 관계가 **이름·상속**이 아니라 **필드·메서드 구조**로 결정됩니다. `B`가 `A`의 모든 필드를 가지면 `B`는 `A`에 할당 가능(**excess property checking**은 리터럴에만 엄격). Java는 명목적, TS는 구조적이라 **어댑터 타입** 작성이 유연합니다.

**핵심 키워드** — 할당 가능성=구조 포함 관계

### Q10. 성능 병목 모니터링·해결 `[경험 연결]`

**질문** — Python/TS 백엔드·파이프라인에서 성능 병목을 어떻게 모니터링·해결하겠습니까?

**답변** — ① **측정** — APM(p50/p99), **OpenTelemetry** 트레이스, 로그 상관 ID. ② **프로파일** — Node **clinic/0x**, Python **cProfile/py-spy**. ③ **병목 유형별** — DB면 **쿼리·인덱스·N+1**, CPU면 **알고리즘·캐시**, I/O면 **비동기·배치·큐**. ④ **아키텍처** — Redis 캐시, 읽기 레플리카, 워커 분리, CDN. [경험 연결] Express API에서 Supabase slow query 로그 + `EXPLAIN ANALYZE`로 인덱스를 추가한 경험을 말할 수 있습니다.

**핵심 키워드** — measure first, p99, EXPLAIN, cache·queue

**꼬리질문** — p99가 중요한 이유? — 평균만 보면 꼬리 지연(차량 API SLA)을 놓침
