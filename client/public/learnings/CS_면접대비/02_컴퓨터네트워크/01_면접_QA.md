## 핵심 요약

차량-클라우드 연동과 웹 백엔드의 기반입니다. **계층별 역할**, **TCP/UDP 선택**, **HTTP 보안**, **실시간 통신**을 연결해서 설명할 수 있어야 합니다.

### Q1. OSI 7계층 vs TCP/IP 4계층

**질문** — OSI 7계층과 TCP/IP 4계층의 차이를 설명하고, 각 계층의 대표 프로토콜을 말해 주세요.

**답변** — OSI는 **이론적 7계층** 모델이고, TCP/IP는 **실제 인터넷 구현**에 가까운 4계층입니다. TCP/IP는 응용·표현·세션을 **Application**으로 묶고, 데이터링크·물리를 **Network Access**로 묶습니다.

| TCP/IP | OSI 대응 | 대표 프로토콜 |
|--------|----------|---------------|
| Application | 5~7 | HTTP, DNS, WebSocket |
| Transport | 4 | TCP, UDP |
| Internet | 3 | IP, ICMP |
| Network Access | 1~2 | Ethernet, Wi-Fi, CAN(물리·데이터링크) |

**핵심 키워드** — OSI=참조 모델, TCP/IP=실무 스택

**꼬리질문** — HTTPS는 어느 계층? → 응용 계층(HTTP) + TLS가 표현 계층 역할

---

### Q2. TCP vs UDP

**질문** — TCP와 UDP의 핵심 차이를 신뢰성과 속도 관점에서 비교하고, 모빌리티 서비스 예시를 들어 주세요.

**답변** — **TCP**는 연결 지향·순서 보장·재전송·흐름/혼잡 제어로 **신뢰성**이 높지만 **오버헤드**가 큽니다. **UDP**는 비연결·비보장·낮은 지연으로 **속도**에 유리합니다. 모빌리티 예: **차량 원격 제어·OTA 펌웨어·결제 API**는 TCP/HTTPS, **실시간 위치 스트림·차량 내 센서 브로드캐스트·V2X 일부**는 UDP(손실 허용·저지연)를 씁니다.

**핵심 키워드** — TCP=3-way handshake·재전송, UDP=fire-and-forget

**꼬리질문** — TCP가 느린 이유? → handshake, ACK, 버퍼링, 혼잡 제어

---

### Q3. TCP 3-Way / 4-Way Handshake

**질문** — TCP 3-Way Handshake와 4-Way Handshake의 목적과 각 단계를 설명해 주세요.

**답변** — **3-Way Handshake**는 연결 **설정**입니다. ① Client `SYN`, ② Server `SYN+ACK`, ③ Client `ACK`. 양쪽이 **초기 시퀀스 번호(ISN)**를 교환해 통신 준비를 확인합니다. **4-Way Handshake**는 연결 **종료**입니다. ① FIN, ② ACK, ③ FIN, ④ ACK. 양방향이 독립적으로 닫히므로 4단계가 필요하고, **TIME_WAIT** 상태로 지연 패킷 처리합니다.

**핵심 키워드** — SYN, ACK, FIN, ISN, TIME_WAIT

**꼬리질문** — SYN flood? → 반쪽 연결만 쌓는 DoS, SYN cookie로 완화

---

### Q4. HTTP vs HTTPS

**질문** — HTTP와 HTTPS의 차이를 SSL/TLS·대칭키·공개키 관점에서 설명해 주세요.

**답변** — HTTP는 **평문** 전송이라 도청·변조에 취약합니다. HTTPS는 HTTP 위에 **TLS**를 올려 **기밀성·무결성·인증**을 제공합니다. 핸드셰이크에서 **공개키(인증서)**로 대칭 세션 키를 안전하게 교환하고, 이후 데이터는 **대칭키(AES)**로 암호화합니다. 공개키는 느려서 **키 교환에만** 쓰고, 본문은 대칭키로 처리합니다. 차량-클라우드 API·사용자 인증은 HTTPS가 필수입니다.

**핵심 키워드** — TLS handshake, 인증서(CA), 대칭키 데이터 암호화

**꼬리질문** — TLS 1.3 개선? → handshake 1-RTT, 구 cipher 제거

### Q5. GET vs POST, Idempotent

**질문** — GET과 POST의 차이와 Idempotent(멱등성) 개념을 설명해 주세요.

**답변** — **GET**은 리소스 **조회**, 본문 없이 URL·쿼리로 전달, **캐시·북마크** 가능합니다. **POST**는 리소스 **생성·처리**, 본문에 데이터를 실어 보냅니다. **멱등성**은 같은 요청을 여러 번 해도 **서버 상태가 한 번만 바뀌는** 성질입니다. GET·PUT·DELETE는 멱등, POST는 일반적으로 비멱등입니다. REST 설계 시 결제·OTA 트리거 API는 **Idempotency-Key**로 중복 요청을 방지합니다.

**핵심 키워드** — GET=조회·멱등, POST=생성·비멱등, PUT=멱등 갱신

**꼬리질문** — GET에 body? → 스펙상 가능하나 프록시·캐시 호환 때문에 비권장

---

### Q6. WebSocket vs REST

**질문** — WebSocket과 REST API의 차이, 실시간 데이터에 WebSocket이 유리한 이유는?

**답변** — REST는 **요청-응답** 기반이라 클라이언트가 폴링하면 **오버헤드·지연**이 큽니다. **WebSocket**은 HTTP로 핸드셰이크 후 **양방향 풀듀플 연결**을 유지해 서버가 **push**할 수 있습니다. 실시간 **차량 위치·대시보드·채팅·원격 진단 스트림**에 적합합니다. 단방향 알림만 필요하면 **SSE**도 대안입니다.

**핵심 키워드** — 지속 연결, 서버 push, 폴링 vs push

**꼬리질문** — WebSocket 보안? → wss://, origin 검사, 인증 토큰

### Q7. Node.js 이벤트 루프와 비동기 I/O

**질문** — Node.js(TypeScript)에서 비동기 Non-blocking I/O가 처리되는 원리를 설명해 주세요.

**답변** — Node.js는 **싱글 스레드 이벤트 루프**가 메인입니다. I/O 요청은 **libuv**가 OS 비동기 API(epoll, IOCP)로 넘기고, 완료 시 **콜백을 이벤트 큐**에 넣습니다. 루프가 **timers → pending → poll → check → close** 순으로 큐를 처리합니다. CPU 작업은 루프를 막으므로 **Worker Threads**로 분리합니다. Express API에서 DB·파일 I/O가 동시에 와도 **블로킹 없이** 다른 요청을 처리할 수 있는 이유입니다.

**핵심 키워드** — libuv, 이벤트 큐, poll 단계, 싱글 스레드

**꼬리질문** — microtask vs macrotask? → Promise.then은 microtask, setTimeout은 macrotask

---

### Q8. DNS 동작 원리

**질문** — 브라우저에 URL을 입력했을 때 DNS가 어떻게 동작하나요?

**답변** — ① 브라우저 **DNS 캐시** 확인 → ② OS 캐시 → ③ **Recursive Resolver**(ISP/8.8.8.8)에 질의 → ④ Root → **TLD**(.com) → **Authoritative** 네임서버 순으로 위임 → ⑤ IP 반환 → ⑥ TCP 연결·TLS·HTTP 요청. 캐시는 **TTL**로 갱신 주기가 정해집니다. CDN·로드밸런서는 DNS로 **가까운 엣지 IP**를 돌려줍니다.

**핵심 키워드** — 재귀 질의, Root/TLD/Authoritative, TTL, 캐시 계층

**꼬리질문** — DNS over HTTPS? → DNS 쿼리도 암호화해 스누핑 방지

### Q9. CORS

**질문** — CORS 에러 원인과 해결 방법을 설명해 주세요.

**답변** — 브라우저 **동일 출처 정책** 때문에 `https://api.example.com`과 `https://app.example.com`이 다르면 기본적으로 XHR/fetch가 차단됩니다. **CORS**는 서버가 `Access-Control-Allow-Origin` 등 헤더로 허용 출처를 명시하면 브라우저가 통과시킵니다. **Preflight(OPTIONS)**는 DELETE·커스텀 헤더 등에 먼저 허가를 받습니다. 해결: ① API 서버 CORS 설정, ② **리버스 프록시**로 같은 출처, ③ 서버-서버 통신(브라우저 아님)은 CORS 무관.

**핵심 키워드** — Same-Origin, ACAO 헤더, Preflight, 프록시

**꼬리질문** — credentials? → `Access-Control-Allow-Credentials: true` + 와일드카드 Origin 불가

### Q10. Load Balancer

**질문** — 로드 밸런서의 역할과 대표 알고리즘을 설명해 주세요.

**답변** — 트래픽을 **여러 서버에 분산**해 가용성·처리량을 높입니다. L4(TCP/IP)는 연결 단위, L7(HTTP)은 URL·헤더 기반 라우팅이 가능합니다. 알고리즘: **Round Robin** — 순환 분배, **Least Connections** — 연결 수 적은 서버, **Weighted** — 성능 비례, **IP Hash** — 세션 고정. 헬스 체크로 **죽은 노드 제외**합니다. 차량 텔레메트리 API 스케일아웃 시 ALB/NLB 뒤에 EC2·컨테이너를 둡니다.

**핵심 키워드** — L4/L7, Round Robin, Least Conn, Health Check

**꼬리질문** — Sticky session? → 쿠키/IP로 같은 서버 고정, 상태 저장 시 필요

---

### Q11. CAN 통신

**질문** — 차량 통신에서 CAN의 특징을 설명해 주세요.

**답변** — **CAN(Controller Area Network)**은 차량 ECU 간 **멀티마스터 버스** 통신입니다. **우선순위 기반 ID**로 충돌 시 낮은 ID(높은 우선순위)가 먼저 전송됩니다. **짧은 프레임·낮은 비트레이트**(클래식 CAN ~1Mbps)지만 **실시간·견고성**에 최적화되어 있습니다. 브레이크·엔진 등 **안전 관련 메시지**에 쓰이며, 대역폭이 부족해 **CAN FD**, 고속에는 **Ethernet(Automotive)**으로 확장됩니다. 클라우드 API(TCP/IP)와 달리 **차량 내부 필드버스** 계층입니다.

**핵심 키워드** — 멀티마스터, ID 우선순위, ECU, CAN FD

**꼬리질문** — CAN vs LIN? → LIN은 저속·저비용 서브 네트워크(도어·시트 등)
