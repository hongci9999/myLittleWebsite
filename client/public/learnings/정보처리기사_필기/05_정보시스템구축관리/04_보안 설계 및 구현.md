# 소프트웨어 개발 보안 설계 및 구현

## 📌 핵심 요약
소프트웨어 개발 전 과정에서 보안을 고려하여 안전한 시스템을 구축하는 기법을 다룹니다. 보안의 3요소인 **[[CIA]]**를 기반으로 자산에 대한 위험을 관리하며, 시큐어 코딩을 통해 **보안 약점**을 사전에 제거하고 **취약점 점검**을 통해 발생 가능한 공격을 차단하는 것이 핵심입니다.

---

## 📝 상세 정리

## 1. 소프트웨어 개발 보안의 개요

### 1.1 정보보호 및 관리 체계
- **정보보호**: 정보의 훼손, 유출을 방지하여 안전성을 확보하는 모든 행위.
- **[[ISMS]] (정보보호 관리체계)**: 기업 내 주요 정보자산을 보호하기 위한 종합적인 관리 체계.
- **[[PIMS]] (개인정보보호 관리체계)**: 개인정보 보호를 위한 관리적, 기술적 보호 조치 체계.

### 1.2 보안의 3요소 ([[CIA]])
1. **기밀성 ([[Confidentiality]])**: 인가된 사용자만 정보에 접근할 수 있도록 보장하는 속성.
2. **무결성 ([[Integrity]])**: 인가되지 않은 사용자에 의해 정보가 변경, 생성, 삭제되지 않도록 보호하는 속성.
3. **가용성 ([[Availability]])**: 인가된 사용자가 필요할 때 지체 없이 정보를 사용할 수 있는 속성.



### 1.3 기타 보안 요소
- **부인 방지 ([[Non-Repudiation]])**: 송수신자가 데이터 전송 사실을 나중에 부인하지 못하도록 증거를 제공함.
- **인증 ([[Authentication]])**: 사용자나 장치의 신원(유효성)을 확인하는 과정.
- **권한 부여 ([[Authorization]])**: 인증된 사용자에게 특정 자원에 대한 접근 권한을 부여함.

### 1.4 위험 관리 용어
- **자산 (Asset)**: 보호해야 할 유무형의 대상.
- **위협원 (Threat Agent)**: 위협을 발생시키는 주체.
- **위협 (Threat)**: 자산에 손실을 줄 수 있는 잠재적 사건.
- **취약점 ([[Vulnerability]])**: 자산에 내재된 약점(공격의 통로).
- **위험 (Risk)**: 취약점을 이용하여 위협이 발생했을 때 나타나는 결과와 가능성.

## 2. 보안 약점 및 시큐어 코딩

### 2.1 7대 보안 약점 유형
소프트웨어 개발 보안 가이드(행정안전부)에서 정의한 유형입니다.
1. **입력 데이터 검증 및 표현**: 사용자 입력값에 대한 부적절한 검증. (예: SQL Injection, XSS)
2. **보안 기능**: 인증, 암호화 등 보안 기능을 부적절하게 구현.
3. **시간 및 상태**: 병렬 처리 시 자원 공유 문제나 레이스 컨디션 발생.
4. **에러 처리**: 에러 메시지에 시스템 내부 정보(경로, 변수명 등)가 노출됨.
5. **코드 오류**: 타입 변환 오류, 자원 해제 미비 등.
6. **캡슐화**: 불충분한 정보 은닉으로 데이터 노출.
7. **API 오용**: 보안에 취약한 API를 사용하거나 부적절하게 호출.

### 2.2 보안 계획 수립
- **[[시큐어 코딩(Secure Coding)]]**: 개발 단계에서 보안 약점을 사전에 제거하여 안전한 소프트웨어를 만드는 기법.
- **하드 코드 (Hard Code)**: 비밀번호나 경로와 같은 중요한 정보를 코드 내에 상수 형태로 직접 작성하는 행위(지양해야 함).

---

## 3. 소프트웨어 취약점 및 보안 테스트

### 3.1 주요 취약점
- **[[XSS]] (Cross-Site Scripting)**: 웹 페이지에 악의적인 스크립트를 삽입하여 사용자 브라우저에서 실행되게 함.
- **[[Injection]] (주입)**: 데이터 입력 창에 SQL이나 시스템 명령어를 주입하여 비정상적인 동작을 유도함.
- **불안전한 직접 객체 참조**: 인가되지 않은 파일이나 DB 기록에 접근하는 행위.
- **검증되지 않은 리다이렉트 및 포워드**: 피싱 사이트로의 이동을 유도함.

### 3.2 보안 테스트 및 분석 기법
- **테스트 등급**: $영향도 \times 긴급도$로 산출하여 우선순위 결정.
- **분석 기법**:
    - **정적 분석 (Static Analysis)**: 소스 코드를 실행하지 않고 잠재적 약점을 분석.
    - **동적 분석 (Dynamic Analysis)**: 실제 실행 환경에서 취약점을 확인.
- **원칙**: 개발자와 운영자(테스트 수행자)는 역할을 분리하여 독립적으로 테스트를 수행해야 함.

### 3.3 소프트웨어 보안 결함 종류
- **발견된 보안 결함**: 제품이 실제 운영 환경으로 넘어가기 전 테스트 과정에서 식별된 결함.
- **잠재적 보안 결함**: 운영 환경으로 전달되었으나 아직 표면화되지 않은 결함.
- **소프트웨어 특이 고장**: 잠재적 결함들이 운영 중에 복합적으로 작용하여 발생하는 이상 징후들의 집합.

---

> [!INFO] 💡 AI 보충 설명: 정적 분석 vs 동적 분석
> **정적 분석**은 개발 초기 단계에서 도구를 활용해 문법과 규칙을 체크하므로 비용이 적게 들지만, 실행 시 발생하는 논리적 오류를 찾기 어렵습니다. 반면 **동적 분석**은 메모리 누수나 런타임 오류를 잡는 데 탁월하지만 환경 구축에 비용이 많이 듭니다. 보통 두 방식을 병행하는 것이 정석입니다.
