# 타로 위젯 해석 스키마·제공자 전략
날짜: 2026-04-08
상태: 채택

## 배경 (왜 이 결정이 필요한가)
메이저 아르카나 3장 위젯을 추가하면서, 해석 생성 경로를 로컬 규칙 기반과 API LLM 두 가지로 운영해야 했다. 이때 경로별 응답 형식이 달라지면 클라이언트 렌더링과 오류 처리가 복잡해진다.

## 결정 (무엇으로 했는지)
1. 타로 카드 자산은 `client/src/widgets/TarotDailyWidget/assets/cards/`에 캡슐화한다.
2. 해석 요청은 서버 단일 엔드포인트 `POST /api/tarot/reading`으로 통합한다.
3. `aiProvider`가 `local`이면 규칙 기반 해석기, `api`면 `AiTextProvider`를 사용한다.
4. 서버는 두 경로 모두 동일한 `TarotReadingResponse` 스키마로 응답한다.
5. API LLM 결과가 비정형이면 서버 정규화 계층에서 보정해 공통 포맷을 강제한다.

## 이유 (다른 선택지를 배제한 이유)
- 클라이언트에서 로컬/API를 직접 분기하면 UI가 제공자 구현 세부사항에 결합된다.
- 위젯 외부 공용 자산으로 먼저 분리하면 현재 범위에서는 과설계 가능성이 크다.
- API 결과 원문을 그대로 전달하면 모델 출력 편차로 파싱 실패가 잦아질 수 있다.

## 결과/참고
- 클라이언트는 제공자 종류와 무관하게 동일 필드만 렌더링하면 된다.
- 해석 품질 개선은 서버 내부 프롬프트/규칙 업데이트로 독립적으로 진행할 수 있다.
