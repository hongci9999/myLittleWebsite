# 프로세스 관리 및 상태 전이 심화

## 📌 핵심 요약
실행 중인 프로그램인 **[[프로세스]]**의 정의와 정보를 기록하는 **[[PCB]]**를 이해하고, CPU가 여러 작업을 전환하며 수행하는 **[[문맥 교환]]**의 원리를 학습합니다. 또한 프로세스의 5가지 상태 변화와 **[[인터럽트]]** 처리 과정, 프로세스 간 통신(**[[IPC]]**) 및 **[[스레드]]**의 계층 구조를 상세히 분석합니다.

---

## 📝 상세 정리

### 1. 프로세스(Process)와 PCB
- **개념**: 하드디스크에 있던 프로그램이 메모리에 적재되어 실행되고 있는 상태입니다.
- **[[PCB]] (Process Control Block)**:
    - 운영체제가 프로세스를 관리하기 위해 필요한 정보를 기록한 고유 테이블입니다.
    - 프로세스가 생성될 때마다 생성되며, 종료 시 삭제됩니다.
    - **// 기록 정보 예시**: PID(프로세스 식별 번호), 상태(Ready, Running 등), PC(Program Counter), 레지스터 저장값, 우선순위 등.
- **[[문맥 교환]] (Context Switching)**:
    - CPU가 한 프로세스에서 다른 프로세스로 전환될 때, 이전 프로세스의 상태를 PCB에 저장하고 새 프로세스의 정보를 적재하는 과정입니다.
    - **// 타임 슬라이스(Time Slice) 영향**:
        - **너무 작을 때**: 잦은 전환으로 인해 **오버헤드**와 인터럽트 횟수가 증가하여 효율이 떨어집니다.
        - **너무 클 때**: 문맥 교환은 줄어드나 한 작업이 독점하게 되어 **시분할 시스템**의 응답 속도가 느려집니다.

### 2. 프로세스 상태 전이
프로세스는 실행되는 동안 다음의 5가지 상태를 거칩니다.
1. **생성(Create)**: 이제 막 생성되어 PCB가 할당된 상태.
2. **준비(Ready)**: CPU를 할당받기 위해 줄을 서서 기다리는 상태.
3. **실행(Running)**: CPU를 할당받아 실제로 명령어를 수행 중인 상태.
4. **대기(Wait/Block)**: 입출력(I/O) 등의 이벤트 완료를 기다리며 멈춘 상태.
5. **종료(Terminated)**: 자원을 모두 반납하고 마친 상태.



- **상태 변화 명령어 (Transition)**:
    - **Dispatch**: 준비 → 실행 (CPU 점유)
    - **Block**: 실행 → 대기 (입출력 요청 시 실행을 멈춤)
    - **Wake up**: 대기 → 준비 (입출력 완료 시 다시 줄을 섬)
    - **Timer Runout**: 실행 → 준비 (주어진 시간을 다 써서 강제로 내려옴)

### 3. 인터럽트(Interrupt) 처리
- **개념**: 프로그램 실행 중 예기치 않은 상황이 발생하여 현재 작업을 일시 중단하고 해당 사건을 처리하는 메커니즘입니다.
- **종류**:
    - **외부**: 입출력 장치 오류, 타이머 종료, 전원 이상 등.
    - **내부 (Trap)**: 잘못된 명령(0으로 나누기 등)이나 잘못된 데이터 참조 시 발생.
    - **소프트웨어**: 사용자의 시스템 호출(System Call) 요청 등에 의해 발생.
- **// 처리 흐름 상세 설명**:
    1. 현재 명령어 수행을 마무리하고 일시 중단합니다.
    2. 현재 상태(PCB, PC 등)를 안전한 곳에 저장합니다.
    3. 인터럽트 처리 루틴(ISR)으로 문맥 교환을 수행하여 해당 사건을 해결합니다.
    4. 처리가 끝나면 저장했던 정보를 복구하여 중단 지점부터 다시 수행합니다.

### 4. 프로세스 통신(IPC) 및 스레드(Thread)
- **[[IPC]] (Inter-Process Communication)**: 독립된 프로세스들이 데이터를 주고받는 기법입니다.
    - **공유 메모리**: 프로세스들이 공통의 메모리 영역을 공유. 속도가 매우 빠르지만 데이터 일관성 문제가 발생할 수 있어 불안정할 수 있습니다.
    - **메시지 전달**: OS 커널이 메시지 큐를 통해 데이터를 전달. 안전하지만 속도가 상대적으로 느립니다.
- **[[스레드]] (Thread)**: '경량 프로세스'라 불리며, 프로세스 내에서 실행되는 흐름의 단위입니다.
    - **장점**: 병행성 증진, 처리율 향상, 메모리 자원 공유로 낭비 감소.
    - **계층 구조와 특징**:
        - **사용자 수준**: 라이브러리가 관리. 커널 개입이 없어 전환 속도가 빠르지만 구현이 복잡하고, 한 스레드가 대기(Block) 상태가 되면 프로세스 전체가 멈추는 단점이 있습니다.
        - **커널 수준**: OS가 직접 관리. 독립적으로 동작하여 안정적이지만 전환 시 커널 모드로 바뀌어야 하므로 속도가 상대적으로 느립니다.
        - **멀티 레벨**: 위 두 방식의 장점을 혼합한 구조입니다.

---

> [!INFO] 💡 AI 보충 설명: 멀티태스킹 vs 멀티스레딩
> 멀티태스킹은 여러 개의 '프로세스'를 빠르게 번갈아 실행하는 것이고, 멀티스레딩은 하나의 프로세스 내부에서 여러 '줄기(스레드)'를 동시에 실행하는 것입니다. 스레드는 부모 프로세스의 자원을 공유하기 때문에 생성과 전환 비용이 훨씬 저렴합니다.

## 🎯 빈출 키워드
- **[[PCB]]**: 프로세스의 상태 정보가 담긴 고유 제어 테이블.
- **[[문맥 교환]]**: CPU 할당 대상이 바뀔 때 정보를 저장하고 적재하는 행위.
- **[[Dispatch]]**: 준비 상태의 프로세스가 CPU를 점유하는 과정.
- **[[IPC]]**: 독립된 프로세스 간의 데이터 교환 기법(공유 메모리 vs 메시지 전달).
- **[[사용자 수준 스레드]]**: 커널의 지원 없이 라이브러리에 의해 제어되는 스레드.

# 프로세스 스케줄링 및 교착상태 상세 정리

## 📌 핵심 요약
프로세스에 자원을 효율적으로 배분하는 **[[스케줄링]]** 알고리즘의 종류와 계산법을 익히고, 임계 구역의 동기화 문제 해결을 위한 **[[세마포어]]**를 학습합니다. 마지막으로 시스템을 멈추게 하는 **[[교착상태]]**의 발생 조건과 이를 해결하기 위한 **[[은행원 알고리즘]]** 및 회복 기법을 마스터합니다.

---

## 📝 상세 정리

### 1. 프로세스 스케줄링 (Scheduling)
CPU를 언제, 어느 프로세스에게 할당할지 결정하는 전략입니다.

#### (1) 비선점형 (Non-Preemptive): 강제 중단 불가능
- **[[FIFO]] (First-In-First-Out)**: 도착한 순서대로 CPU 할당.
    - **// 문제 예시**: 
        - P1(실행시간 20), P2(실행시간 4)가 순차 도착 시 (도착시간 0 가정)
        - P1 대기시간: 0, 종료시간: 20
        - P2 대기시간: 20, 종료시간: 24
        - **평균 반환 시간**: $(20 + 24) / 2 = 22$
        - **평균 대기 시간**: $(0 + 20) / 2 = 10$
- **[[SJF]] (Shortest Job First)**: 실행 시간이 가장 짧은 것부터 처리. 
    - **// 특징 설명**: 평균 대기 시간이 가장 짧지만, 실행 시간이 긴 프로세스는 영구히 뒤로 밀리는 **기아 현상(Starvation)**이 발생할 수 있습니다.
- **[[HRN]] (Highest Response-ratio Next)**: SJF의 기아 현상을 보완하기 위해 대기 시간을 고려.
    - $우선순위 = \frac{대기시간 + 서비스(실행)시간}{서비스 시간}$
    - **// 예시**: 
        - A: 대기 10, 서비스 5 → 우선순위 $(10+5)/5 = 3$
        - B: 대기 20, 서비스 20 → 우선순위 $(20+20)/20 = 2$
        - 결과: 수치가 높은 **A가 우선순위**를 가짐.

#### (2) 선점형 (Preemptive): 강제 중단 가능, 실시간 시스템에 유리
- **[[RR]] (Round Robin)**: 동일한 **타임 슬라이스**를 순환하며 할당. 시분할 시스템에 최적.
    - **// 예시**: 타임 슬라이스가 4ms일 때, 10ms 작업은 4ms 수행 후 다시 큐의 끝으로 이동하여 다음 차례를 기다림.
- **[[SRT]] (Shortest Remaining Time)**: 남은 실행 시간이 가장 짧은 프로세스를 우선 실행.
- **[[MFQ]] (Multilevel Feedback Queue)**: 여러 단계의 큐를 두고, 입출력 위주 프로세스에 높은 우선순위 부여.



### 2. 상호배제와 세마포어
- **[[임계 구역]] (Critical Section)**: 공유 자원에 접근하는 프로그램 코드 영역. 한 번에 한 프로세스만 진입해야 함.
- **상호배제 (Mutual Exclusion)**: 한 프로세스가 임계 구역에 있으면 다른 프로세스의 접근을 차단함.
    - **// 특징**: 임계 구역 진입은 무한정 연기되지 않아야 하며, 프로세스 개수나 속도에 영향을 받지 않아야 함.
- **[[세마포어]] (Semaphore)**: **바쁜 대기(Busy Wait)** 문제를 해결하기 위한 동기화 도구.
    - **// 작동 예시**:
        - `P(S)` (Wait/Sleep): 자원 수($S$)를 확인. $S > 0$이면 $S$를 1 감소시키고 진입. $S \le 0$이면 대기 큐로 이동.
        - `V(S)` (Signal/Wakeup): 작업 완료 후 자원 반납. $S$를 1 증가시키고 대기 중인 프로세스를 깨움.

### 3. 교착상태 (Deadlock)
두 개 이상의 프로세스가 서로가 가진 자원을 기다리며 무한 대기에 빠지는 상태입니다.

- **발생 4가지 필수 조건 (전부 충족 시 발생)**:
    1. **상호배제**: 자원은 한 번에 한 놈만 씀.
    2. **점유와 대기**: 자원을 하나 가진 채로 다른 걸 기다림.
    3. **비선점**: 남이 가진 자원을 강제로 뺏을 수 없음.
    4. **환형 대기**: 프로세스들이 원형으로 물려 있음.

- **해결 기법**:
    - **예방 (Prevention)**: 4가지 조건 중 하나를 제거 (자원 낭비 심함).
    - **회회 (Avoidance)**: **[[은행원 알고리즘]]**. 자원 할당 전 시스템이 안전 상태인지 미리 계산함.
    - **발견 (Detection)**: 시스템 감시를 통해 교착 여부 파악.
    - **회복 (Recovery)**: 교착 프로세스 중 일부를 강제 종료하거나 자원을 뺏음.
        - **// 희생양 선정 기준**: 우선순위가 낮거나, 진행률이 적거나, 자원을 적게 소모 중인 프로세스를 먼저 희생시킴.



---

> [!INFO] 💡 AI 보충 설명: 은행원 알고리즘 (Banker's Algorithm)
> 다익스트라가 제안한 기법으로, 은행이 모든 고객의 대출 요구를 수용할 수 있을 만큼의 현금을 보유해야 파산을 피할 수 있다는 원리에서 유래했습니다. 프로세스가 자원을 요청할 때, 이를 할당한 후에도 '안전 순서(Safe Sequence)'가 존재하는지 확인하여 할당 여부를 결정합니다.

- **[[세마포어]]**: P와 V 연산을 통해 바쁜 대기를 해결하는 동기화 기법.
