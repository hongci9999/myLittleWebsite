# 데이터 수집 기술 및 저장 시스템의 이해

## 📌 핵심 요약
본 강의는 데이터의 형태(정형, 반정형, 비정형)에 따른 다양한 수집 기술과 빅데이터 저장 시스템의 핵심인 [[Hadoop|HDFS]] 및 [[NoSQL]]의 특성을 다룹니다. 특히 기존 [[RDBMS]]와 [[NoSQL]]의 차이점 및 데이터 모델에 따른 [[NoSQL]]의 분류 체계를 상세히 학습합니다.

---

## 📝 상세 정리

### 1. 데이터 수집 방법 (Data Collection Methods)
- **설문조사 (Survey):** 응답자에게 질문지를 통해 정형화된 데이터를 수집하는 정량적 방법입니다.
- **관찰 (Observation):** 현장에서 대상의 행동이나 현상을 직접 보고 기록하여 데이터를 수집합니다.
- **실험 (Experiment):** 변수를 통제한 환경에서 원인과 결과의 인과관계를 파악하기 위해 데이터를 수집합니다.
- **웹 크롤링 (Web Crawling):** 자동화된 봇을 이용하여 웹 페이지의 데이터를 추출하고 수집합니다.
- **소셜미디어 분석 (Social Media Analysis):** SNS상의 텍스트, 관계망 등을 분석하여 사용자 트렌드 및 감성을 파악합니다.
- **데이터베이스 및 기존 데이터 활용:** 이미 구축된 내부 시스템이나 공공 데이터의 수집 기법을 활용합니다.
- **집단면접 (FGI, Focus Group Interview):** 소수의 응답자 그룹과 중재자의 대화를 통해 심층적인 질적 데이터를 수집합니다.
- **스크래퍼 (Scraper):** 웹사이트에서 필요한 특정 정보만을 골라 추출하는 소프트웨어 도구입니다.

### 2. 정형 데이터 수집 기술 (Structured Data Collection)
- **ETL (Extraction, Transformation, Loading):** 소스 시스템에서 데이터를 추출, 변환하여 대상 데이터베이스나 데이터 웨어하우스에 적재하는 기술입니다.
- **FTP (File Transfer Protocol):** 장치 간 파일을 전송하기 위한 표준 네트워크 프로토콜입니다.
- **API (Application Programming Interface):** 응답과 요청을 통해 프로그램 간 데이터를 주고받는 인터페이스입니다.
- **DBtoDB:** 서로 다른 데이터베이스 간에 데이터를 직접 전송하고 동기화하는 방식입니다.
- **Sqoop (SQL-to-Hadoop):** [[RDBMS]]와 [[Hadoop]] 간에 대용량 데이터를 효율적으로 전송하기 위한 도구입니다.

### 3. 비정형 데이터 수집 기술 (Unstructured Data Collection)
- **크롤링 (Crawling):** 분산된 웹 페이지를 순회하며 텍스트, 이미지 등 비구조화된 데이터를 수집합니다.
- **RSS (Rich Site Summary):** 웹사이트의 콘텐츠 업데이트 정보를 사용자에게 자동으로 전달하는 신디케이션 방식입니다.
- **Open API:** 누구나 사용할 수 있도록 공개된 API를 통해 실시간 데이터를 수집합니다.
- **척와 (Chukwa):** 대규모 분산 시스템의 로그를 수집, 분석하고 [[Hadoop]]에 저장하는 프레임워크입니다.
- **카프카 (Kafka):** 고성능 분산 메시징 시스템으로, 실시간 데이터 스트림을 발행/구독 방식으로 처리합니다.

### 4. 반정형 데이터 수집 기술 (Semi-structured Data Collection)
- **플럼 (Flume):** 대량의 로그 데이터를 효율적으로 수집, 집계 및 이동시키기 위한 신뢰성 있는 분산 서비스입니다.
- **스크라이브 (Scribe):** 페이스북에서 개발한 실시간 스트리밍 로그 수집 서버입니다.
- **센싱 (Sensing):** 센서로부터 생성되는 물리적 환경 데이터를 수집하는 기술입니다.
- **스트리밍 (Streaming):** 데이터가 생성되는 즉시 실시간으로 전송받아 처리하는 기법입니다.

### 5. 빅데이터 저장 시스템 (Big Data Storage Systems)
신뢰성(Reliability)과 가용성(Availability)을 보장하는 시스템으로 구성됩니다.

#### 5.0 저장소(스토리지) 유형

| 유형 | 정의 |
|------|------|
| **DAS (Direct Attached Storage)** | 서버와 저장소가 **물리적으로 직접 연결**된 방식 |
| **NAS (Network Attached Storage)** | **LAN 네트워크를 통해** 여러 컴퓨터가 **동일 저장소를 공유**하는 방식 |
| **SAN (Storage Area Network)** | **특수 목적용 고속 네트워크**와 스위치를 통해 연결된 방식 |

#### 5.1 분산 파일 시스템 (Distributed File System)

확장 가능한 분산 환경에서 데이터를 저장하며, 저사양 서버를 활용한 대용량·분산·데이터 집중형 어플리케이션을 지원합니다. **결함 허용(Fault Tolerance)**으로 일부 장애 시에도 정상 운영이 가능합니다.

**(1) GFS (Google File System)**

구글의 대용량 데이터 처리를 위해 설계된 분산 파일 시스템입니다.

| 구성 요소 | 역할 |
|----------|------|
| **마스터 (Master)** | 관리·통제 |
| **청크 서버 (Chunk Server)** | 물리적 저장소 처리 |
| **클라이언트 (Client)** | 어플리케이션 |

**(2) HDFS (Hadoop Distributed File System)**

GFS를 기반으로 개발된 **오픈 소스** 분산 파일 시스템입니다.

| 구성 요소 | 역할 |
|----------|------|
| **마스터노드 (= NameNode)** | 슬레이브 노드 관리 |
| **슬레이브노드 (= DataNode)** | 데이터 처리 및 저장 |

* **블록 크기:** 데이터는 기본 **64MB**의 여러 블록으로 고정 저장되며, 사용자가 변경 가능합니다.
* **효과:** 분산 처리로 **시스템의 과부하 및 병목 현상 해소**에 기여합니다.

**(3) 그 외 파일 시스템**

* **Ceph:** 오픈소스 분산 스토리지 플랫폼
* **아마존 S3 (Amazon S3):** 객체 스토리지 서비스

> [!NOTE] 개념 구분
> **분산 파일 시스템**과 **분산 데이터베이스**는 구분되는 개념입니다. 파일 시스템은 파일/블록 단위 저장에, 데이터베이스는 구조화된 데이터의 CRUD·질의에 각각 특화되어 있습니다.

#### 5.2 데이터베이스 저장 방식
- **구분:** 관계형 데이터베이스 시스템([[RDBMS]])과 [[NoSQL]]로 구분됩니다.
- **특징:** 수평적 확장성, 데이터 복제, 일관성 보장을 목표로 합니다.

#### 5.3 데이터웨어하우스·데이터마트·데이터레이크

**(1) 데이터 웨어하우스 (DW, Data Warehouse)**

| 특징 | 설명 |
|------|------|
| **주제지향성** | 분석 목적 설정이 중요. 특정 주제(예: 고객, 매출) 중심으로 데이터 구성 |
| **데이터 통합** | 다양한 소스(DB, 외부)의 데이터를 **표준화된 형식**으로 저장 |
| **시계열성** | 과거 **히스토리**를 포함한 시계열 데이터 보유 |
| **비휘발성** | 읽기 전용(Read-only), 자주 변경되지 않음 |

* **구성 요소**
  * **ETL (Extraction, Transform, Load):** 다양한 소스 데이터를 추출·정제·적재
  * **ODS (Operational Data Store):** 여러 DBMS에서 추출한 데이터를 **일시 저장**하는 공간

**(2) 데이터 마트 (DM, Data Mart)**

* DW의 **부분집합**으로, 특정 목적(부서·업무)에 맞는 소규모 데이터 웨어하우스
* 특정 용도에 최적화된 소규모 DW

**(3) 데이터 레이크 (Data Lake)**

* **비정형 데이터**를 원본 형태로 저장
* [[Hadoop]]과 연동하여 처리·분석 수행

#### 5.4 하둡 분산 파일 시스템 (HDFS)
- **분산 처리:** 대용량 파일을 클러스터 내 여러 노드에 블록 단위로 분산 저장합니다.
- **블록 구조:** 마지막 블록을 제외하고 모든 블록은 동일한 크기를 가집니다.
- **구조:** 하나의 마스터([[NameNode]])와 여러 개의 슬레이브([[DataNode]])로 구성됩니다.
- **복제 기법:** 데이터 복제를 통해 가용성을 높이며, 비정형 데이터 처리에 효율적입니다.
- **확장성:** 장비(노드)가 증가할수록 처리 성능이 향상됩니다.

### 6. NoSQL (Not Only SQL)
유연한 데이터 저장 및 검색 메커니즘을 제공하며, 높은 확장성과 가용성을 특징으로 합니다.

#### 6.1 RDBMS vs NoSQL 비교
| 구분 | RDBMS | NoSQL |
| :--- | :--- | :--- |
| **특징** | 데이터 무결성 및 정확성 보장 | 무결성/정확성 비보장 (유연성 중점) |
| **구조** | 정규화된 테이블, 고정 스키마 | 스키마 없음(Schema-less), 다양한 형태 |
| **트랜잭션** | [[ACID]] 트랜잭션 지원 | [[BASE]] (가용성 및 성능 중점) |
| **확장성** | 수직적 확장(Scale-up) 위주, 분산 부적합 | 수평적 확장(Scale-out) 용이 |
| **변경 방식** | 수정/삭제 작업 수행 | 수정/삭제 지양, 신규 입력으로 대체 |
| **일관성** | 강한 일관성(Strong Consistency) | 최종 일관성(Eventual Consistency) |

#### 6.2 기술적 특성 및 분류
- **탄력성 (Elasticity):** 장애 발생 시에도 클라이언트가 시스템에 접근 가능하며, 다운타임 없이 대용량 데이터를 처리합니다.
- **캐싱 (Caching):** 성능 향상을 위해 메모리 기반 캐싱 기술을 적용합니다.

#### 6.3 데이터 모델에 따른 분류
1. **키-값 저장 구조 (Key-Value Store):**
   - 가장 단순한 구조로, 범위 질의(Range Query)가 어려우나 모델링이 간단합니다.
   - 예: [[DynamoDB]], [[Redis]]
2. **열 기반 저장 구조 (Column-Family Store):**
   - 연관된 데이터 위주로 읽기에 유리하며 압축 효율이 높고 범위 질의에 유리합니다. 단, 레코드 수정 시 비용이 발생합니다.
   - 예: [[BigTable]], [[Cassandra]], [[HBase]], [[Hypertable]]
3. **문서 기반 저장 구조 (Document Store):**
   - 문서마다 다른 스키마를 가질 수 있으며 레코드 간 관계 설명이 가능합니다. [[RDBMS]]와 유사한 질의가 가능합니다.
   - 예: [[SimpleDB]], [[CouchDB]], [[MongoDB]]
4. **그래프 기반 저장 구조 (Graph Store):**
   - 노드와 에지로 데이터 간의 관계를 표현합니다.

### 7. 하둡 (Hadoop)

여러 컴퓨터를 연결하여 대용량 데이터를 처리하는 **오픈소스 빅데이터 솔루션**입니다.

#### 7.1 하둡 코어 프로젝트

| 구성 요소 | 설명 |
|----------|------|
| **HDFS** | 분산 파일 시스템 (Distributed File System). 대용량 데이터를 블록 단위로 분산 저장 |
| **MapReduce** | 분산 데이터를 **병렬 처리**하는 프로그래밍 모델 |

**MapReduce 처리 단계**

* **Map 단계:** 데이터를 (Key, Value) 쌍으로 변환
* **Reduce 단계:** Map 단계 결과를 **집계(Aggregation)**

**MapReduce 패턴 종류:** Join 패턴, Grouping 패턴, 단어 카운트(Word Count) 패턴, 통계 패턴, 필터링 패턴

#### 7.2 하둡 에코시스템 (Hadoop Ecosystem)

기능별로 다양한 부가 프로젝트가 결합된 집합체입니다.

| 기능 | 예시 |
|------|------|
| **분산 코디네이터** | Zookeeper |
| **분산 리소스 관리** | YARN, Mesos |
| **데이터 저장** | HBase, HDFS, Kudu |
| **데이터 수집** | Chukwa, Flume, Scribe, Kafka |
| **데이터 처리** | Pig, Mahout, Spark, Impala, **Hive**, **MapReduce** |

> [!INFO] 💡 AI 보충 설명
> 
> **1. ACID vs BASE**
> - **ACID (RDBMS):** Atomicity(원자성), Consistency(일관성), Isolation(고립성), Durability(지속성)를 강조하여 데이터의 정확성을 엄격히 관리합니다.
> - **BASE (NoSQL):** Basically Available(기본적 가용성), Soft-state(유연한 상태), Eventual Consistency(최종 일관성)를 지향하여 가용성과 확장성을 우선시합니다.
>
> **2. Sqoop과 Flume의 차이**
> - **Sqoop:** 관계형 DB와 하둡 간의 데이터 이동(정형 데이터)에 최적화되어 있습니다.
> - **Flume:** 로그 파일이나 이벤트 데이터(반정형/비정형)의 실시간 수집에 최적화되어 있습니다.
>
> **3. 가용성(Availability)과 신뢰성(Reliability)**
> - **가용성:** 시스템이 장애 없이 서비스를 계속 제공할 수 있는 능력입니다.
> - **신뢰성:** 시스템이 의도된 기능을 오류 없이 수행할 수 있는 능력입니다. HDFS는 데이터 복제(Replication)를 통해 이 두 가지를 동시에 확보합니다.
