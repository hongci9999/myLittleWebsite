# 시스템 연동 설계 및 소프트웨어 아키텍처

## 핵심 요약
데이터의 효율적 처리를 위한 **[[코드]]** 설계부터, 시스템 전체 구조를 정의하는 **[[소프트웨어 아키텍처]]** 패턴(MVC, 파이프-필터 등)을 학습함. 또한 시스템 간 연결을 위한 **[[미들웨어]]**와 장애 발생 시 연속성을 보장하기 위한 **[[재해 복구 시스템(DRS)]]** 및 지표(RTO, RPO)를 정리함.

---

## 1. 코드 (Code) 설계

### 1) 코드의 3대 기능
1.  **식별 (Identification)**: 데이터 간의 성격에 따라 구분 (유일성).
2.  **분류 (Classification)**: 특정 기준이나 동일한 유형에 대해 그룹화.
3.  **배열 (Arrangement)**: 의미를 부여하여 순서대로 나열.

### 2) 코드의 종류 `중요`

| 종류 | 설명 | 예시 |
| :--- | :--- | :--- |
| **[[순차 코드]]**<br>(Sequence) | 자료 발생 순서대로 일련번호를 부여. | 1, 2, 3, ... |
| **[[블록 코드]]**<br>(Block) | 공통성이 있는 것끼리 블록으로 구분하고, 그 안에서 순서대로 번호 부여.<br>(장점: 분류/집계 용이, 예비 코드 확보 가능) | 01-01, 01-02 (총무부-01) |
| **[[10진 코드]]**<br>(Decimal) | 0~9까지 10진수 분할 원칙을 따름. 우측으로 갈수록 세분화.<br>(도서 분류 등에 사용) | 100(과학) -> 110(물리) |
| **[[그룹 분류 코드]]**<br>(Group Classification) | 대분류, 중분류, 소분류 등으로 구분.<br>각 분류 개수에 제한을 두지 않음. | 1-01-001 (본사-총무-인사) |
| **[[연상 코드]]**<br>(Mnemonic) | 코드만 보고 대상을 연상할 수 있도록 문자/숫자 조합. (표의성이 뛰어남) | TV-40 (40인치 TV) |
| **[[표의 숫자 코드]]**<br>(Significant Digit) | 대상 자료의 물리적 수치(길이, 무게, 용량 등)를 그대로 코드에 적용.<br>(단점: 코드가 길어질 수 있음) | 120-500-100 (두께-너비-길이) |
| **합성 코드**<br>(Combined) | 둘 이상의 코드 체계를 조합하여 사용. | |

## 2. 시스템 및 소프트웨어 아키텍처

### 1) 개념
* **시스템 아키텍처**: 시스템 전체에 대한 논리적 기능 체계와 동작 원리.
* **[[소프트웨어 아키텍처]]**: 소프트웨어의 골격(틀). 모듈, 컴포넌트 간의 상호 관계를 정의하여 의사소통 및 표준화를 돕는다.

### 2) 주요 아키텍처 패턴 `시험 빈출`



#### ① 계층화 패턴 (Layered Pattern)
시스템을 논리적인 계층(Layer)으로 분리.
* **N-Tier**: 클라이언트, 프레젠테이션, 비즈니스, 데이터 계층 등으로 분리.
* **[[SOA]] (Service Oriented Architecture)**: 서비스 지향 아키텍처. (Presentation, Process, Service, Business, Persistency 계층 등)

#### ② 클라이언트-서버 패턴 (Client-Server)
* **서버**: 서비스를 제공하는 하나의 컴포넌트.
* **클라이언트**: 서비스를 요청하는 다수의 컴포넌트.

#### ③ 마스터-슬레이브 패턴 (Master-Slave)
* **마스터**: 작업을 분리하여 슬레이브에 배포하고, 결과를 취합하여 최종 결과 반환.
* **슬레이브**: 마스터의 요청을 처리. (데이터 동기화 활용)

#### ④ [[파이프-필터 패턴]] (Pipe-Filter)
* 데이터 스트림을 생성하고 처리하는 구조.
* **필터**: 데이터를 처리하고 캡슐화.
* **파이프**: 처리된 데이터를 다음 서브 시스템으로 전달 (버퍼링, 동기화).
* *예시*: UNIX의 Shell Script.

#### ⑤ [[브로커 패턴]] (Broker)
* 분리된 컴포넌트(서버-클라이언트) 간의 연결을 조정하는 역할.
* **원격 서비스 실행**을 통해 상호작용. (예: CORBA)

#### ⑥ 피어 투 피어 패턴 (P2P)
* 각 컴포넌트(Peer)가 클라이언트와 서버의 역할을 **유동적으로** 바꾸며 서비스. (예: 파일 공유, 블록체인)

#### ⑦ 이벤트 버스 패턴 (Event Bus)
* 소스가 특정 채널에 메시지(이벤트)를 발행하면, 해당 채널을 구독(Subscribe)하는 리스너가 메시지를 수신.

#### ⑧ [[MVC 패턴]] (Model-View-Controller) `매우 중요`
사용자 인터페이스(UI)와 비즈니스 로직을 분리하여 **상호 영향(의존성)을 최소화**함.
* **Model**: 데이터 및 비즈니스 로직 처리.
* **View**: 결과의 시각화 담당 (UI).
* **Controller**: 사용자 입력 처리 및 Model-View 상호작용 조정.

#### ⑨ 블랙보드 패턴 (Blackboard)
* **비결정성(Non-deterministic) 알고리즘** 해결에 유용.
* 지식 소스가 블랙보드(공유 데이터)에 데이터를 기록하고 패턴 매칭을 통해 검색. (예: 음성 인식, 차량 인식)

#### ⑩ 인터프리터 패턴 (Interpreter)
* 특정 언어로 작성된 표현식을 수행하는 방법을 정의. (예: SQL 구문 분석, 통신 프로토콜)

## 3. 시스템 연동 및 미들웨어

### 1) 웹 서비스 연동 기술
* **[[SOAP]] (Simple Object Access Protocol)**: HTTP/HTTPS 등을 통해 XML 기반의 메시지를 교환하는 프로토콜.
* **[[UDDI]] (Universal Description, Discovery and Integration)**: 비즈니스 업체 목록(서비스)을 등록하고 검색하는 저장소.
* **[[WSDL]] (Web Services Description Language)**: 웹 서비스명, 제공 위치, 프로토콜 등 서비스 내용을 기술한 XML 언어.

### 2) [[미들웨어]] (Middleware) 솔루션
운영체제와 애플리케이션 사이에서 원만한 통신을 돕는 소프트웨어.

| 종류 | 특징 |
| :--- | :--- |
| **[[DBMS]]** | 데이터베이스와 애플리케이션을 연결. |
| **[[RPC]]**<br>(Remote Procedure Call) | 원격에 있는 프로시저를 마치 로컬 프로시저처럼 호출. |
| **[[MOM]]**<br>(Message Oriented Middleware) | 메시지 기반의 **비동기형** 메시지 전달.<br>이기종 분산 데이터 시스템의 데이터 동기화에 사용. |
| **[[TP-Monitor]]** | **온라인 트랜잭션** 업무를 처리, 감시, 제어.<br>빠른 응답 속도 유지 (예: 항공기 예매, 금융). |
| **[[ORB]]**<br>(Object Request Broker) | 객체지향 미들웨어. 객체 간의 통신 보장 (CORBA). |
| **[[WAS]]**<br>(Web Application Server) | 동적인 콘텐츠(HTTP 세션 등) 처리를 위한 웹 환경 구현. |

> [!INFO] 💡 AI 보충 설명: Web Server vs WAS
> * **웹 서버 (Web Server)**: HTML, 이미지 등 **정적인(Static)** 콘텐츠를 제공. (예: Apache, Nginx)
> * **WAS**: DB 조회나 로직 처리가 필요한 **동적인(Dynamic)** 콘텐츠를 제공. (예: Tomcat, Jeus, WebLogic)

## 4. 연동 오류 및 재해 복구

### 1) 입력 데이터 검증 오류 유형
* **생략 오류 (Omission)**: 입력된 코드 중 한 자리가 누락됨. (1234 -> 123)
* **필사 오류 (Transcription)**: 특정 자리를 잘못 입력함. (1234 -> 1235)
* **전위 오류 (Transposition)**: 좌우 자리가 바뀜. (1234 -> 1243)
* **이중 전위 오류 (Double Transposition)**: 전위 오류가 두 군데 이상 발생.
* **추가 오류 (Addition)**: 자릿수가 추가됨.
* **임의 오류 (Random)**: 위의 오류가 복합적으로 발생.

### 2) 오류 대응 및 [[이중화]] (Redundancy)
가용성을 극대화하기 위해 시스템을 복수로 구성.
* **Active-Active**: 두 시스템이 동시에 서비스를 제공 (부하 분산).
* **Active-Standby**: 주 시스템(Active) 장애 시 대기 시스템(Standby)이 서비스 제공.

### 3) [[DRS]] (Disaster Recovery System) 유형 `중요`



| 유형 | 특징 | 복구 시간 (RTO) | 비용 |
| :--- | :--- | :--- | :--- |
| **Mirror Site** | 주 센터와 **동일한 수준**의 자원을 실시간으로 동기화 (즉시 복구). | 즉시 (0에 수렴) | 가장 비쌈 |
| **Hot Site** | 대기 상태 유지. 데이터는 동기화되어 있으나 서비스 전환에 약간의 시간 소요. | 수 시간 이내 | 비쌈 |
| **Warm Site** | 중요 데이터만 백업해두고, 장비는 부분적으로 보유. | 수 일 ~ 수 주 | 중간 |
| **Cold Site** | 데이터만 보관하거나 최소한의 공간만 확보. | 수 주 ~ 수 달 | 가장 저렴 |

### 4) 핵심 지표
* **[[RTO]] (Recovery Time Objective)**: 목표 복구 시간. (언제까지 복구할 것인가? = 최대 허용 중단 시간)
* **[[RPO]] (Recovery Point Objective)**: 목표 복구 시점. (어느 시점의 데이터까지 복구할 것인가? = 허용 가능한 데이터 손실량)
* **[[BIA]] (Business Impact Analysis)**: 업무 영향 분석. 장애 발생 시 손실 비용과 영향을 분석하는 활동.

## 5. 핵심 단어 정리
- **코드**: 블록(구분/여유), 연상(의미), 표의숫자(수치).
- **아키텍처**: MVC(UI분리), 파이프-필터(스트림), 브로커(중개).
- **미들웨어**: RPC(원격), MOM(비동기/메시지), TP-Monitor(트랜잭션).
- **DRS**: Mirror(즉시) > Hot > Warm > Cold.
