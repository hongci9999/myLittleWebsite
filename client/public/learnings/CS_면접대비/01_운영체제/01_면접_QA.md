## 핵심 요약

### Q1. 프로세스와 스레드의 차이, 메모리 구조 비교

**질문** — 프로세스와 스레드의 차이를 설명하고, 각각의 메모리 구조(Code, Data, Heap, Stack)를 비교해 주세요.

**답변** — 프로세스는 OS가 자원을 할당하는 **독립 실행 단위**이고, 스레드는 프로세스 안에서 CPU를 사용하는 **실행 흐름 단위**입니다. 프로세스마다 **독립된 주소 공간**(Code, Data, Heap, Stack)을 가지지만, 한 프로세스의 스레드들은 **Code·Data·Heap을 공유**하고 **Stack만 스레드별로 분리**합니다. 그래서 스레드 간 통신은 빠르지만, Heap 공유 시 동기화가 필요합니다.

#### 프로세스(Process)란?

- **실행 중인 프로그램**입니다. 디스크에 있는 실행 파일(코드)이 메모리에 올라가 OS에 의해 관리되기 시작하면 프로세스가 됩니다.
- OS는 프로세스마다 **PID**, **메모리 주소 공간**, **파일 디스크립터**, **CPU 레지스터 상태** 등을 **PCB(Process Control Block)**에 기록해 관리합니다.
- 프로세스끼리는 **주소 공간이 완전히 분리**되어 있어, 한 프로세스가 다른 프로세스 메모리를 직접 읽거나 쓸 수 없습니다. 통신하려면 **IPC(파이프, 소켓, 공유 메모리 등)**가 필요합니다.
- 예: 차량 IVI에서 **내비 앱**, **미디어 앱**, **텔레메틱스 에이전트**가 각각 별도 프로세스로 떠 있으면, 하나가 죽어도 나머지에 영향이 적습니다.

#### 스레드(Thread)란?

- 프로세스 **내부**에서 실제로 CPU 명령을 실행하는 **작업 단위**입니다. 한 프로세스가 여러 일을 동시에 하려면 스레드를 여러 개 둡니다.
- 스레드는 **자신만의 PC(Program Counter)·레지스터·스택**을 갖지만, **같은 프로세스의 Code·Data·Heap**을 공유합니다.
- 그래서 스레드 생성·전환은 프로세스보다 **가볍고 빠릅니다**(주소 공간 전체를 새로 만들지 않음).
- 예: 같은 내비 프로세스 안에서 **지도 렌더링 스레드**, **GPS 수신 스레드**, **음성 안내 스레드**가 동시에 돌아갈 수 있습니다.

#### 메모리 구조 (Code, Data, Heap, Stack)

프로세스 주소 공간은 보통 아래 순서로 구성됩니다(구현·OS마다 세부 배치는 다름).

```
높은 주소
┌─────────────┐
│    Stack    │  ← 함수 호출, 지역 변수 (스레드마다 별도)
│      ↓      │
│   (빈 공간)  │
│      ↑      │
│    Heap     │  ← malloc/new, 동적 할당 (스레드 공유)
├─────────────┤
│    BSS      │  ← 초기화되지 않은 전역·정적 변수
│    Data     │  ← 초기화된 전역·정적 변수
├─────────────┤
│    Code     │  ← 실행 코드(Text), 읽기 전용
└─────────────┘
낮은 주소
```

| 영역 | 역할 | 프로세스 | 스레드 |
|------|------|----------|--------|
| **Code** | 기계어 명령(프로그램 본문) | 프로세스별 독립 | 스레드 간 **공유** |
| **Data** | 전역·정적 변수 | 프로세스별 독립 | **공유** |
| **Heap** | 런타임 동적 할당 | 프로세스별 독립 | **공유** (동기화 필요) |
| **Stack** | 함수 호출 프레임, 지역 변수, 반환 주소 | 프로세스당 1개(단일 스레드 시) | **스레드마다 별도** |

- **Stack이 스레드별로 갈라지는 이유** — `main()` → `foo()` → `bar()`처럼 함수가 호출될 때마다 지역 변수·복귀 주소가 쌓이는데, 스레드마다 **호출 경로가 다르기 때문**입니다. 스택이 공유되면 한 스레드가 다른 스레드의 지역 변수를 덮어씁니다.
- **Heap을 공유할 때 주의** — 여러 스레드가 같은 객체·버퍼에 동시에 접근하면 **Race Condition**이 납니다. `mutex`, `lock` 등으로 보호해야 합니다.

#### 한 줄 비교

| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 정의 | 자원 할당·격리 단위 | CPU 실행 흐름 단위 |
| 주소 공간 | 독립 | 프로세스 내 공유 |
| 생성 비용 | 큼 | 상대적으로 작음 |
| 통신 | IPC 필요 | 공유 메모리로 즉시 가능 |
| 장애 격리 | 강함 | 한 스레드 버그가 프로세스 전체에 영향 |

**핵심 키워드** — 독립 주소 공간 vs 공유 주소 공간, PCB, Stack 분리, IPC vs 공유 Heap

**꼬리질문** — 스레드가 가벼운 이유? → 생성·전환 시 주소 공간 복사 없음, 컨텍스트(레지스터·PC·스택 포인터)만 저장

### Q2. 멀티 프로세스 vs 멀티 스레드

**질문** — 멀티 프로세스와 멀티 스레드의 장단점은 무엇이며, 어떤 상황에서 각각을 선택해야 하나요?

**답변** — **멀티 프로세스**는 격리가 강해 한 프로세스 장애가 다른 프로세스에 전파되지 않지만, IPC 오버헤드와 메모리 사용량이 큽니다. **멀티 스레드**는 공유 메모리로 통신이 빠르고 자원 효율이 좋지만, 동기화 버그·데드락 위험이 있습니다. **안정성·보안이 중요한 서비스**(브라우저 탭, 마이크로서비스)는 멀티 프로세스, **I/O 바운드·실시간 데이터 처리**(웹 서버 워커, 센서 데이터 수집)는 멀티 스레드를 선택하는 경우가 많습니다. 차량 ECU처럼 **안전 등급이 높은 모듈**은 프로세스 격리, **같은 앱 내 UI·통신**은 스레드가 흔합니다.

**핵심 키워드** — 격리 vs 효율, IPC 오버헤드, 동기화 비용

**꼬리질문** — Node.js는? → 싱글 스레드 이벤트 루프 + 워커 스레드/자식 프로세스로 확장

### Q3. Context Switching

**질문** — Context Switching(문맥 교환)의 정의와 이 과정에서 발생하는 오버헤드의 원인을 설명해 주세요.

**답변** — 문맥 교환은 CPU가 한 프로세스/스레드에서 다른 프로세스/스레드로 **실행 권한을 넘길 때**, 현재 실행 상태(레지스터, PC, 스택 포인터 등)를 저장하고 다음 대상의 상태를 복원하는 과정입니다. 오버헤드 원인은 ① **레지스터·PC 등 CPU 상태 저장/복원**, ② **캐시·TLB 무효화**로 인한 캐시 미스, ③ **스케줄러 결정**, ④ 프로세스 전환 시 **페이지 테이블 교체**입니다. RTOS에서는 문맥 교환 횟수를 줄이는 것이 실시간성 확보의 핵심입니다.

**핵심 키워드** — 레지스터 저장/복원, 캐시·TLB 미스, 스케줄러

**꼬리질문** — 문맥 교환과 시스템 콜의 차이? → 시스템 콜은 커널 모드 진입, 문맥 교환은 실행 주체 교체

### Q4. 데드락

**질문** — 데드락의 발생 조건 4가지와 해결 방법(예방, 회피, 탐지, 회복)을 설명해 주세요.

**답변** — 데드락은 네 조건이 **동시에** 만족될 때 발생합니다: ① **상호 배제**, ② **점유와 대기**, ③ **비선점**, ④ **순환 대기**. 해결은 네 가지입니다. **예방** — 조건 중 하나를 원천 차단(예: 자원 순서 강제). **회피** — 은행원 알고리즘처럼 할당 전 안전 상태 검사. **탐지** — Wait-for 그래프로 순환 탐지 후 **회복**(프로세스 종료 또는 자원 선점). 실무에서는 **락 순서 통일**, **타임아웃**, **데드락 없는 자료구조**(ConcurrentHashMap 등)를 많이 씁니다.

**핵심 키워드** — 상호배제·점유대기·비선점·순환대기, 예방·회피·탐지·회복

**꼬리질문** — 데드락과 기아(starvation) 차이? → 데드락은 영원히 진행 불가, 기아는 우선순위 낮아 오래 대기

### Q5. Mutex vs Semaphore, Race Condition

**질문** — 뮤텍스와 세마포어의 차이를 설명하고, Race Condition에 대해 말해 주세요.

**답변** — **뮤텍스**는 **상호 배제(Mutual Exclusion)**용으로, 한 번에 하나의 스레드만 임계 구역에 진입합니다. 소유권 개념이 있어 **같은 스레드가 unlock**해야 합니다. **세마포어**는 **카운터**로 N개의 자원 접근을 허용하며, 소유권이 없고 signal/wait로 동작합니다. **Race Condition**은 여러 스레드가 **동기화 없이 공유 자원**에 동시 접근해 결과가 실행 순서에 따라 달라지는 버그입니다. 해결은 뮤텍스·세마포어·원자적 연산으로 **임계 구역 보호**입니다.

**핵심 키워드** — Mutex=1, Semaphore=N, 임계 구역, 원자성

**꼬리질문** — 바이너리 세마포어와 뮤텍스? → 둘 다 1이지만 뮤텍스는 소유권·재진입 정책이 다름

### Q6. Python GIL

**질문** — 파이썬의 GIL이 무엇이며, 멀티 스레딩 환경에서 어떤 제약이 생기나요?

**답변** — GIL(Global Interpreter Lock)은 CPython에서 **한 번에 하나의 스레드만 Python 바이트코드를 실행**하게 하는 락입니다. 레퍼런스 카운팅 GC와 C 확장 모듈 안전을 위한 설계입니다. 제약은 **CPU 바운드 작업**에서 멀티 스레드가 **진정한 병렬 실행**을 하지 못하고, 스레드가 GIL을 두고 경쟁한다는 점입니다. **I/O 바운드**는 I/O 대기 중 GIL을 놓아 효과가 있습니다. CPU 집약 작업은 **`multiprocessing`**, **C 확장**, **다른 런타임(PyPy 등)**을 고려합니다.

**핵심 키워드** — CPython, 바이트코드 단일 실행, CPU-bound vs I/O-bound

**꼬리질문** — GIL이 없는 Python? → Jython, IronPython; CPython 3.13+ nogil 실험

### Q7. RTOS vs 일반 OS 스케줄링

**질문** — RTOS와 일반 OS 스케줄링의 가장 큰 차이는 무엇인가요?

**답변** — **RTOS**는 **기한(deadline) 준수**가 최우선입니다. 고정 우선순위·선점형 스케줄링(Rate Monotonic, EDF 등)으로 **최악 응답 시간(WCET)**을 보장하려 합니다. **일반 OS**(Linux, Windows)는 **처리량·공정성·사용자 체감**을 우선하며, RR·CFS 등으로 **평균 성능**을 높입니다. 차량의 브레이크·센서 퓨전은 RTOS, 인포테인먼트·클라우드 API는 GPOS가 담당하는 식으로 역할이 나뉩니다.

**핵심 키워드** — deadline 보장, 선점·고정 우선순위, 처리량 vs 실시간성

**꼬리질문** — Hard vs Soft real-time? → Hard는 기한 miss=시스템 실패, Soft는 품질 저하

### Q8. 가상 메모리, 페이징 vs 세그멘테이션

**질문** — 가상 메모리의 개념과 페이징·세그멘테이션 체제의 차이를 설명해 주세요.

**답변** — **가상 메모리**는 프로세스에게 **연속된 큰 주소 공간**을 제공하고, 실제로는 RAM 일부와 디스크(스왑)를 **Demand Paging**으로 매핑하는 기법입니다. **페이징**은 주소 공간을 **고정 크기 페이지**로 나누고 물리 프레임에 매핑합니다. 외부 단편화가 없고 OS 관리가 단순합니다. **세그멘테이션**은 **논리 단위**(코드, 데이터, 스택)로 가변 크기 세그먼트를 둡니다. 내부 단편화는 적지만 **외부 단편화**가 생깁니다. 현대 OS는 **페이징 위에 세그먼트 개념을 혼합**하는 경우가 많습니다.

**핵심 키워드** — 논리≠물리 주소, 페이지·프레임, 외부/내부 단편화

**꼬리질문** — TLB 역할? → 페이지 테이블 캐시, 주소 변환 가속

### Q9. Page Fault 처리 과정

**질문** — Page Fault가 발생했을 때 OS가 처리하는 과정을 설명해 주세요.

**답변** — CPU가 가상 주소 접근 시 페이지 테이블에 **유효 비트가 0**이거나 메모리에 없으면 Page Fault 트랩이 발생합니다. OS는 ① **유효한 접근인지** 검사(보호 위반 vs 정상 fault), ② 디스크에 있으면 **페이지를 RAM 프레임에 로드**, ③ 페이지 테이블·TLB 갱신, ④ **중단된 명령 재실행**합니다. 프레임이 없으면 **페이지 교체 알고리즘**(LRU 등)으로 victim을 스왑 아웃합니다. 잦은 fault는 **스래싱**으로 이어집니다.

**핵심 키워드** — 트랩, 프레임 할당, 스왑 인/아웃, 재실행

**꼬리질문** — Minor vs Major fault? → Major는 디스크 I/O 필요, Minor는 이미 메모리에 있으나 매핑만 갱신

### Q10. Thread-Safe `[경험 연결]`

**질문** — 스레드 안전의 의미는 무엇이며, 파이썬이나 타입스크립트 코드 작성 시 고려해 본 적이 있나요?

**답변** — **Thread-Safe**는 여러 스레드가 동시에 접근해도 **데이터 정합성이 깨지지 않음**을 의미합니다. TypeScript/Node.js는 기본적으로 **싱글 스레드 이벤트 루프**라 대부분의 JS 객체는 스레드 안전 이슈가 적지만, **Worker Threads**나 **공유 ArrayBuffer**를 쓰면 동기화가 필요합니다. Python에서는 전역 dict/list를 멀티스레드로 수정할 때 Race가 날 수 있어 **`threading.Lock`**이나 **`queue`**를 사용합니다. [경험 연결] Express API에서는 요청마다 로컬 변수를 쓰고, 공유 캐시(Redis)는 단일 진입점으로 접근해 동시성을 관리했습니다.

**핵심 키워드** — 불변 객체, 락, 싱글 스레드 이벤트 루프, 공유 상태 최소화

**꼬리질문** — JS는 왜 대체로 안전? → 한 이벤트 루프에서 콜백 순차 실행 (단, async interleaving은 주의)

### Q11. System Call, User Mode vs Kernel Mode

**질문** — 시스템 콜이란 무엇이며, 유저 모드와 커널 모드의 차이는 무엇인가요?

**답변** — **시스템 콜**은 유저 프로그램이 **OS 커널 기능**(파일 I/O, 네트워크, 프로세스 생성)을 요청하는 인터페이스입니다. **유저 모드**는 권한이 제한되어 하드웨어·커널 메모리 직접 접근이 불가하고, **커널 모드**는 OS가 모든 자원에 접근할 수 있습니다. `read()`, `write()`, `socket()` 호출 시 **모드 스위치**가 일어나 커널이 대신 실행합니다. 잦은 시스템 콜은 성능 비용이 있어 **배치 I/O·비동기 I/O**로 줄입니다.

**핵심 키워드** — 권한 링, 모드 스위치, 커널 진입점

**꼬리질문** — printf도 시스템 콜? → 최종적으로 write() 등 커널 호출로 이어짐
