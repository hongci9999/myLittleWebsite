# 요구공학 (Requirements Engineering)

## 핵심 요약
소프트웨어 개발의 첫 단추인 **[[요구공학]]**의 4단계(도출, 분석, 명세, 검증)를 학습함. 특히 **[[유스케이스]]**의 구성 요소와 관계, 구조적 분석 도구인 **[[DFD]]**와 **[[DD]]**, 그리고 요구사항 검증 기법(동료검토, 워크스루, 인스펙션)의 차이점을 명확히 구분하는 것이 중요함.

---

## 1. 요구공학 개요

### 1) 목적 및 효과
* 이해관계자 간의 효과적인 소통 수단 제공.
* 불필요한 개발 비용 절감.
* **주도화(Baselin)** 된 요구사항은 변경 추적이 가능하며 요구사항 손실을 최소화함.

### 2) 요구공학 프로세스 (4단계)
1.  **[[요구사항 도출]]** (Elicitation)
2.  **[[요구사항 분석]]** (Analysis)
3.  **[[요구사항 명세]]** (Specification)
4.  **[[요구사항 검증]]** (Verification)

### 3) 관리 도구의 주요 기능
* 변경에 따른 영향 분석 (Impact Analysis).
* 변경 이력 추적.
* 우선순위 및 리스크 관리.
* 외부 연동 및 협업 환경 제공.

---

## 2. [[요구사항 도출]] (Elicitation)

### 1) 개념
* 시스템, 사용자, 개발자 등 관련된 사람들이 서로 의견을 교환하여 요구사항을 수집하는 과정.
* **이해관계자(Stakeholder)** 식별이 선행되어야 함.
* 사용자의 요구사항은 불명확하고 변하기 쉽기 때문에 다양한 기법을 사용해야 함.

### 2) 주요 기법
* 인터뷰, 설문조사.
* **[[사용자 스토리]]** (User Story).
* 업무 절차 조사.
* **[[브레인스토밍]]**: 비판 없이 자유롭게 아이디어 제시.
* **[[프로토타이핑]]**: 시제품을 통해 요구사항 확인.
* **[[유스케이스]]** (Use Case).

---

## 3. [[유스케이스]] (Use Case)

### 1) 유스케이스 다이어그램 (Diagram)
시스템의 기능을 **사용자 관점**에서 표현한 도표. 시스템 범위와 외부 요소(Actor)와의 상호작용을 파악할 수 있음.

#### ① 구성 요소
* **시스템 범위 (System Scope)**: 사각형으로 묶어 표현 (시스템 내부와 외부 구분).
* **액터 (Actor)**: 시스템과 상호작용하는 모든 외부 요소.
    * **주 액터 (Primary Actor)**: 시스템을 사용하여 이득을 얻는 대상 (주로 사람, 사용자).
    * **부 액터 (Secondary Actor)**: 주 액터의 목적 달성을 위해 시스템에 서비스를 제공하는 외부 시스템.
* **유스케이스 (Use Case)**: 사용자 관점에서 시스템이 제공하는 서비스나 기능을 타원으로 표현.
* **관계 (Relationship)**: 액터와 유스케이스, 유스케이스 간의 연관성.

#### ② 작성 시 주의사항
* 순서도(Flowchart)처럼 흐름을 그리지 않는다.
* 사용자 관점에서 최대한 간결하게 작성한다.

### 2) 유스케이스 기술서 (Description)
다이어그램의 각 유스케이스에 대한 상세 명세.

#### 주요 항목 및 AI 설명
* **기본 항목**: 유스케이스명, 액터명, 개요, 사전조건, 사후조건.
* **[[기본 흐름]] (Basic Flow)**:
    > [!INFO] 💡 AI 보충 설명: 기본 흐름
    > 오류나 예외 상황 없이 사용자의 목표가 가장 이상적으로 달성되는 절차입니다. 'Happy Path'라고도 부릅니다.
* **[[트리거]] (Trigger)**:
    > [!INFO] 💡 AI 보충 설명: 트리거
    > 유스케이스를 시작하게 만드는 사건이나 조건을 의미합니다. (예: 사용자가 '주문하기' 버튼을 클릭함)
* **[[대체 흐름]] (Alternative Flow)**:
    > [!INFO] 💡 AI 보충 설명: 대체 흐름
    > 기본 흐름에서 벗어나지만, 오류는 아닌 선택적인 흐름이나 예외 상황을 처리하는 절차입니다. (예: 결제 중 '잔액 부족' 발생 시 처리 과정)



### 3) 유스케이스의 관계 (Relationships)
> [!INFO] 💡 AI 보충 설명: 관계의 시각적 표현 및 의미

| 관계 유형 | 표기법 (UML) | 설명 |
| :--- | :--- | :--- |
| **[[포함 관계]]**<br>(Include) | 점선 화살표<br>`<<include>>` | - 하나의 유스케이스가 다른 유스케이스의 기능을 **반드시** 실행하는 경우.<br>- 여러 유스케이스에서 **공통적으로 사용되는 기능**을 별도로 추출할 때 사용.<br>- (방향: 원래 유스케이스 $\rightarrow$ 포함되는 유스케이스) |
| **[[확장 관계]]**<br>(Extend) | 점선 화살표<br>`<<extend>>` | - 특정 조건이 만족될 때만 **선택적으로** 실행되는 기능.<br>- (방향: 확장되는 기능 $\rightarrow$ 기준 유스케이스) |
| **[[일반화 관계]]**<br>(Generalization) | 속이 빈 실선 화살표<br>($\triangle$) | - 객체지향의 상속과 유사한 개념.<br>- 하위(구체적) 유스케이스가 상위(추상적) 유스케이스의 특성을 물려받음.<br>- (방향: 하위 $\rightarrow$ 상위) |

## 4. [[요구사항 분석]] (Analysis)

### 1) 개념
요구사항이 명확하지 않거나 이해관계자 간에 **상충(Conflict)**될 경우 이를 조정하고 해결하는 활동.

### 2) 요구사항 분류 (중요)
1.  **[[기능적 요구사항]] (Functional)**:
    * 시스템이 무엇을 해야 하는가? (What)
    * 시스템의 기능, 입력/출력, 저장 데이터 등.
2.  **[[비기능적 요구사항]] (Non-Functional)**:
    * 시스템이 어떻게 동작해야 하는가? (How)
    * 성능, 보안, 품질, 제약사항, 인터페이스 등.

* *표현 관점에 따른 분류*:
    * **사용자 요구사항**: 친숙한 표현 (사용자 관점).
    * **시스템 요구사항**: 기술적 용어와 표현 (개발자 관점).

### 3) 주요 분석 활동
* **개념 모델링**: 업무 처리의 실체(Entity)와 관계를 모델링 (UML 사용). 핵심 단계.
* **요구사항 할당**: 요구사항을 만족시키기 위한 아키텍처 구성요소 식별.
* **요구사항 협상**: 서로 상충되는 요구사항에 대해 우선순위를 부여하고 합의점을 찾음.
* **정형 분석**: 구문과 의미를 갖춘 정형화된 언어(수학적 기호)를 이용해 요구사항을 표현.

## 5. 구조적 분석 도구

도형을 중심으로 분석하며 **추상화, 정형화, 분할 정복, 계층화**의 원리를 가짐.

### 1) [[DFD]] (Data Flow Diagram, 자료 흐름도)
데이터의 흐름에 중점을 둠. (**시간의 흐름이나 제어 순서는 표현하지 않음**)

* **단말 (Terminator)**: 입출력 주체 (사각형 `□`).
* **프로세스 (Process)**: 데이터 처리 과정 (타원 `○` 또는 둥근 사각형).
* **자료 흐름 (Data Flow)**: 데이터 흐름 방향 (화살표 `→`).
* **자료 저장소 (Data Store)**: 데이터 저장 (평행선 `=`).



[Image of data flow diagram example]


### 2) [[DD]] (Data Dictionary, 자료 사전)
DFD에 있는 자료 흐름이나 저장소의 데이터를 더 자세히 정의한 **메타 데이터(Meta Data)**.

| 기호 | 의미 | 비고 |
| :---: | :---: | :--- |
| **=** | 정의 (Definition) | ~이다 |
| **+** | 연결 (Composition) | AND |
| **( )** | 생략 (Optional) | 없음 가능 |
| **[ \| ]** | 선택 (Selection) | OR (택일) |
| **{ }** | 반복 (Iteration) | |
| **\*** | 주석 (Comment) | 설명 |

### 3) 기타 도구
* **[[NS 차트]] (Nassi-Shneiderman Chart)**:
    * 논리 중심의 도형 표현.
    * **순차, 선택, 반복**의 3가지 제어 구조를 명확히 표현 (GOTO 사용 안 함).
* **[[HIPO]] (Hierarchy Input Process Output)**:
    * 시스템의 기능을 **계층 구조(Tree)**로 표현.
    * 입력, 처리, 출력을 중심으로 기능과 자료의 의존성을 동시에 표현.
    * **하향식(Top-Down)** 소프트웨어 개발에 유용.
    * **종류**: 가시적 도표(Visual), 총체적 도표(Overview), 세부적 도표(Detail).

## 6. [[요구사항 명세]] (Specification)

* **정형 명세 기법**: 수학적 기호, 정형화된 언어 사용. (정확성 높음, 이해 어려움)
* **비정형 명세 기법**: 자연어, 그림 중심. (친숙함, 의사소통 용이, 모호성 발생 가능)

## 7. [[요구사항 검증]] (Verification)

잘못된 요구사항으로 설계/구현된 경우 나중에 수정 비용이 막대하므로 필수적임.
* **검증 항목**: 유효성, 일관성, 완결성, 현실성, 검증 가능성 등.

### 1) 검증 기법 (Review) `시험 출제 빈도 높음`
1.  **[[동료 검토]] (Peer Review)**: 작성자가 명세서 내용을 직접 설명하고 동료들이 결함을 발견.
2.  **[[워크스루]] (Walkthrough)**: 검토 회의 전 명세서를 미리 배포하여 사전 검토 후, 짧은 회의를 통해 오류 검출.
3.  **[[인스펙션]] (Inspection)**: 저작자 외의 다른 **전문 검토 그룹**이 명세서를 상세히 분석하여 결함을 발견.

### 2) 기타 검증 방법
* **[[프로토타이핑]]**: 시제품을 신속하게 만들어 피드백 반영. (단점: 비용 증가, 사용성 과대평가 우려)
* **모델 검증**: 정적 분석(Static Analysis)의 일종. 실제 실행하지 않고 모델의 구조나 논리를 검증.
* **[[인수 테스트]]**: 실제 사용 환경에서 사용자가 직접 테스트하여 충족 여부 판단.
