# 애플리케이션 통합 테스트 및 테스트 결과 분석

## 핵심 요약
본 강의는 소프트웨어 개발 생명주기 중 검증 단계에 해당하는 **V-모델**을 기반으로 한 **통합 테스트**의 기법과 절차를 다룹니다. 특히 상향식·하향식 통합 방식의 차이점과 **테스트 하네스**의 구성 요소, 그리고 테스트 실행 후 발생하는 **결함 관리** 및 **코드 커버리지**의 유형을 상세히 학습합니다.

---

## 1. 애플리케이션 통합 테스트

### 1.1 V-모델 (V-Model)
개발 단계에 대응하는 테스트 단계를 시각화하여 소프트웨어의 품질을 단계별로 검증합니다.
- **요구사항 분석** → [[인수 테스트]]
- **시스템 설계** → [[시스템 테스트]]
- **아키텍처 설계** → [[통합 테스트]]
- **모듈 설계** → [[단위 테스트]]
- **구현** (V의 최하단)

### 1.2 단위 테스트 (Unit Test)
- 설계의 최소 단위인 **모듈**의 기능을 중심으로 테스트합니다.
- 내부 구조를 들여다보는 **화이트박스 테스트(White Box Test)** 방식을 주로 사용합니다.

### 1.3 통합 테스트 (Integration Test)
단위 테스트가 완료된 모듈들을 결합하여 설계 단계와 동일한 구조와 기능을 수행하는지 확인합니다.

| 구분 | **비점증적 방식 (Big-Bang)** | **점증적 방식 (Incremental)** |
| :--- | :--- | :--- |
| **특징** | 모든 모듈을 한꺼번에 통합하여 테스트 | 단계별로 모듈을 통합하며 테스트 |
| **장점** | 단기간 내 테스트 완료 가능 | 장애 위치 파악이 용이하고 체계적임 |
| **단점** | 결함 위치 파악 및 수정이 매우 어려움 | 테스트 드라이버나 스텁 등 보조 도구 필요 |

### 1.4 점증적 통합 방식 상세 비교

#### **하향식 통합 테스트 (Top-Down)**
- **절차:** 최상위 제어 모듈부터 아래로 내려가며 통합합니다.
- **수행 방식:**
    - **깊이 우선 방식(Depth-First):** 특정 경로의 종속 모듈들을 먼저 테스트.
    - **너비 우선 방식(Breadth-First):** 동일 레벨의 모듈들을 먼저 테스트.
- **특징:**
    - 아직 개발되지 않은 하위 모듈 대신 **[[STUB]]**(스텁)을 사용합니다.
    - 주요 모듈을 조기에 테스트하여 설계 결함을 빨리 발견할 수 있습니다.
    - 하위 모듈의 테스트가 불충분할 수 있고, 많은 스텁이 필요합니다.

#### **상향식 통합 테스트 (Bottom-Up)**
- **절차:** 최하위 모듈부터 상위로 올라가며 통합합니다.
- **수행 방식:**
    - 하위 모듈들을 **클러스터(Cluster)**로 묶어 테스트합니다.
- **특징:**
    - 상위 모듈의 역할을 대신하는 **[[DRIVER]]**(드라이버)가 필요합니다.
    - 모든 모듈이 개발되어야 진행 가능하지만, 장애 위치 파악이 쉽습니다.

> [!INFO] 💡 핵심 비교: 드라이버(Driver) vs 스텁(Stub)
> - **Driver:** 상향식 테스트에서 사용. 상위 모듈 역할을 하며 하위 모듈에 테스트 데이터를 전달하고 결과를 수신합니다.
> - **Stub:** 하향식 테스트에서 사용. 아직 개발되지 않은 하위 모듈을 대신하여 제어 모듈의 호출에 응답하는 최소한의 기능만 수행합니다.

### 1.5 회귀 테스트 (Regression Test)
- 통합 후 수정된 모듈이 다른 부분에 영향을 주지 않았는지 확인하기 위해 기존 테스트 케이스를 다시 수행합니다.
- **대상 선정:** 대표 샘플, 영향도가 높은 기능, 실제 수정이 발생한 모듈 위주로 선정합니다.

## 2. 시스템 및 인수 테스트
- **시스템 테스트 (System Test):** 목표 컴퓨터 시스템에서 전체 기능이 정상적으로 수행되는지 확인합니다.
- **인수 테스트 (Acceptance Test):** 사용자가 직접 시스템을 테스트하여 요구사항 충족 여부를 확인합니다.
    - **알파 테스트:** 개발자 장소에서 사용자가 통제된 환경 하에 수행.
    - **베타 테스트:** 실무 환경(제한되지 않은 환경)에서 다수의 사용자가 수행.

## 3. 테스트 자동화 도구 및 하네스

### 3.1 도구 유형
- **정적 분석 도구:** 코드를 실행하지 않고 소스 코드 자체를 분석 (구조 이해 중심).
- **테스트 실행 도구:** 스크립트를 통해 테스트를 자동화합니다.
    - **데이터 주도 접근:** 데이터 시트 기반 테스트.
    - **키워드 주도 접근:** 수행 동작과 데이터를 함께 관리하여 테일러링 가능.
- **성능 테스트 도구:** 시스템의 부하 및 응답 시간을 측정 (전문가 필요).

### 3.2 테스트 하네스 (Test Harness)
테스트를 지원하기 위한 환경 구성 요소의 총칭입니다.
- **[[Test Driver]]**: 상향식 테스트 시 상위 모듈 역할.
- **[[Test Stub]]**: 하향식 테스트 시 하위 모듈 역할.
- **Test Suites**: 테스트 케이스의 집합.
- **Test Case**: 입력값, 실행 조건, 기대 결과의 집합.
- **Test Script**: 자동화된 테스트 실행 절차.
- **Mock Object**: 특정 행위를 수행하도록 미리 정의된 가짜 객체.

## 4. 테스트 결과 분석 및 결함 관리

### 4.1 소프트웨어 결함 용어
- **오류(Error):** 사람(개발자)의 실수로 인해 발생한 문제.
- **결함(Defect/Bug):** 오류가 원인이 되어 제품 코드 내에 포함된 문제.
- **실패(Failure):** 제품 실행 시 결함으로 인해 나타나는 잘못된 결과.

### 4.2 결함 관리 지표 및 상태
- **심각도 단계:** Critical → Major → Normal → Minor → Simple
- **측정 지표:**
    - **결함 분포:** 특정 속성(모듈 등)에 따른 결함 수.
    - **결함 추세:** 시간에 따른 결함 발생 추이.
    - **결함 에이징:** 결함이 수정되지 않고 유지된 시간.
- **추적 상태 흐름:**
    1. **등록(Registered)**
    2. **검토(Reviewed)** → (수정 필요 시) **할당(Assigned)** → **수정(Resolved)** → **종료(Closed)**
    3. 만약 결함이 아니라면 **해제(Clarified)** 후 종료, 보류 시 **보류(Deferred)** 처리.

## 5. 테스트 커버리지 (Test Coverage)

### 5.1 유형
- **기능 기반 커버리지:** 전체 기능 중 실제 테스트된 기능의 비율 (100% 목표).
- **라인 커버리지:** 전체 소스 코드 라인 중 실행된 라인 수.
- **코드 커버리지:** 소스 코드의 내부 구조를 얼마나 테스트했는지 측정.

### 5.2 코드 커버리지 상세 (화이트박스 테스트)
1. **구문 커버리지 (Statement):** 모든 문장을 한 번 이상 실행.
2. **결정 커버리지 (Decision):** 전체 결정문(if 등)의 결과가 참(T)/거짓(F)이 되도록 수행.
3. **조건 커버리지 (Condition):** 결정문 내부의 개별 조건식들이 각각 T/F가 되도록 수행.
4. **조건/결정 커버리지:** 결정문 전체와 개별 조건식 모두 T/F가 되도록 수행.
5. **변형 조건/결정 커버리지 (MC/DC):** 각 개별 조건식이 다른 조건식에 상관없이 전체 결과에 독립적인 영향을 주도록 테스트 (가장 안전하고 엄격함).

---

## 3. 빈출 내용 정리
- **@상향식/하향식:** 상향식은 **드라이버**, 하향식은 **스텁**을 사용한다는 차이점은 단골 문제이므로 반드시 암기해야 합니다.
- **@스텁, 드라이버:** 각각 하위 모듈(Stub), 상위 모듈(Driver)의 대역임을 명확히 구분하십시오.
- **@결함:** 오류(사람 실수) -> 결함(코드 문제) -> 실패(현상)의 인과관계를 기억하십시오.
- **@커버리지:** 코드 커버리지의 5가지 단계(구문, 결정, 조건, 조건/결정, MC/DC)는 정의와 함께 우선순위를 파악해 두어야 합니다.
