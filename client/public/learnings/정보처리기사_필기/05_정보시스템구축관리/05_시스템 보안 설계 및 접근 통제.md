# 시스템 보안 설계 및 접근 통제

## 📌 핵심 요약
시스템의 안전성을 확보하기 위한 **보안 설계 원칙**과 사용자의 신원을 확인하는 **인증(Authentication)** 기술을 정리합니다. 또한 자원에 대한 접근 권한을 관리하는 **접근 통제(Access Control)** 정책(DAC, MAC, RBAC)과 보안 모델(BLP, Biba 등)의 차이점을 상세히 다룹니다.

---

## 📝 상세 정리

## 1. 시스템 보안 설계 및 구현

### 1.1 시스템 보안 설계 대상 및 원칙
시스템의 [[CIA]](기밀성, 무결성, 가용성)를 보장하기 위한 구체적인 설계 지침입니다.
- **계정 관리**: 디폴트(Default) 계정 사용을 제한하고, 개발 과정에서 생성한 임시 계정은 반드시 삭제합니다.
- **권한 관리**: 정기적으로 권한을 갱신하며, 최소 권한의 원칙을 준수합니다.
- **보안 설정**: 
    - 로그인 실패 시 사용자에게 **구체적인 실패 사유(예: 비밀번호 틀림)를 알리지 않음**으로써 무차별 대입 공격을 방어합니다.
    - 일정 시간 입력이 없을 경우 자동 **로그오프(Session Timeout)**를 설정합니다.

### 1.2 보안 계획의 3대 차원
- **관리적 차원**: 보안 정책 수립, 지침 제정, 보안 교육 등.
- **기술적 차원**: 암호화, 방화벽, 침입 탐지 시스템 등 기술적 솔루션 적용.
- **물리적 차원**: 출입 통제, 일정 장소 보안, CCTV 설치 등 물리적 자산 보호.

### 1.3 시스템 보안 구현 도구
- **[[MBSA]] (Microsoft Baseline Security Analyzer)**: 윈도우 시스템의 보안 설정 결함을 점검하는 도구.
- **[[Nmap]]**: 네트워크 스캐닝 도구로, 시스템에 열려 있는 포트(Port)를 확인.
- **[[NBTScan]]**: NetBIOS 네임 서버를 스캔하여 네트워크를 점검하는 도구.

## 2. 시스템 인증 (Authentication)

### 2.1 인증의 4가지 방법
- **지식 기반 ([[Something you know]])**: 사용자가 알고 있는 정보 (비밀번호, Passphrase).
- **소유 기반 ([[Something you have]])**: 사용자가 가진 물건 (OTP, 스마트카드, USB 토큰).
- **존재 기반 ([[Something you are]])**: 사용자의 생체적 특징 (지문, 홍채, 정맥).
- **행위 기반 ([[Something you do]])**: 사용자의 행동 패턴 (서명, 목소리, 타이핑 습관).

### 2.2 세션 및 인증 유지
- **[[Session]]**: 로그인부터 로그아웃까지 사용자 인증 상태가 유지되는 논리적 연결.
- **세션 가로채기 ([[Session Hijacking]])**: 활성화된 세션을 공격자가 탈취하는 공격.
    - **통제 기법**: 비동기화 탐지(서버와 시퀀스 넘버 체크), ACK Storm 탐지, 패킷 유실 및 재전송 증가율 감시, 예상치 못한 접속 리셋(RST) 탐지 등.

### 2.3 인증 및 무결성 도구
- **[[디지털 서명]] (Digital Signature)**: 송신자의 비밀키(개인키)로 암호화하고 수신자가 송신자의 공개키로 복호화.
    - 특징: 인증, 무결성 보장, **부인 방지**, 위조 불가능.
- **[[디지털 인증서]]**: 신뢰할 수 있는 인증기관([[CA]])이 발행한 전자적 증명서.
- **[[MAC]] (Message Authentication Code)**: 메시지의 무결성을 확인하기 위한 코드.
- **[[해시 함수]] (Hash Function)**: 임의 길이 메시지를 고정 길이의 해시값으로 변환하는 단방향 알고리즘.

## 3. 인증 관련 프로토콜

### 3.1 AAA 체계
- **[[Authentication]] (인증)**: 신원 확인.
- **[[Authorization]] (권한 부여)**: 자원 접근 허용.
- **[[Accounting]] (계정 관리)**: 사용 내용 기록 및 추적.

### 3.2 주요 프로토콜
- **[[PAP]]**: 아이디/패스워드를 평문으로 전송 (보안 취약).
- **[[CHAP]]**: 3-Way Handshaking 방식을 사용하여 보안성 강화.
- **[[EAP]]**: 다양한 인증 방식을 지원하기 위한 확장형 프레임워크.
- **[[RADIUS]]**: 분산된 환경의 중앙 집중식 AAA 관리 제공.
- **[[Diameter]]**: RADIUS의 후속 프로토콜로 확장성, 보안성, 장애 복구가 강화됨.

## 4. 접근 통제 (Access Control) 정책 및 모델

### 4.1 접근 통제 정책
| 구분 | 내용 | 특징 |
| :--- | :--- | :--- |
| **[[DAC]] (임의적 접근 통제)** | 데이터 소유자가 사용자 신분에 따라 권한 부여 | 유연함, ACL(Access Control List) 사용 |
| **[[MAC]] (강제적 접근 통제)** | 자산의 보안 등급과 사용자의 권한 등급 비교 | 매우 엄격, 관리 어려움, 성능 저하 |
| **[[RBAC]] (역할 기반 접근 통제)** | 사용자의 '역할(직무)'에 따라 권한 부여 | 관리 효율성 높음, 인사 이동 시 용이 |

### 4.2 접근 통제 모델
- **[[Bell-LaPadula]] (BLP) 모델**: **기밀성** 강조. (No Read Up / No Write Down)
- **[[Biba]] 모델**: **무결성** 강조. (No Read Down / No Write Up)
- **[[Chinese Wall]] (Chinn) 모델**: 이해 상충 관계에 있는 데이터 간의 접근을 차단.
- **[[Clark-Wilson]] 모델**: 무결성 중심, 상업적 용도(금융/회계)에 최적화된 모델.

---

> [!INFO] 💡 AI 보충 설명: BLP vs Biba 모델의 규칙
> - **BLP (기밀성)**: 상위 등급의 정보를 읽지 못하게 함으로써 정보 유출을 막습니다 ($Simple\ Security\ Property$).
> - **Biba (무결성)**: 하위 등급의 신뢰할 수 없는 정보를 읽지 못하게 하여 데이터 오염을 막습니다 ($Simple\ Integrity\ Axiom$). 두 모델은 규칙이 정반대이므로 혼동하지 않도록 주의해야 합니다.
