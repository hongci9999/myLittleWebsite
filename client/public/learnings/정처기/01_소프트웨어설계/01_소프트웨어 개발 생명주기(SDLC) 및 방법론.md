# 소프트웨어 개발 생명주기(SDLC) 및 방법론

## 핵심 요약
소프트웨어 개발의 전체 과정을 다루는 **[[SDLC]]**의 대표적 모델(폭포수, 프로토타입, 나선형, 애자일)을 정리하고, 구체적인 애자일 기법인 **[[스크럼]]**과 **[[XP]]**의 절차 및 실천 항목을 학습함. 또한 구조적, 정보공학, 객체지향, CBD 등 시대별 **소프트웨어 개발 방법론**과 시큐어 코딩을 위한 **보안 개발 방법론**을 다룸.

---

## 1. 소프트웨어 개발 생명주기 (SDLC)

### [[폭포수 모델]] (Waterfall Model)
* 과거에 가장 많이 사용된 보편적인 모델.
* **특징**: 단계별로 철저하게 검증 후 다음 단계로 진행 (이전 단계로 복귀 어려움).
* **단점**: 개발 도중 요구사항 변경이나 기능 보완이 어려움.
* **산출물**: 각 단계마다 **매뉴얼(문서화)** 작성이 필수.

[Image of SDLC models comparison waterfall spiral agile]

### [[프로토타입 모델]] (Prototype Model)
* **특징**: 정식 개발 전 **시제품(Prototype)**을 만들어 고객의 반응을 확인.
* **장점**: 사용자와 시스템 사이의 **인터페이스**에 중점을 두어 개발하므로 요구사항 불일치 감소.

### [[나선형 모델]] (Spiral Model)
* **특징**: 폭포수 모델과 프로토타입 모델의 장점에 **[[위험 분석]]** 기능을 추가.
* **과정**: 계획 수립 → 위험 분석 → 개발 및 검증 → 고객 평가 (반복).
* **목적**: 점진적 개발을 통해 프로젝트 수행 중 발생하는 **위험을 최소화**하는 것이 목적.

### [[애자일 모델]] (Agile Model) `중요`
* **개념**: 고객과의 **소통**에 중심을 둔 경량화된 방법론들의 통칭.
* **핵심**: 문서보다는 실행 가능한 소프트웨어, 계약 협상보다는 **고객 협업**, 계획 준수보다는 **변화 대응**에 가치를 둠.
* **특징**: 짧은 주기(Iteration/Sprint)로 개발하며 고객 피드백을 즉시 반영.

## 2. 애자일(Agile) 대표 기법

### [[스크럼]] (Scrum)
팀이 중심이 되어 개발의 효율성을 높이는 기법. 단기 프로젝트에 적합.

[Image of Scrum process flow]

#### 1) 구성원
* **[[제품 책임자]] (PO, Product Owner)**:
    * 요구사항을 작성하는 주체(의사결정권자).
    * **백로그(Backlog)** 작성 및 요구사항 우선순위 지정.
    * 시스템 이해도가 높아야 하며, 요구사항을 **User Story** 형태로 구체화.
* **[[스크럼 마스터]] (Scrum Master)**:
    * 일일 회의를 주관하고 팀의 장애 요소를 제거하는 가이드 역할.
* **[[개발팀]] (Development Team)**:
    * 실제 제품을 개발하는 팀원.

#### 2) 스크럼 개발 절차
1.  **제품 백로그 (Product Backlog)**: 모든 요구사항의 목록.
2.  **스프린트 계획 회의**: 백로그에서 단기(Sprint)에 수행할 작업을 선택하고 구체화.
3.  **스프린트 백로그**: 스프린트 기간 동안 수행할 작업 목록 작성.
4.  **[[스프린트]] (Sprint)**: 2~4주 기간의 실제 개발 진행.
5.  **일일 스크럼 회의**: 매일 짧게 진행하는 진행 상황 공유 및 의사소통.
6.  **스프린트 검토 회의**: 개발된 제품 시연 및 검토.
7.  **스프린트 회고**: 프로세스 개선점 논의.

### [[XP]] (eXtreme Programming)
고객의 참여와 짧은 개발 과정의 반복을 극대화하여 생산성을 높이는 방법.

[Image of eXtreme Programming process cycle]

* **특징**: 소규모 프로젝트에 적합.
* **주요 용어**:
    * **스파이크(Spike)**: 기술적 난제 해결을 위한 간단한 프로그램 개발.
    * **이터레이션(Iteration)**: 스크럼의 스프린트와 유사한 반복 주기.
    * **승인 검사(Acceptance Test)**: 이터레이션을 통해 완료된 제품을 고객이 직접 테스트.

#### XP의 5가지 핵심 가치
> 의사소통, 단순성, 용기, 피드백, 존중

#### XP의 12가지 실천 항목 (Practice)
1.  **Planning Game**: 게임처럼 즐겁게 계획 수립.
2.  **Small Release**: 짧은 주기로 지속적인 배포.
3.  **System Metaphor**: 시스템의 구조를 비유적으로 표현(공통 어휘 사용).
4.  **Simple Design**: 현재 요구사항에 적합한 가장 단순한 설계.
5.  **[[TDD]] (Test-Driven Development)**: 테스트 주도 개발 (테스트 코드를 먼저 작성).
6.  **Design Improvement**: **[[리팩토링]] (Refactoring)**을 통해 기능 변경 없이 코드 구조 개선.
7.  **Pair Programming**: 두 명이 짝을 이루어 코딩 (짝 프로그래밍).
8.  **Collective Ownership**: 모든 코드는 팀 전체의 소유 (공동 소유권).
9.  **[[CI]] (Continuous Integration)**: 지속적인 통합 (매일 여러 번 통합).
10. **Sustainable Pace**: 지속 가능한 개발 속도 유지 (야근 지양).
11. **Whole Team**: 고객을 포함한 모든 구성원이 한 팀.
12. **Coding Standards**: 공통된 코딩 표준 준수.

## 3. 소프트웨어 개발 방법론의 변천

[Image of software development methodologies comparison]

### 1) [[구조적 방법론]] (Structured Methodology)
* **특징**: 정형화된 분석 절차에 따라 사용자 요구사항을 파악하고 문서화.
* **도구**:
    * **[[DFD]]** (Data Flow Diagram, 자료 흐름도)
    * **[[DD]]** (Data Dictionary, 자료 사전)
    * **Mini-Spec** (소단위 명세서)
* **설계**: 모듈 중심 설계, **순차·선택·반복**의 논리 구조 구성.

### 2) [[정보공학 방법론]] (Information Engineering)
* **특징**: 정보 시스템의 개발을 위해 정형화된 기법들을 통합 적용 (데이터 중심).
* **도구**:
    * **[[ERD]]** (Entity Relationship Diagram, 개체 관계도)
    * 데이터베이스 설계를 위한 도구들 사용.
    * 프로세스 의존도(**PDD**), 프로세스 계층도(**PHD/PFD**).

### 3) [[객체지향 방법론]] (Object-Oriented Methodology)
* **개념**: 현실 세계의 **개체(Entity)**를 속성(Attribute)과 메소드(Method)가 결합된 **[[객체]] (Object)**로 표현.
* **주요 특징**:
    * **[[캡슐화]] (Encapsulation)**: 데이터와 연산을 하나로 묶음.
    * **[[정보은닉]] (Information Hiding)**: 다른 객체에게 자신의 상세 구현 정보를 숨김.
    * **[[추상화]] (Abstraction)**: 객체의 공통적인 속성과 기능을 추출하여 상위 객체로 정의.
    * **[[상속성]] (Inheritance)**: 상위 객체의 속성과 기능을 하위 객체가 물려받아 사용.
    * **[[다형성]] (Polymorphism)**: 하나의 메시지에 대해 객체마다 다르게 반응(Overloading, Overriding).

### 4) [[CBD]] (Component Based Development)
* **개념**: 기존에 개발된 **컴포넌트**들을 조립(Assembly)하여 새로운 프로그램을 개발.
* **장점**: 개발 기간 단축(신속성), 확장성(유연성), 표준화 용이.
* **단점**: 초기 컴포넌트 개발을 위한 선행 투자 비용 발생, 지적 재산권 문제.
* **단계별 산출물**:
    * 분석: 사용자 요구사항 정의서, 요구사항 명세서.
    * 설계: 설계서, 테스트 계획서, 시나리오, 기술서.
    * 구현: 소스 코드, 테스트 결과서, 테이블 정의서.
    * 시험: 시험 결과서, 사용자 지침서.

### 5) [[제품 계열 방법론]] (Product Line)
* **특징**: 특정 제품군(Product Family)에 적용하고 버전을 관리. 주로 **임베디드 소프트웨어** 개발에 사용.
* **구성**:
    * **영역 공학 (Domain Engineering)**: 핵심 자산 개발.
    * **응용 공학 (Application Engineering)**: 제품 개발.

## 4. 보안 개발 방법론 (Secure SDLC)

1.  **Code Review**: 코드 검토.
2.  **Architecture Risk Analysis**: 아키텍처 위험 분석.
3.  **Penetration Testing**: 침투 테스트.
4.  **Risk Based Security Testing**: 위험 기반 보안 테스트.
5.  **Abuse Cases**: 악용 사례 분석.
6.  **Security Requirements**: 보안 요구사항 정의.
