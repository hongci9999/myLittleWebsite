# AWS 배포 후 운영·보안·트러블슈팅

날짜: 2026-05-04
태그: [배포, aws, 인프라, 백엔드, 개발방법론]

## 요약

배포 후에는 **로그·메트릭·알람**으로 이상 징후를 보고, 문제가 나면 **HTTP 상태코드와 어느 레이어**(DNS, CDN, 로드밸런서, 앱)인지부터 가른다. 보안은 **최소 권한·시크릿 보관·불필요 공개**를 반복 점검한다.

---

## 핵심 개념

| 개념 | 설명 |
|------|------|
| **CloudWatch** | AWS 메트릭·로그 수집(서비스별 연동 범위는 다름) |
| **ALB 액세스 로그** | 로드밸런서 앞단 요청 기록 |
| **애플리케이션 로그** | Node `console` 또는 구조화 로깅 → 플랫폼 로그 스트림 |
| **502/504** | 보통 앱이 죽었거나, 타임아웃, 타겟 그룹 불건강 |
| **403** | S3/CloudFront 권한·OAC·버킷 정책 문제가 흔함 |
| **CORS 에러** | 브라우저 콘솔 메시지 + API 응답 헤더로 원인 좁힘(0031) |

---

## 상세 설명

### 1. 운영 습관

- **태그**: 리소스에 `project`, `env`, `owner` 등을 붙여 나중에 청소하기 쉽게 한다.
- **비용 알람**: 예산 초과 알림을 설정한다.
- **백업**: 상태가 S3에만 있으면 버킷 버저닝·수명 주기 정책을 검토한다. DB는 Supabase 측 백업 정책을 확인([0017](./0017-supabase.md)).

### 2. 보안 체크리스트 (요약)

- [ ] 루트·IAM 사용자에 불필요한 관리 정책이 붙어 있지 않은가
- [ ] 버킷이 **실수로 퍼블릭 읽기/쓰기**가 되지 않았는가(블록 퍼블릭 액세스 설정 확인)
- [ ] API 키·DB 키가 **클라이언트 번들**에 포함되지 않았는가
- [ ] 관리 엔드포인트(`/admin` 등)에 **인증**이 있는가

### 3. 트러블슈팅 순서 (권장)

1. **어느 URL**에서 문제가 나는가(프론트 vs API).
2. 브라우저 **네트워크 탭**: 상태코드, CORS, 리다이렉트 루프.
3. API라면 **로드밸런서 타겟 건강 상태**, 앱 로그에서 예외 스택.
4. 정적이라면 **CloudFront 캐시**, S3 키 경로, SPA 404 설정(0029).

### 4. 자주 하는 조치

- 설정 변경 후 **캐시 무효화** 또는 **시크릿 창**으로 재확인.
- 배포 직후 **이전 버전**으로 되돌리기(이전 태스크 정의, 이전 EB 버전 등 플랫폼 기능).
