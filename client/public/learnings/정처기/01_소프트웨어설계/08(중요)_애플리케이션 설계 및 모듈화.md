# 애플리케이션 설계 및 모듈화

## 핵심 요약
소프트웨어 설계의 기본 원칙과 **모듈화(Modularity)**의 핵심인 **[[응집도]]**와 **[[결합도]]**를 학습함. 특히 응집도는 높이고 결합도는 낮추는 것이 좋은 설계임을 이해하고, 각각의 단계별 순서(강도)와 특징을 암기하는 것이 시험의 핵심 포인트임. 또한 소프트웨어 **재사용**과 **재공학**의 개념을 다룸.

---

## 1. 소프트웨어 설계 개요

### 1) 설계 유형 비교
* **절차지향 (Procedural)**:
    * 순차적인 처리를 중시.
    * 장점: 속도가 빠름.
    * 단점: 유지보수가 어려움.
* **객체지향 (Object-Oriented)**:
    * 데이터와 기능을 객체로 캡슐화.
    * 장점: 코드 재활용성이 높고 유지보수가 쉬움.
    * 단점: 설계 난이도가 높고 절차지향에 비해 속도가 느릴 수 있음.

### 2) 소프트웨어 설계 모델
1.  **데이터 설계**: 데이터 구조 및 DB 설계.
2.  **[[아키텍처]] 설계**: 시스템의 전체적인 구조.
3.  **[[인터페이스]] 설계**: 시스템 내/외부 및 사용자와의 연결.
4.  **절차 설계**: 내부 로직 및 알고리즘.

### 3) 설계의 단계
* **상위 설계 (High-Level)**: 아키텍처 설계, 예비 설계. 시스템 구조, DB, 화면 레이아웃 등.
* **하위 설계 (Low-Level)**: 모듈 설계, 상세 설계. 컴포넌트 내부 로직 등.

### 4) [[추상화]] (Abstraction)
복잡한 문제를 대략적이고 포괄적으로 구상하여, 구체화를 효율적으로 수행하게 하는 기법.
* **제어 추상화**: 제어의 정확한 메커니즘을 정의하지 않고 포괄적으로 표현.
* **과정 추상화**: 전반적인 흐름만 기술.
* **자료 추상화**: 데이터의 세부사항을 정의하지 않고 대표적인 구조로 표현.

---

## 2. [[모듈화]] (Modularity) `중요`

시스템을 분해하고 추상화하여 독립적인 기능 단위인 **모듈(Module)**로 구성하는 기법.



* **특징**:
    * 모듈의 개수가 많아지면: 모듈 크기는 작아지지만, 모듈 간 **통합 비용(Interface Cost)**은 상승한다.
    * 모듈의 개수가 적어지면: 통합 비용은 줄지만, 모듈 하나의 복잡도와 개발 비용이 상승한다.
    * **결론**: 적당한 모듈 개수를 찾아 총비용을 최소화해야 한다.
* **설계 목표**:
    * 사용법을 공개하고 유지보수가 편해야 함.
    * **핵심 원칙**: **[[응집도]]는 높이고(Max), [[결합도]]는 낮추도록(Min) 설계한다.**

---

## 3. [[소프트웨어 재사용]] (Reuse)

이미 검증된 기능을 재구성하여 시스템에 적용하는 것. (함수, 객체, 컴포넌트, 애플리케이션 등)

### 재사용 기술의 종류
1.  **생성 중심 기술 (Generative)**: 추상화 명세를 구체화하여 프로그램을 생성.
2.  **합성 중심 기술 (Compositional)**: 모듈을 블록처럼 조립하여 구성.

> [!INFO] 💡 AI 보충 설명: 생성 vs 합성 중심
> * **생성 중심 (Generative)**: "설계도(패턴/수식)를 넣으면 코드가 튀어나오는 방식". 구체적인 구현 코드를 자동으로 생성해주는 도구나 프로그램을 이용하는 방법입니다. (예: Source Generator)
> * **합성 중심 (Compositional)**: "레고 블록 조립 방식". 이미 만들어진 부품(모듈, 컴포넌트)을 끼워 맞춰서 완성하는 방법입니다. (예: 라이브러리 import 하여 사용)

---

## 4. [[소프트웨어 재공학]] (Re-engineering)

새로운 개발이 아닌, 기존 시스템을 이용하여 보다 나은 시스템을 구축하는 유지보수 측면의 활동. (소프트웨어 위기 해결책)

### 1) 주요 활동 (과정)
1.  **분석 (Analysis)**: 기존 시스템 명세 확인 및 재공학 가치 판단.
2.  **재구성 (Restructuring)**: 기능 변경 없이 코드 구조만 개선 (리팩토링).
3.  **[[역공학]] (Reverse Engineering)**: 소스 코드가 없는 상태(외계인 코드)에서 소스 코드를 복구하거나 설계도를 추출.
4.  **이식 (Migration)**: 다른 플랫폼에서 사용할 수 있도록 변환.

---

## 5. 공통 모듈 설계

### 1) 식별 프로세스
1.  단위 업무 기능 분석.
2.  업무 기능 표준화 (누락/중복 검토).
3.  유스케이스 분석 (포함 관계 `<<include>>` 확인).
4.  검토 회의 및 담당자 선정.
5.  상세 기능 명세 및 통합.

### 2) 명세 원칙 (5가지) `암기`
* **정확성 (Correctness)**: 기능이 정확해야 함.
* **명확성 (Clarity)**: 중의적 해석 배제.
* **완전성 (Completeness)**: 누락이 없어야 함.
* **일관성 (Consistency)**: 모순이 없어야 함.
* **추적성 (Traceability)**: 요구사항 출처 추적 가능.

## 6. [[응집도]] (Cohesion) `매우 중요`

모듈 **내부** 요소들이 서로 얼마나 밀접하게 관련되어 있는가. (높을수록 좋음)

### 응집도 순서 (강함/좋음 → 약함/나쁨)
> **암기 팁**: **기 - 순 - 통 - 절 - 시 - 논 - 우**

1.  **[[기능적 응집도]] (Functional)**: 모듈 내부의 모든 기능 요소가 **단일 문제 해결**을 위해 수행됨. (가장 이상적)
2.  **[[순차적 응집도]] (Sequential)**: 모듈 내 하나의 활동 출력값이 다음 활동의 **입력값**으로 사용됨.
3.  **[[통신적/교환적 응집도]] (Communicational)**: 동일한 **입출력 데이터**를 사용하여 서로 다른 기능을 수행.
4.  **[[절차적 응집도]] (Procedural)**: 모듈 안의 기능들이 다수 관련되어 있고, **순차적**으로 수행됨. (데이터 흐름 연관성 적음)
5.  **[[시간적 응집도]] (Temporal)**: 특정 **시간**에 함께 수행되어야 하는 기능들을 모음. (예: 초기화, 종료 루틴)
6.  **[[논리적 응집도]] (Logical)**: 유사한 성격이나 형태의 기능을 모아둠. (예: 모든 입력을 처리하는 모듈)
7.  **[[우연적 응집도]] (Coincidental)**: 아무 관련 없는 기능들이 모임. (가장 나쁨)

---

## 7. [[결합도]] (Coupling) `매우 중요`

모듈과 **모듈 간**의 관련성. (낮을수록/약할수록 독립성이 높아 좋음)



### 결합도 순서 (약함/좋음 → 강함/나쁨)
> **암기 팁**: **자 - 스 - 제 - 외 - 공 - 내**

1.  **[[자료 결합도]] (Data)**: 모듈 간의 인터페이스가 **자료 요소(파라미터, 인수, 매개변수)**로만 구성됨.
    > [!INFO] 💡 AI 예시
    > `Call(int a, int b)` 처럼 단순한 값만 넘겨주는 경우. 가장 이상적입니다.

2.  **[[스탬프 결합도]] (Stamp)**: 모듈 간의 인터페이스로 **자료 구조(배열, 리스트, 객체 등)** 전체를 전달함.
    > [!INFO] 💡 AI 예시
    > `Call(struct User)` 처럼 User 구조체 전체를 넘길 때, 해당 함수가 User 안의 일부 데이터만 쓰더라도 구조체 변경 시 영향을 받습니다.

3.  **[[제어 결합도]] (Control)**: 값뿐만 아니라 **제어 요소(Flag, 제어 신호)**를 전달하여 상대 모듈의 논리적 흐름을 제어함.
    > [!INFO] 💡 AI 예시
    > `Print(Target, isBold)`에서 `isBold`가 true면 굵게, false면 얇게 출력하도록 내부 로직을 좌지우지하는 경우.

4.  **[[외부 결합도]] (External)**: 어떤 모듈이 다른 모듈 내부의 **외부 변수(External Variable)**를 참조함.
    > [!INFO] 💡 AI 예시
    > C언어의 `extern` 키워드를 사용하여 다른 파일에 있는 변수를 가져다 쓰는 경우.

5.  **[[공통 결합도]] (Common)**: 여러 모듈이 **공통 데이터 영역(전역 변수)**을 함께 사용함.
    > [!INFO] 💡 AI 예시
    > 전역 변수(Global Variable) 하나를 여러 함수가 동시에 읽고 쓰다가 값이 꼬이는 상황.

6.  **[[내용 결합도]] (Content)**: 한 모듈이 다른 모듈의 **내부 기능이나 데이터를 직접 참조**하거나 수정함.
    > [!INFO] 💡 AI 예시
    > `goto` 문을 사용하여 다른 함수의 중간으로 점프해버리는 경우. 절대 피해야 합니다.

## 8. 복잡도와 Fan-In / Fan-Out

### 1) 개념
* **[[Fan-In]] (공유도/팬인)**: 어떤 모듈을 **호출하는** 모듈의 수. (들어오는 화살표)
    * Fan-In이 높다 = 많은 모듈이 나를 사용한다 = **재사용성이 좋다**.
    * 단, 단일 장애점(SPOF)이 될 수 있으므로 중점 관리가 필요함.
* **[[Fan-Out]] (제어도/팬아웃)**: 어떤 모듈이 **호출하는** 모듈의 수. (나가는 화살표)
    * Fan-Out이 높다 = 내가 너무 많은 일을 시킨다 = 복잡도가 높다.
    * 단순화시킬 필요가 있음.

### 2) 계산 예시 (사용자 노트 기반)
* **구조**:
    * A가 B, C, D를 호출함.
    * C가 F, G, J를 호출함.
    * H가 B, E를 호출함.
    * (B는 A와 H에게 호출당함)

* **분석 결과**:

| 모듈 | Fan-In (들어옴) | Fan-Out (나감) | 비고 |
| :---: | :---: | :---: | :--- |
| **A** | 0 | 3 (B,C,D) | 최상위 모듈 |
| **C** | 1 (from A) | 3 (F,G,J) | 중간 모듈 |
| **H** | 0 | 2 (B,E) | 다른 상위 모듈 |
| **B** | 2 (from A,H) | 0 | 공통 모듈 성격 |

## 9. 핵심 단어 정리
- **응집도(Strong->Weak)**: 기-순-통-절-시-논-우 (높을수록 좋음).
- **결합도(Weak->Strong)**: 자-스-제-외-공-내 (낮을수록 좋음).
- **재공학 3R**: Reverse-engineering, Re-engineering, Reuse.
- **Fan-In**: 높으면 재사용성 좋음 / **Fan-Out**: 높으면 복잡도 높음.
