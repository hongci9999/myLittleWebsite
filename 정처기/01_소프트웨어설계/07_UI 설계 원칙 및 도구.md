# UI 설계 원칙 및 도구

## 핵심 요약
UI 설계의 **4대 원칙(직관성, 유효성, 학습성, 유연성)**을 중심으로 설계 프로세스와 문서화(시나리오) 요건을 학습함. 특히 사용자의 의도와 시스템 실행 간의 격차인 **실행 차와 평가 차**를 줄이는 방법, 그리고 **[[와이어프레임]]**, **[[스토리보드]]**, **[[목업]]** 등 설계 도구의 차이점을 명확히 구분해야 함.

---

## 1. [[UI 설계 원칙]] (4가지) `중요`

사용자가 시스템을 편리하고 정확하게 사용할 수 있도록 하는 4가지 핵심 원칙.

1.  **[[직관성]] (Intuitiveness)**
    * 누구나 쉽게 이해하고, **즉시 사용**할 수 있어야 한다.
2.  **[[유효성]] (Efficiency)**
    * 사용자의 **목적을 정확하고 완벽하게 달성**할 수 있어야 한다.
3.  **[[학습성]] (Learnability)**
    * 초보자도 **쉽게 배우고** 익힐 수 있어야 한다.
4.  **[[유연성]] (Flexibility)**
    * 사용자의 요구사항을 최대한 수용하고, **오류를 최소화**해야 한다. (실수 포용)

> [!INFO] 💡 AI 보충 설명: 설계 지침
> * **사용자 중심 설계**: 개발자 편의가 아닌 사용자 관점에서 설계.
> * **오류 최소화**: 사용자가 실수하더라도 시스템 고장으로 이어지지 않도록 적은 노력으로 복구 가능하게 구현.

## 2. UI 설계 절차

### 1) 일반적인 설계 프로세스
1.  **문제 정의**: 무엇을 해결할 것인가?
2.  **사용자 모델링**: 사용자를 특성(초급, 중급, 숙련자 등)에 따라 그룹으로 분류.
3.  **작업 분석**: 해결해야 할 문제를 세부적으로 나눔.
4.  **컴퓨터 장치 및 기능 정의**: 사용할 하드웨어 및 SW 기능 정의.
5.  **사용자 인터페이스 정의**: 입출력 장치 식별, 시스템 상태 표시 방법 정의.
6.  **디자인 평가**: 사용성 테스트 등.

### 2) 흐름 설계 및 세부 프로세스
* 기능 식별 → 입력 요소 식별 → **UI 요구사항 식별** → **[[유스케이스]]** 설계 → 기능 및 양식 확인.

### 3) UI 화면 구성 요소 (WIMP)
* **윈도우 (Window)**: 독립적인 박스 형태의 표시 영역.
* **메뉴 (Menu)**: 사용할 수 있는 기능을 텍스트로 나열한 영역.
* **아이콘 (Icon)**: 기능이나 정보를 작은 이미지로 표현한 것.
* **포인터 (Pointer)**: 마우스 커서 등 정보 입력이 이뤄지는 위치 표시.

## 3. [[유용성]] (Usability)과 실행/평가 차

### 1) 유용성의 정의
사용자가 목표를 얼마나 효과적으로 달성했는가에 대한 척도.

### 2) 유용성을 저해하는 원인 (Gap)
사용자(User)와 시스템(System) 간의 생각 차이로 인해 발생.

#### ① [[실행 차]] (Gulf of Execution)
* **개념**: 사용자가 하려는 행동(의도)과 시스템이 실제로 제공하는 기능(실행)이 달라서 생기는 차이.
* **해결 방안**:
    * 불필요한 기능 제거.
    * 목표 달성 단계를 최소화.
    * 사용자의 의도대로 순차적 실행 보장.
    * 과도한 상호작용 배제.

#### ② [[평가 차]] (Gulf of Evaluation)
* **개념**: 시스템이 수행한 결과와 사용자가 기대한 결과가 달라서 생기는 차이. (결과를 사용자가 이해하지 못함)
* **해결 방안**:
    * UI 변화를 즉각적이고 직접적으로 파악할 수 있게 피드백 제공.
    * 시스템의 현재 상태를 이해하기 쉽게 표시.

## 4. [[UI 시나리오]]

### 1) 개념
기능 구조, 대표 화면, 화면 간의 **상호작용 흐름(Flow)** 등을 문서화한 것. 누구나 한눈에 쉽게 이해할 수 있어야 하며 일반적인 규칙(표준)을 따른다.

### 2) 문서 작성 요건 (5가지)
1.  **완전성**: 기능에 대한 누락이 없어야 한다. (빠짐없이 작성)
2.  **일관성**: UI 스타일, 용어, 포맷이 전체적으로 한결같아야 한다.
3.  **이해성**: 누구나 쉽게 이해할 수 있어야 한다. (추상적 표현 지양)
4.  **가독성**: 여백, 들여쓰기, 줄 간격 등을 조정하여 읽기 편해야 한다.
5.  **수정 및 추적 용이성**: 변경 사항을 쉽게 추적하고 수정할 수 있어야 한다.

## 5. UI 설계 도구 `매우 중요`

기획 단계에서 아이디어를 구체화하고 전달하기 위한 도구들.

| 도구명 | 특징 | 비고 |
| :--- | :--- | :--- |
| **[[와이어프레임]]**<br>(Wireframe) | - 화면 단위의 **대략적인 뼈대**(레이아웃) 설계.<br>- 선(Line)과 면 위주로 간략하게 표현. | 기획 초기 단계 |
| **[[목업]]**<br>(Mockup) | - 실제 화면과 유사하게 만든 **정적인(Static) 모델**.<br>- 디자인, 색상 등을 입혔으나 **기능은 작동하지 않음**. | 시각적 디자인 검토 |
| **[[스토리보드]]**<br>(Storyboard) | - 와이어프레임에 **설명(Description)**과 **화면 간 이동 흐름**을 추가한 문서.<br>- 개발자/디자이너 간의 **최종 의사소통 문서**. | **가장 상세한 설계서** |
| **[[프로토타입]]**<br>(Prototype) | - 실제 구현 전, 핵심 기능만 작동하도록 만든 **간단한 시제품**.<br>- 동적인(Dynamic) 상호작용 테스트 가능. | 인터랙션 검증 |
| **유스케이스**<br>(Use Case) | - 사용자 요구사항을 **기능 단위**로 표현한 다이어그램 또는 명세서. | 요구사항 분석용 |

## 6. [[HCI]] / [[UX]] / 감성공학

### 1) 감성공학
인간의 감성을 정량적으로 분석하여 제품 설계(UI)에 반영하는 기술.

### 2) [[HCI]] (Human Computer Interaction)
* 인간과 컴퓨터(시스템) 간의 상호작용을 연구하여, 시스템을 더 **편리하고 안전하게** 사용할 수 있도록 하는 학문.
* **목표**: 최적의 사용자 경험(UX) 제공.

### 3) [[UX]] (User Experience)
* 시스템을 이용하면서 사용자가 느끼게 되는 **종합적인 경험** (지각, 반응, 행동, 감정 등).
* **UI vs UX**: UI는 편의성/기술(수단), UX는 만족감/감정(결과)에 초점.

#### UX의 3가지 특징
1.  **주관성 (Subjectivity)**: 사람마다 느끼는 경험이 다름.
2.  **정황성 (Contextuality)**: 사용 환경이나 주변 상황에 따라 경험이 달라짐.
3.  **총체성 (Holistic)**: 개별적인 기능이 아닌, 심리적/이성적 요인이 복합된 총체적인 경험임.
