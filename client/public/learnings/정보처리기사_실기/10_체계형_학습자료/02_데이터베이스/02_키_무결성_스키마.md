# 키, 무결성, 스키마

## 핵심 요약

데이터베이스 문제에서 키는 "튜플을 어떻게 식별하는가"를, 무결성은 "데이터가 어떻게 일관되게 유지되는가"를 설명한다. 실기에서는 후보키/기본키/외래키 구분과 개체 무결성, 참조 무결성, 스키마 3계층이 핵심이다.

## 왜 중요한가

- 키는 정규화와 SQL 제약조건 문제의 출발점이다.
- 무결성은 `PRIMARY KEY`, `FOREIGN KEY`, `CASCADE`, `RESTRICT` 해석과 바로 연결된다.
- 스키마 3계층은 단답형 출제 빈도가 매우 높다.

## 1. 스키마 3계층

| 스키마 | 관점 | 특징 |
|---|---|---|
| 외부 스키마 | 사용자/응용 프로그램 | 여러 개 존재 가능 |
| 개념 스키마 | 전체 논리 구조 | 데이터베이스당 1개 |
| 내부 스키마 | 물리 저장 구조 | 실제 저장 방식 정의 |

### 기억 포인트

- 외부: 사용자 관점
- 개념: 전체 논리 구조
- 내부: 물리 저장 구조

## 2. 데이터 독립성

- 논리적 데이터 독립성: 개념 스키마가 바뀌어도 외부 스키마에 영향이 적음
- 물리적 데이터 독립성: 내부 스키마가 바뀌어도 개념 스키마에 영향이 적음

시험에서는 보통 "저장 방식 변경이 응용 프로그램에 영향을 주지 않는 것"을 물리적 데이터 독립성으로 묻는다.

## 3. 키의 종류

| 키 | 특징 |
|---|---|
| 후보키 | 유일성과 최소성을 만족 |
| 기본키 | 후보키 중 대표로 선택된 키 |
| 대체키 | 후보키 중 기본키를 제외한 나머지 |
| 슈퍼키 | 유일성은 만족, 최소성은 불만족 가능 |
| 외래키 | 다른 릴레이션의 기본키를 참조 |

## 4. 유일성과 최소성

- 유일성: 각 튜플을 서로 구별할 수 있어야 함
- 최소성: 꼭 필요한 속성만으로 구성돼야 함

예를 들어 `학번`만으로 학생을 구별할 수 있다면:

- `학번`은 후보키 가능
- `학번 + 이름`은 유일할 수 있어도 최소성이 없으므로 슈퍼키

## 5. 무결성 제약조건

### 5.1 개체 무결성

- 기본키는 중복될 수 없고 NULL일 수 없다.

### 5.2 참조 무결성

- 외래키 값은 참조 테이블의 기본키 값이거나 NULL이어야 한다.

예:

- `사원.부서코드` 가 `부서.부서코드`를 참조한다면
- 존재하지 않는 부서코드를 사원 테이블에 넣으면 참조 무결성 위반

## 6. 예시로 이해하는 키

테이블: `수강(학번, 과목코드, 이름, 성적)`

가정:

- 한 학생은 여러 과목 수강 가능
- 한 과목은 여러 학생 수강 가능

이 경우:

- `학번`만으로는 한 행 식별 불가
- `과목코드`만으로도 식별 불가
- `학번 + 과목코드` 조합으로 행 식별 가능
- 따라서 후보키는 `(학번, 과목코드)` 가 된다.

## 7. 자주 틀리는 함정

### 7.1 후보키와 기본키를 같은 것으로 착각

- 후보키는 될 수 있는 키들의 집합
- 기본키는 그중 실제 선택된 하나

### 7.2 슈퍼키에 최소성이 있다고 생각

- 슈퍼키는 최소성까지 요구하지 않는다.

### 7.3 외래키는 항상 중복 불가라고 오해

- 외래키는 중복될 수 있다.
- 여러 행이 같은 부모 키를 참조할 수 있기 때문이다.

## 8. 시험장에서 푸는 순서

1. 문제에서 각 행을 유일하게 구분할 수 있는 속성 조합을 찾는다.
2. 최소 속성만 남기면 후보키가 된다.
3. 기본키와 외래키를 구분한다.
4. 스키마 문제는 사용자/논리/물리 관점으로 나눠 본다.
5. 무결성 문제는 "중복 금지/NULL 금지/참조 존재 여부" 순으로 판단한다.

## 한 페이지 요약

- 외부/개념/내부 스키마는 사용자/논리/물리 관점이다.
- 후보키는 유일성+최소성, 슈퍼키는 유일성만 본다.
- 기본키는 대표 후보키, 외래키는 다른 릴레이션의 기본키를 참조한다.
- 개체 무결성은 기본키, 참조 무결성은 외래키 중심으로 본다.
- 키와 무결성은 정규화와 제약조건 문제의 기반이다.
