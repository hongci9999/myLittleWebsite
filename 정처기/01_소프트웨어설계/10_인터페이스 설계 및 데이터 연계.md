# 인터페이스 설계 및 데이터 연계

## 핵심 요약
서로 다른 시스템 간에 정보를 주고받기 위한 **[[인터페이스]]**의 설계 원칙과 구성 요소를 학습함. 특히 **직접 연계**와 **간접 연계(EAI)**의 장단점 비교, **연계 기술(DBLink, API, Socket 등)**의 종류, 그리고 인터페이스 설계서(목록/정의서)의 구성 항목이 주요 출제 포인트임.

---

## 1. 인터페이스(Interface) 개요

### 1) 정의
서로 다른 두 시스템, 장치, 소프트웨어 사이에서 정보나 신호를 주고받을 수 있게 하는 **물리적, 논리적 매개체**.
* **시스템 인터페이스**: 시스템 간의 연동.
* **[[UI]] (User Interface)**: 사용자와 시스템 간의 연동.

### 2) [[연계 시스템]] 구성

1.  **송신 시스템**: 데이터를 생성하여 전송하는 시스템.
2.  **중계 시스템**: 송신 시스템과 수신 시스템 사이에서 데이터를 변환하거나 매핑하는 시스템 (예: 연계 서버).
3.  **수신 시스템**: 데이터를 받아 업무에 활용하는 시스템.

---

## 2. 송수신 데이터 식별

### 1) 식별 대상 데이터
시스템 간에 교환되는 데이터는 **표준 전문** 형식을 따름.

* **전문 공통부 (Common)**: 시스템 간 연동 시 공통적으로 필요한 정보 (표준 코드, 송수신 시간 등).
* **전문 개별부 (Individual)**: 실제 업무를 수행하는 데 필요한 구체적인 데이터.
* **전문 종료부 (End)**: 전송 데이터의 끝을 알리는 표시.

### 2) 인터페이스 설계를 위한 DB 산출물
* **[[개체 정의서]]**: 데이터베이스 개념 모델링 단계의 산출물 (Entity 정의).
* **[[테이블 정의서]]**: 논리/물리 설계 단계의 테이블 구조 명세.
* **[[코드 정의서]]**: 공통으로 사용하는 코드에 대한 명세.

> [!INFO] 💡 AI 보충 설명: 도메인(Domain)
> * **[[도메인]]**: 하나의 속성(Attribute)이 가질 수 있는 **데이터의 유형(Type)과 범위(Range)**, 제약조건 등을 지정한 것. (예: '성별' 속성의 도메인은 '남/여', '나이' 속성의 도메인은 '0 이상의 정수')

## 3. 인터페이스 상세 설계 (연계 방식) `중요`



### 1) [[직접 연계 방식]]
중계 서버 없이 송신 시스템과 수신 시스템이 직접 연결되는 방식.

* **장점**:
    * 데이터 처리 속도가 빠름.
    * 구현이 단순함.
    * 개발 비용이 상대적으로 저렴함.
* **단점**:
    * 시스템 변경에 민감함 (한쪽이 바뀌면 다른 쪽도 수정 필요).
    * 통합된 보안이나 환경 구축이 어려움.

### 2) [[간접 연계 방식]]
**[[EAI]]** 서버나 연계 솔루션을 통해 시스템을 연결하는 방식.

* **장점**:
    * 서로 다른(이질적인) 환경의 시스템과 연계 용이.
    * 시스템 변경 시 유연한 대처 가능.
    * 보안 및 통합 관리 용이.
* **단점**:
    * 중계 서버를 거치므로 성능 저하 가능성 있음.
    * 구축 및 유지보수 비용이 높음.

---

## 4. [[시스템 연계 기술]]

직접 또는 간접 연계를 위해 사용되는 구체적인 기술들.

1.  **[[DB Link]]**: DB에서 제공하는 링크 객체를 이용해 수신 측 DB 테이블을 직접 참조하는 방식.
2.  **DB Connection**: 수신 측 DB의 연결 객체(Connection Pool)를 이용해 프로그래밍 언어에서 접속.
3.  **[[API]] / Open API**: 송신 시스템이 DB 데이터를 읽어와 제공하는 인터페이스 프로그램.
4.  **[[JDBC]]**: Java 언어에서 DB에 접속하기 위한 표준 API.
5.  **Hyper Link**: 웹 애플리케이션에서 링크를 통해 이동.
6.  **[[Socket]]**: 서버와 클라이언트가 특정 포트(Port)를 통해 연결하여 통신하는 기술.
7.  **[[Web Service]]**: WSDL, SOAP, UDDI 등을 이용하여 네트워크 상에서 서로 다른 기종 간 통신 지원.

## 5. 연계 통신 유형 및 오류 처리

### 1) 연계 통신 유형
* **실시간 처리 방식**:
    * **단방향**: 요청만 보내고 응답이 필요 없음.
    * **동기 (Sync)**: 요청 후 응답이 올 때까지 대기했다가 바로 처리.
    * **비동기 (Async)**: 요청 후 신호를 받으면 다시 처리 (대기하지 않음, 많은 데이터 처리에 적합).
* **지연 처리 방식**: 데이터를 즉시 처리하지 않고 지연 처리가 필요한 경우.
* **[[배치 처리]] (Batch)**: 처리할 작업을 모아두었다가 정해진 시간에 일괄 수행.

### 2) 인터페이스 오류 관리
* **오류 유형**:
    * 서버 오류 (Server Down 등).
    * 연계 프로그램 오류.
    * 연계 데이터 오류 (데이터 형식 불일치 등).
* **처리 절차**: 오류 발생 시 **로그 파일(Log File)**에 오류 코드와 내용을 기록하고, 관리자에게 알림 또는 재처리 프로세스 수행.

## 6. 인터페이스 설계서 작성

최종 산출물로서 두 가지 형태의 문서가 존재함.

### 1) [[인터페이스 목록]]
* 연계 업무를 식별하고 전체적인 현황을 관리하는 문서.
* **주요 항목**: 송신/수신 시스템 정보, 연계 방식, 통신 유형 등.

### 2) [[인터페이스 정의서]] (명세서)
* 인터페이스별로 상세한 정보를 기록한 문서.
* **주요 항목**:
    * 데이터 속성 (데이터 타입, 길이, 필수 여부 등).
    * 데이터 저장소의 상세 내역 (어느 테이블, 어느 컬럼에 매핑되는지).

> [!INFO] 💡 AI 보충 설명: 설계서 작성 팁
> 인터페이스 정의서는 개발자가 실제 코딩을 하거나 매핑 규칙을 설정할 때 보는 "지도" 역할을 하므로, 데이터의 **길이(Length)**와 **타입(Type)**을 정확하게 명시하는 것이 가장 중요합니다.

---

## 7. 핵심 단어 정리
- **연계 방식**: 직접(빠름, 민감), 간접(EAI, 유연, 느림).
- **연계 기술**: DB Link, API, Socket, Web Service.
- **전문 구성**: 공통부, 개별부, 종료부.
- **산출물**: 인터페이스 목록(전체), 인터페이스 정의서(상세).
